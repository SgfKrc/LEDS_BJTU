## 2. 下一阶段目标

下一阶段定义为 **TC-N1：可恢复任务状态与 Provider 边界**。

它不要求真实从节点在线，目标是先在单机和模拟 Worker 上证明：

1. 每次状态变化都能持久化、重放和审计；
2. 主节点重启不会把未完成任务误报为运行中，也不会重复提交结果；
3. 逻辑 Stage、执行 attempt 和 Provider 槽位相互独立；
4. 两个独立 ready Stage 可以由不同模拟 Provider 并行执行；
5. 迟到、重复和旧 epoch 结果不能污染聚合；
6. 上述能力通过后，真实 PC Full Worker 只需接入协议和传输，不再改变核心状态机。

该阶段细化现有 T0、T1 和 T2 的交界部分：先完成 T0，再用本机模拟完成 T1 的协议内核和 T2 的并行汇合，最后等待真实设备验收 T1。

## 4. 先修正状态机

### 4.1 增加结果提交屏障

当前协调器在最终 Stage 完成时立即把 Workflow 标为 `completed`，但 API 尚未把回答写入会话历史。取消发生在两者之间时，代码再把 `completed` 改为 `cancelled`。这会让事件重放无法判断哪个终态才有效。

建议 Workflow 状态调整为：

```text
created
  -> running
  -> result_ready
       -> completed
       -> cancelled
  -> failed
  -> cancelled
```

语义：

- `result_ready`：最终 Stage 输出已经在任务侧原子提交，但尚未确认写入用户会话。
- `completed`：会话历史/最终响应提交屏障已经通过。
- `cancelled`：提交屏障前取消，结果不得进入用户会话。
- `failed`：图、Stage、Provider 或提交校验失败。

`completed`、`failed`、`cancelled` 都是不可逆终态。删除 `completed -> cancelled` 转移，用 `commit_result()` 和 `discard_result()` 从 `result_ready` 分流。

### 4.2 Stage 与 Attempt 状态

Stage：

```text
blocked -> ready -> leased -> running -> validating -> completed
                                      \-> failed|cancelled
blocked|ready                          -> skipped|cancelled
```

Attempt：

```text
offered -> accepted -> running -> committing -> completed
   |          |          |            |
   +----------+----------+------------+-> rejected|failed|expired|cancelled
```

首个本地 Provider 可以省略网络 offer/accept，但事件仍使用同一语义，避免之后为远端 Worker 再改状态模型。

### 4.3 统一状态变更入口

禁止业务代码继续直接写 `workflow.state = ...`。每次转移通过单一方法完成：

```python
transition(
    entity_type,
    entity_id,
    expected_state,
    next_state,
    event_type,
    event_payload,
)
```

转移方法必须：

- 在实体锁内比较 expected state；
- 生成唯一 `event_id` 和工作流内递增 `sequence`；
- 先持久化事件和最新投影，再对外发布新状态；
- 重复 `event_id` 幂等成功；
- 非法转移返回结构化错误；
- 不在事件中写入完整 prompt、候选正文、密钥或数据库凭据。

## 5. 本地持久化方案

### 5.1 存储选择

采用 Python 标准库 `sqlite3` 建立独立任务 journal，不复用当前聊天历史 JSON 文件。

理由：

- 事件 ID、工作流 sequence 和唯一提交需要数据库约束；
- 一个事务可以同时追加事件和更新查询投影；
- SQLite 能处理进程异常后的事务回滚；
- 不需要联网安装新依赖；
- 比整文件 JSON 重写更适合高频状态事件。

事件表保持 append-only；可更新的 snapshot 只是查询加速投影，不是事实来源。

### 5.2 可写目录

安装包默认位于 Program Files，任务 journal 不能写在 exe 目录。新增独立 `STATE_DIR`：

```text
Windows: %LOCALAPPDATA%\QLH-Edge-Inference\state
Linux:   $XDG_STATE_HOME/qlh-edge-inference
         或 ~/.local/state/qlh-edge-inference
Override: QLH_STATE_DIR
```

任务数据库建议为：

```text
STATE_DIR/task_graph/task_graph.sqlite3
```

CPU/集显版和 CUDA 版若允许同时运行，应使用实例/节点命名空间，避免两个进程共用同一个协调器数据库。命名空间至少包含稳定 `node_id`；不得用会变化的显示名称。

### 5.3 最小 schema

```sql
CREATE TABLE workflow_events (
    event_id TEXT PRIMARY KEY,
    workflow_id TEXT NOT NULL,
    sequence INTEGER NOT NULL,
    entity_type TEXT NOT NULL,
    entity_id TEXT NOT NULL,
    event_type TEXT NOT NULL,
    occurred_at REAL NOT NULL,
    payload_json TEXT NOT NULL,
    UNIQUE (workflow_id, sequence)
);

CREATE TABLE workflow_snapshots (
    workflow_id TEXT PRIMARY KEY,
    last_sequence INTEGER NOT NULL,
    state_json TEXT NOT NULL,
    updated_at REAL NOT NULL
);
```

后续 payload blob、数据库同步 outbox 和保留策略可以增加独立表，不能把候选正文直接塞进事件表。

### 5.4 SQLite 运行约束

- 数据库必须位于本地文件系统，不放在 Tailscale/SMB 网络共享目录。
- 启用 WAL、foreign keys、busy timeout，并显式选择满足崩溃恢复要求的 synchronous 级别。
- 每个写事务短小，不在事务内做模型推理或网络等待。
- 初始化/迁移失败时，任务图功能应返回“journal 不可用”，不能静默退回不可恢复内存模式后仍宣称可恢复。
- 普通聊天模式不应被任务 journal 故障拖垮。
- 打包测试必须确认 `_sqlite3` 被包含且安装后的用户目录可写。

## 6. 重启恢复语义

TC-N1 不尝试续跑崩溃时的模型调用，采用保守恢复：

| 恢复前状态 | 恢复动作 |
|------------|----------|
| `completed` | 恢复为只读终态，可查询，不重复写会话 |
| `failed`/`cancelled` | 恢复原终态 |
| `result_ready` | 检查最终提交标记；未确认时标记 `failed`，错误码 `coordinator_restarted_before_result_commit` |
| `running/validating` Stage | 当前 attempt 标记 `expired`，Workflow 标记 `failed` |
| `created/ready/leased` | 释放 reservation/lease，Workflow 标记 `failed` |

恢复本身也必须追加事件，例如 `workflow_recovered`、`attempt_expired`、`workflow_interrupted`，不能只改内存快照。

后续实现 Stage checkpoint 和 payload store 后，才允许创建新 attempt 从已提交 Stage 输出继续。恢复绝不接续 PyTorch hidden state、KV cache 或 llama.cpp 内部上下文。

## 7. Provider 边界

### 7.1 最小接口

建议先采用结构化 Protocol，而不是把模型回调继续扩展：

```python
class ExecutionProvider(Protocol):
    provider_id: str

    def inspect(self) -> ProviderCapabilities: ...
    def reserve(self, request: StageRequest) -> Reservation: ...
    def execute(
        self,
        attempt: StageAttempt,
        reservation: Reservation,
        cancel_event: threading.Event,
    ) -> StageResult: ...
    def cancel(self, attempt_id: str) -> None: ...
    def release(self, reservation_id: str) -> None: ...
```

`reserve()` 必须原子占用并发槽。单纯读取能力快照后直接 execute 会在并发时超卖 Worker。

### 7.2 首批 Provider

只实现：

1. `LocalFullModelProvider`：包装当前 `model_manager.chat`，`max_concurrency=1`。
2. `DeterministicFakeProvider`：测试用，可控制延迟、失败、阻塞、重复结果和迟到结果。
3. `InProcessWorkerProvider`：使用与未来 PC Worker 相同的 offer/accept/lease/result 数据对象，但传输保持进程内。

真实 PC Worker 不在此批次接线。先通过 fake/in-process Provider 证明调度和状态机，再把传输替换为 TCP/WebSocket adapter。

### 7.3 Provider registry

Registry 负责：

- Provider ID 唯一性和生命周期；
- 能力快照及更新时间；
- Stage 硬约束过滤；
- reservation 和 release；
- Provider 健康状态；
- 选择原因和排除原因；
- 取消广播和关闭清理。

Registry 不负责修改 DAG，也不直接持久化会话正文。

## 8. ready Stage 并行调度

N1.4 已完成按 Provider 槽位的有界并行；生产模板的两个候选因共用单槽本地 Provider 仍串行，指定不同模拟 Provider 时执行以下流程：

1. 在锁内计算 ready Stage 集合。
2. 为每个 Stage 选择并 reserve Provider。
3. 释放工作流锁后提交执行，模型调用期间不持有状态锁。
4. 每个完成回调只提交对应 attempt 结果。
5. 在锁内经 fencing 闸门提交当前 epoch 的唯一 winner，再释放下游。
6. aggregate 只触发一次。

重要约束：

- 同一个 `LocalFullModelProvider` 仍保持并发 1，因此当前单模型 POC 行为不会因为线程池而变成不安全并发。
- 两个不同 fake/in-process Provider 可以并行，用于无从节点条件下验证 DAG 调度。
- 任务图全局并发、每 Workflow 并发和每 Provider 并发分别限制。
- 取消后不再创建新 attempt；已运行 Provider 收到取消并最终释放 reservation。
- 任一 future 抛异常都转为 Stage/Attempt 失败事件，不允许调度线程退出后留下永久 `running`。

## 9. Lease、epoch 与唯一提交

每个 Stage 维护：

- `lease_epoch`：每次重派单调增加；
- `winner_attempt_id`：成功提交后不可更改；
- `output_digest`：已提交输出摘要；
- `reservation_id`：用于释放 Provider 槽位。

提交条件：

```text
workflow 非终态
AND stage 尚无 winner
AND attempt_id 属于该 stage
AND attempt.lease_epoch == stage.lease_epoch
AND lease 未过期
AND output schema/digest 校验通过
```

重复提交同一 `attempt_id + digest` 幂等返回已提交；同一 attempt 不同 digest、旧 epoch 或不同 attempt 的迟到结果必须拒绝并记录原因。

TC-N1 只在模拟 Worker 上验证 lease 到期和重派。真实网络心跳、时钟偏差和断线检测在 PC Worker 接线时再做，但核心提交规则不再改变。

### N1.1：用户态 STATE_DIR 与 SQLite journal

状态：**已完成（2026-07-14）**。

- 新增路径配置和目录初始化。
- 实现 schema、事务追加、snapshot 投影和查询。
- Coordinator 注入 journal 接口；测试可使用内存或临时 SQLite。

实现结果：

- 新增 `task_journal.py`，事件 append 与 snapshot 投影在同一 SQLite 事务中提交，使用 WAL、foreign keys、busy timeout 和 `synchronous=FULL`。

- 验收时专项测试 `58 passed`，完整测试 `748 passed, 23 skipped`。

### N1.2：保守恢复与保留策略

状态：**已完成（2026-07-14）**。

- 启动时重放/加载 snapshot。
- 非终态 attempt 统一过期，工作流进入可解释失败。
- 增加按终态时间和数量清理 snapshot/event 的管理操作。
- 运行中的记录不得因 `max_records` 满而被静默删除。

验收：在 planning/running/result_ready 等位置模拟崩溃，重启后无幽灵运行态、无重复结果提交。

实现结果：

- 应用创建 Coordinator 时先扫描非终态 snapshot；运行中的 attempt 进入 `expired`，执行中的 Stage 进入 `failed`，尚未开始的 Stage 进入 `skipped`，Workflow 进入 `failed`。
- `running` 与 `result_ready` 使用不同固定错误码；恢复事件与新 snapshot 在同一 SQLite 事务中提交，sequence 连续，重复恢复不会追加第二个事件。
- 不恢复 KV、采样器、隐藏状态或未提交的最终正文；`result_ready` 重启后也不会自动提交会话历史。
- journal 文件使用跨进程单实例锁，旧 Coordinator 关闭后不可继续写入，避免两个进程同时恢复同一实例。
- 支持 `QLH_TASK_GRAPH_RETENTION_DAYS` 与 `QLH_TASK_GRAPH_RETENTION_MAX_RECORDS`；启动自动清理，也可由主节点调用 `POST /api/workflows/journal/cleanup`。
- 清理在一个事务中分批删除终态事件和 snapshot，非终态记录始终不进入候选集。

### TC-N2.2：取消与故障收敛

状态：**已完成（2026-07-17）**。

- Coordinator 在远端 attempt 接近截止时间时，仅对明确支持续租的 Provider 发送 `lease_renew`；Worker 收到后校验 workflow/stage/attempt/lease/epoch，过期租约禁止复活。
- 线上续租成功后复用 `renew_stage_lease()` 的同一校验和 journal 事件语义；传输失败或持久化失败会取消当前 attempt，不在 adapter 内旁路修改 winner 或 Stage output。
- `StageSpec.accept_timeout_seconds` 已传入远端 attempt，远端 `stage_accept` 不再固定等待 Provider 默认值；接受超时、执行断线和 lease 到期最终都释放 reservation。
- Coordinator 取消 running attempt 时发送 `stage_cancel`；Worker 立即设置协作取消事件并返回 `stage_cancelled`。无论确认是否送达，Coordinator 的本地取消和槽位回收都能独立收敛。
- Worker 为 `stage_offer`、`lease_renew` 和 `stage_cancel` 建立 1024 条有界消息缓存；同 ID/同内容重放既有响应，同 ID/不同内容稳定拒绝，重复 offer 不会二次执行模型。
- Worker 使用协议中的 `lease_expires_at_ms - sent_at_ms` 计算本地单调时钟时长，不直接比较主从设备的绝对系统时间；最终是否过期仍由 Coordinator 自己的 deadline fencing 决定。
- Worker 先缓存 Stage 响应再发送。结果发送瞬时失败后，相同 offer 可重放原 `stage_accept + stage_result`，不会重新执行，也不会生成冲突终态。
- 主节点 Provider 对 `stage_accept/result/error/cancelled` 做消息 ID 去重；重复响应在 reservation 释放后仍可幂等处理，错误 epoch、身份不匹配和未请求的取消确认继续拒绝。
- 主从 TCP 断线会唤醒所有等待者并协作取消 Worker active attempt；单个协议消息异常只记录拒绝，不会终止共享 TCP 收包循环或其他推理路径。
- N2.2 验收时自动 Provider 选择与 fallback 尚未开放；前两项已在 N2.3 补齐，双进程模型执行和物理从节点实测仍未完成。`adapter_connected=false` 与自动 `task_dispatch_enabled=false` 保持不变。

## 12. 必测故障矩阵

| 场景 | 通过条件 |
|------|----------|
| 事件重复写入 | 同 `event_id` 幂等，sequence 不跳变 |
| sequence 冲突 | 事务失败且内存状态不前进 |
| journal 文件不存在 | 自动建库和迁移 |
| journal 路径不可写 | 任务图不可用原因明确，普通聊天可用 |
| SQLite 正在 busy | 有界等待，超时后不留下半状态 |
| 写事务中进程终止 | 重启后只见完整事务 |
| `result_ready` 前取消 | 不写会话，Workflow cancelled |
| 会话提交成功后重复请求 | 不重复追加消息或统计 |
| running 时重启 | attempt expired，Workflow failed，不显示运行中 |
| 两个 Provider 同时完成 | 各自 Stage 正确提交，aggregate 一次 |
| 同一 Stage 新旧 attempt 同时返回 | 只有当前 epoch winner 可提交 |
| Provider execute 抛异常 | Stage 失败、槽位释放、工作流终态明确 |
| Provider cancel 抛异常 | Coordinator 继续完成取消，记录清理错误 |
| executor worker 意外退出 | 不留下永久 ready/running Stage |
| 数据库离线 | journal 和任务执行不受影响 |
| 安装目录只读 | journal 写入用户 STATE_DIR |
| PyInstaller 环境 | `_sqlite3` 可导入、迁移和恢复 |

## 13. 总体验收门槛

TC-N1 完成必须同时满足：

1. 所有 Workflow、Stage、Attempt 转移都有事件且可重放。
2. 终态不可逆，最终结果提交不会在取消竞态中重复或丢失。
3. 主节点重启后不存在幽灵 `running`，终态记录可查询。
4. 外部数据库不可用不影响 journal。
5. 两个模拟 Provider 能并行完成独立候选并正确 fan-in。
6. lease/epoch 能拒绝重复、迟到和旧 attempt 结果。
7. Provider reservation 在成功、失败、取消和异常路径都释放。
8. 现有任务链 API、thinking、历史隔离和模型切换测试不回归。
9. 集显和 CUDA PyInstaller 环境都能创建并恢复 journal。
10. 统计继续如实标记模拟/本地执行，不能把模拟 Worker 宣称为真实分布式参与。
