# 三种分布式拆分细化实施方案

> 状态：实施规划
>
> 更新日期：2026-07-14
>
> 范围：PyTorch 层间拆分、任务链/DAG 拆分、张量拆分/张量并行
>
> 总体架构与技术调研见[混合分布式推理体系规划](混合分布式推理体系规划.md)

## 1. 文档目标

QLH 后续不再把“分布式推理”理解成一条统一的模型层流水线，而是提供三种边界清晰、可以独立启停和独立验收的拆分方式：

| 拆分方式 | 拆分对象 | 主要价值 | 本文处理深度 |
|----------|----------|----------|--------------|
| PyTorch 层间拆分 | Transformer 连续层段 | 多个 PC 节点共同容纳并执行同一个 PyTorch 模型 | 已实现，本文只列待测试项 |
| 任务链/DAG 拆分 | 检索、推理、校验、汇总等业务 Stage | 让异构 PC、Android、不同模型和不同引擎协作 | 重点实施路线 |
| 张量拆分/张量并行 | 单层权重、激活或 KV 的 tensor shard | 高速同构集群中的容量聚合和低延迟并行 | 独立实验与实施路线 |

完整推理 Worker 是任务链的基础执行单元，不单独算作第四种拆分。GGUF/Skippy 层段属于另一种层间运行时实验，继续在混合体系规划中跟踪，不纳入本文近期实施范围。

## 2. 共同原则

### 2.1 统一控制语义，不统一内部张量协议

三种拆分只统一以下控制面信息：

- `request_id`：用户请求身份。
- `workflow_id`：任务链实例身份；非任务链请求可以为空。
- `stage_id`：业务 Stage 身份；非任务链请求可以为空。
- `attempt_id`：一次可独立成功或失败的执行尝试。
- `lease_id`、`lease_epoch`：Worker 或 Provider 的执行租约和栅栏版本。
- `logical_model_id`、`artifact_id`、`revision`、摘要：模型身份。
- `execution_mode`、`provider`：实际执行方式。
- 开始、进度、取消、完成、失败、超时和清理事件。
- 参与节点、耗时、资源、网络、重试和降级指标。

以下内容不得强行统一：

- PyTorch hidden states 与 KV cache 表示。
- MLX/CUDA/GGML 的 tensor shard 和 collective 协议。
- 不同引擎的量化权重布局。
- Provider 内部的进程拓扑、rank 和设备调度。

### 2.2 Provider 是故障边界

一个 attempt 进入某个 Provider 后，只能由该 Provider 完成或失败。不能在中间 hidden state、tensor collective 或部分 KV 状态上切换到另一 Provider 继续执行。

允许的恢复方式是：

1. 取消并清理失败 attempt。
2. 丢弃迟到结果。
3. 判断是否存在可重放输入或业务检查点。
4. 创建新的 `attempt_id`。
5. 从 Stage 输入、完整 messages 或其他明确检查点重新执行。

### 2.3 能力必须由运行时声明

设备名称、安装包版本和节点角色不能直接推导执行能力。节点或 Provider 至少声明：

```json
{
  "node_id": "node-a",
  "runtime": "pytorch|llama.cpp|mlx|cuda_provider",
  "capabilities": [
    "full_inference_worker",
    "task_stage_worker",
    "pytorch_layer_worker",
    "tensor_parallel_rank"
  ],
  "artifacts": [
    {
      "artifact_id": "model-artifact-id",
      "revision": "immutable-revision",
      "sha256": "...",
      "state": "missing|downloading|loading|ready|error"
    }
  ],
  "limits": {
    "max_concurrency": 1,
    "available_memory_bytes": 0,
    "thermal_state": "normal"
  }
}
```

调度器先进行兼容性硬过滤，再进行性能排序。不能用单一“设备评分”覆盖模型、网络和运行时约束。

### 2.4 外部数据库不是执行前提

任务链和 Provider 状态必须优先写入主节点本地持久化日志，再异步同步 PostgreSQL。数据库断开时：

- 已接收请求仍可以执行、取消和完成。
- attempt 和 lease 仍有唯一身份。
- UI 明确显示“本地状态源”。
- 数据库恢复后按事件 ID 幂等补写。

不能因为 Navicat、云数据库或节点表暂时不可用而阻塞本地 Worker 发现和执行。

### 2.5 功能开关与显式模式

建议保留独立开关：

```text
ENABLE_PYTORCH_LAYER_PROVIDER=true
ENABLE_TASK_GRAPH_PROVIDER=false
ENABLE_FULL_WORKER_PROVIDER=false
ENABLE_TENSOR_PARALLEL_PROVIDER=false
EXECUTION_MODE=auto|local|pytorch_layer|task_graph|tensor_parallel
```

规划能力默认关闭。显式选择某模式但能力不满足时，应返回结构化失败原因，不能静默改走本地推理后仍显示原模式。

## 3. PyTorch 层间拆分：只保留待测试项

PyTorch 层间拆分已有实现和专项文档，本文不再重复协议、分层算法、模型同步、容错设计或实施阶段。当前状态统一记为：**代码已具备基础能力，等待真实多设备和真实模型测试。**

### 3.1 待测试矩阵

| 类别 | 必测内容 | 通过条件 |
|------|----------|----------|
| 模型一致性 | 主从节点使用同一 `artifact_id`、revision、tokenizer 和摘要 | 不一致节点在执行前被排除，错误原因可见 |
| 分层正确性 | 连续覆盖全部层、无重叠、无缺层、Embedding/LM Head 所有权正确 | 分层不变量和 ready ACK 一致 |
| 主节点参与 | 独显主节点与弱从节点组合 | 主节点实际执行层数符合算力和内存，不再成为纯协调节点 |
| 数值正确性 | 同一 prompt、确定性参数下比较完整模型与流水线 | token 或 logits 达到预设一致性标准 |
| 模型准备 | 从节点缺模型、断点下载、摘要校验、层加载 | 未 ready 前不接收任务；失败可重试或排除 |
| Prefill/Decode | 短 prompt、长 prompt、长输出和多轮会话 | KV 长度、位置和输出连续，无重复 token |
| 网络异常 | 建链失败、执行中断线、超时、重连、迟到结果 | 当前 attempt 正确失败或降级，不污染下一请求 |
| 主节点状态 | 从节点已注册且 TCP 正常、数据库不可用、主节点发现切换 | 不再同时显示“已连接”和“主节点离线” |
| 取消 | 排队中、加载中、Prefill、Decode 四个阶段取消 | 释放锁、KV、队列和等待事件，不继续统计任务 |
| 回退 | Worker 未 ready、模型不一致、执行错误 | 只在 attempt 边界回退，`fallback_reason` 准确 |
| 统计 | 主节点和每个实际 Worker 的任务计数 | `nodes_participated` 与真实路径一致，不把在线节点算作参与 |
| 性能 | 首 token、decode、各节点耗时、网络 bytes/token | 能解释瓶颈，并与最快单节点基线对比 |
| 持续运行 | 多轮、并发、30 分钟以上持续推理 | 无明显内存、线程、KV 和任务状态泄漏 |

### 3.2 建议测试设备组合

- NVIDIA 独显主节点 + CPU/集显从节点。
- 高算力独显主节点 + 较弱独显从节点。
- 两节点和三节点拓扑。
- 数据库在线与数据库离线两种状态。
- Tailscale 直连与可控高延迟/丢包环境。
- Qwen-1.8B 基线模型；DeepSeek-R1-Distill-Qwen 模型只有在所有节点工件一致且层结构适配后才进入测试。

### 3.3 测试结论要求

层间拆分只有在真实测试报告中同时出现以下证据时，才能标为“已验收”：

- 实际执行模式为 `pytorch_layer`。
- 至少一个从节点完成层前向并返回结果。
- 主节点和从节点层段、耗时、任务计数可追溯。
- 输出正确性通过基线比较。
- 故障和取消测试通过。
- 性能或容量至少有一项明确收益；仅仅“用了多个节点”不算收益。

## 4. 任务链/DAG 拆分方案

### 4.1 目标与非目标

任务链把复杂请求拆成具有稳定输入输出契约的业务 Stage。它适合：

- PC、Android、不同操作系统和不同引擎之间协作。
- 多候选并行生成、交叉验证和最终汇总。
- 检索、文档分块、分析、代码执行、校验等异构任务。
- 把某个 Stage 交给本地完整模型、完整 Worker、PyTorch 层流水线或张量 Provider。
- 在单个 Worker 失败时，只重试受影响 Stage。

近期不做：

- 把一段自然语言随意切成无法验证的多个 prompt。
- 允许模型生成无限深度、无限 fan-out 的任务图。
- 在没有幂等保护时自动执行发邮件、删除数据、付款等外部副作用。
- 把中间 Worker 的流式 token 直接拼接成最终回答。
- 在同一个 Stage 内混用多个模型内部张量协议。

### 4.2 总体架构

```text
Chat/API Request
       |
       v
TaskGraphCoordinator
  - workflow state
  - deadline/cancel
  - local journal
       |
       +--> GraphBuilder / TemplateRegistry
       |
       +--> StageScheduler
       |      |
       |      +--> LocalProvider
       |      +--> FullWorkerProvider (PC/Android)
       |      +--> PyTorchLayerProvider
       |      +--> TensorParallelProvider
       |
       +--> ResultValidator / Aggregator
       |
       +--> Metrics & UI Events
```

`TaskGraphCoordinator` 与现有 `GraphOrchestrator` 必须分离：

- `GraphOrchestrator` 处理通信拓扑和 Transformer 层分配。
- `TaskGraphCoordinator` 处理业务依赖、Stage 状态、租约、重试和聚合。

建议新增独立模块目录，避免继续扩大 `scheduler.py`：

```text
src/execution/
  base.py                 # Provider 接口和公共数据类型
  registry.py             # Provider 注册与能力查询
  events.py               # 统一执行事件
src/task_graph/
  models.py               # Workflow/Stage/Attempt 数据结构
  templates.py            # 受控任务图模板
  validator.py            # DAG、schema、预算与权限校验
  coordinator.py          # 工作流状态机
  scheduler.py            # ready Stage 调度
  aggregation.py          # 结果校验与汇总
  journal.py              # 本地事件日志与 DB 同步
src/workers/
  gateway.py              # PC/Android Worker 控制协议
  leases.py               # 租约与 fencing
```

目录只是建议边界，实施时可根据现有模块调整；核心要求是任务图状态机不能与模型层状态机混为一体。

### 4.3 数据模型

#### Workflow

```json
{
  "workflow_id": "wf_...",
  "request_id": "req_...",
  "graph_version": 1,
  "state": "created|planning|running|aggregating|completed|partial|failed|cancelled",
  "deadline_at": 0,
  "root_input": {
    "messages": []
  },
  "policy": {
    "max_stages": 8,
    "max_parallelism": 3,
    "max_total_attempts": 12,
    "allow_partial": true,
    "allow_external_side_effects": false
  }
}
```

#### Stage

```json
{
  "stage_id": "research_a",
  "type": "full_inference",
  "depends_on": ["retrieve"],
  "state": "blocked|ready|leased|running|validating|completed|failed|skipped|cancelled",
  "input_refs": ["retrieve.output.documents"],
  "input_schema": "qlh.stage.research_input.v1",
  "output_schema": "qlh.stage.evidence.v1",
  "requirements": {
    "logical_model_id": "qwen-compatible",
    "capabilities": ["full_inference_worker"],
    "min_context_tokens": 8192,
    "allowed_providers": ["pc_full_worker", "android_full_worker", "local"]
  },
  "retry": {
    "max_attempts": 2,
    "retryable_errors": ["worker_lost", "timeout", "overloaded"]
  },
  "failure_policy": "fail_workflow|allow_partial|use_fallback_stage"
}
```

#### Attempt

```json
{
  "attempt_id": "att_...",
  "workflow_id": "wf_...",
  "stage_id": "research_a",
  "provider": "android_full_worker",
  "node_ids": ["android-01"],
  "lease_id": "lease_...",
  "lease_epoch": 3,
  "state": "offered|rejected|accepted|running|committing|completed|failed|expired|cancelled",
  "started_at": 0,
  "deadline_at": 0,
  "artifact_id": "...",
  "metrics": {}
}
```

Stage 的输出必须是带版本的结构化对象。即使底层模型只返回文本，也应包装成明确 schema，避免下游依赖不可解析的自由文本。

#### Payload、来源与持久化

Workflow 状态表只保存小型元数据和 payload 引用，不直接承载大文档、长候选或模型中间输出：

```json
{
  "payload_ref": "blob://workflow/wf_.../stage/research_a/output/sha256",
  "media_type": "application/vnd.qlh.evidence+json",
  "schema_version": 1,
  "size_bytes": 0,
  "sha256": "...",
  "producer": {
    "attempt_id": "att_...",
    "provider": "android_full_worker",
    "node_id": "android-01",
    "artifact_id": "..."
  }
}
```

存储规则：

- 小于配置阈值的 JSON 可以内联；大 payload 写入主节点本地 blob 目录或后续对象存储。
- Worker 先上传到临时对象，Coordinator 校验大小、摘要和 schema 后原子提交引用。
- payload 路径由服务端生成，不能接受 Worker 提供的任意本地路径，防止目录穿越。
- 下游 Stage 只能读取依赖图授权的 payload，不能按任意 `payload_ref` 获取其他会话数据。
- 每个引用带 workflow/stage/attempt 来源，聚合结果保留证据 provenance。
- Workflow 终态后按保留策略清理临时对象；已提交检查点由引用计数或 TTL 管理。
- 数据库只同步元数据和摘要。数据库断线不影响本地 payload 提交，恢复后幂等补写。
- 所有 Worker 输出均视为不可信输入：限制解压后大小、JSON 深度、字符串长度和数组元素数，再做 schema 与语义校验。

敏感输入还需带数据作用域，例如 `local_only`、`trusted_pc`、`trusted_tailnet`。调度器只能选择满足作用域的 Provider；Android 或外部 Provider 不因在线就自动获得全部会话数据。

### 4.4 Stage 类型

第一阶段只提供少量稳定类型：

| Stage 类型 | 作用 | 可执行后端 |
|------------|------|------------|
| `retrieve` | 检索本地或授权数据源 | 主节点工具、本地服务 |
| `transform` | 分块、清洗、格式转换 | 主节点 CPU、轻量 Worker |
| `full_inference` | 完整模型生成结构化候选 | PC/Android Worker、本地、其他 Provider |
| `verify` | 按规则或另一模型校验候选 | 本地/Worker/层或张量 Provider |
| `aggregate` | 合并证据、冲突和最终答案 | 可靠主节点或指定 Provider |

后续再增加代码执行、图像、Embedding、外部工具等 Stage。每种类型必须先定义输入 schema、输出 schema、超时、权限和幂等规则。

### 4.5 任务图生成

采用两阶段策略：

#### 受控模板优先

首个可用版本只支持系统定义模板，例如：

```text
双候选校验：
  candidate_a ----+
                  +--> verify --> aggregate
  candidate_b ----+

检索增强：
  retrieve --> analyze --> verify --> aggregate

分块汇总：
  split --> map[0..N] --> reduce --> final_verify
```

模板能限制 Stage 数量、输入输出和失败策略，便于测试。

#### 模型辅助拆解后置

模型生成的任务图只能作为草案，必须经过确定性校验器：

- 图必须无环。
- Stage ID 唯一，依赖存在。
- Stage 数、深度和 fan-out 不超过预算。
- 类型、schema、Provider 和模型均在 allow-list。
- token、时间和 attempt 预算可计算。
- 不允许未授权副作用。
- 必须存在可执行的最终聚合 Stage。

校验失败时回退固定模板或普通单模型推理，不能执行未经验证的图。

### 4.6 完整 Worker 协议

Worker 协议统一语义，不要求 PC 与 Android 使用相同传输实现：

```text
WORKER_HELLO / CAPABILITY_UPDATE
STAGE_OFFER
STAGE_ACCEPT | STAGE_REJECT
STAGE_STARTED
STAGE_PROGRESS
STAGE_RESULT
STAGE_ERROR
STAGE_CANCEL
STAGE_CANCELLED
LEASE_RENEW
RESULT_COMMITTED
```

PC 可以复用现有集群 TCP 连接的控制能力；Android 优先使用由移动端主动建立的 WebSocket 或受控长轮询，避免要求主节点主动访问手机监听端口。

`STAGE_OFFER` 至少包含：

- workflow/stage/attempt/lease 身份。
- 输入 schema 和内容摘要。
- 模型与 artifact 要求。
- 采样参数和最大输出。
- deadline、最大运行时间和取消通道。
- 结果 schema 和最大结果大小。
- 输入数据作用域和允许读取的 payload 引用。

Worker 接受任务前必须重新检查模型 ready、内存、温控、电量和并发槽位。能力上报与实际接受之间可能发生变化，`STAGE_REJECT` 是正常调度结果，不应视为节点崩溃。

### 4.7 租约、幂等与结果提交

每个 Stage 维护单调递增的 `lease_epoch`，每个 attempt 使用独立 `attempt_id` 和 `lease_id`：

1. Coordinator 创建新的 `attempt_id`、`lease_id`，并绑定 Stage 当前 `lease_epoch`。
2. Worker 接受后定期续租。
3. 超过租约期限未续租，attempt 进入 `expired`。
4. 重派时 Stage 的 `lease_epoch` 增加，并创建新的 attempt 和 lease；不复用旧 `attempt_id`。
5. 提交时同时校验 Stage 当前 epoch、获胜 `attempt_id` 和 lease。
6. 旧 Worker 的迟到结果因 epoch 或 attempt 不匹配被拒绝。

结果采用两阶段语义：

```text
Worker -> STAGE_RESULT(candidate, digest)
Coordinator -> validate schema / attempt / lease
Coordinator -> commit output atomically
Coordinator -> RESULT_COMMITTED
```

同一个 Stage 只能提交一个获胜结果。重复消息必须幂等，不能重复增加任务计数或触发两次下游 Stage。

控制消息按“可能重复、可能迟到”设计，即传输层只保证至少一次处理机会，不宣称网络层 exactly-once。系统通过事件 ID、attempt 身份、lease fencing 和原子结果提交实现“一个 Stage 最多提交一个获胜输出”。外部副作用仍需业务自己的幂等键。

状态机还必须满足：

- Workflow、Stage 和 Attempt 的终态不可逆。
- Stage 只能在所有必要依赖输出已提交后从 `blocked` 进入 `ready`。
- 一个 Stage 同一 epoch 最多有一个可提交 attempt；如做推测执行，只能有一个获胜提交者。
- Stage 完成后才释放下游；收到 `STAGE_RESULT` 但尚未原子提交不算完成。
- Workflow 只有在最终聚合输出提交后才能进入 `completed`。
- 取消先写入本地事件日志，再向 Worker 发送通知，避免主节点崩溃后恢复成运行态。

### 4.8 Stage 调度策略

调度分为硬约束和排序：

#### 硬约束

- Stage 类型和 Provider 能力匹配。
- 模型 artifact 已存在或允许在 deadline 内准备完成。
- 上下文长度、内存、并发和权限满足要求。
- 节点在线且执行通道健康。
- Android 满足前台服务、温控、电量或充电策略。
- 用户数据允许发送到该节点。
- Worker 并发槽位已通过原子 reservation 保留；仅查看上一次能力快照不算占用成功。

#### 排序因素

- 预计完成时间：排队 + 模型准备 + 推理 + 传输。
- 模型是否已驻留。
- 历史同类 Stage 的 p50/p95 耗时和成功率。
- 节点温控、能耗和电量惩罚。
- 当前工作流关键路径。
- 数据本地性和结果大小。
- 重派风险和网络稳定性。

初期使用可解释的加权估算；有足够历史样本后再研究学习型调度。调度日志必须记录被选原因和排除原因。

### 4.9 检查点、失败与降级

Stage 只在输出成功提交后成为业务检查点。恢复规则：

| 失败位置 | 处理 |
|----------|------|
| 图生成失败 | 回退受控模板或普通推理 |
| Worker 接受前失败 | 选择其他 Worker，不计执行 attempt 或标记 rejected |
| Worker 执行中失联 | 租约到期后重派；旧结果失效 |
| Stage 输出 schema 不合法 | 当前 attempt 失败，可换 Provider 重试 |
| 可选分支失败 | 按 `allow_partial` 继续，并把缺失信息交给聚合 Stage |
| 必要分支失败 | 执行 fallback Stage 或终止工作流 |
| 聚合失败 | 可使用已提交 Stage 输出重新聚合，不重跑全部前置 Stage |
| 用户取消 | 停止发新任务，向运行中 Worker 广播取消，忽略迟到结果 |

外部副作用 Stage 必须使用业务幂等键，并默认要求人工确认。没有幂等保证时不得自动重试。

### 4.10 流式输出

任务链首版只流式发送状态事件，不直接流出各分支的正文：

```text
planning -> stage_started -> stage_completed -> aggregating -> final_stream
```

只有最终聚合 Stage 可以向用户输出正文 token。这样可以：

- 避免两个候选分支交错显示。
- 在前置 Stage 失败时安全重派。
- 防止已经展示的中间结论被后续校验推翻。

后续若支持可撤回草稿，必须在 UI 中与最终回答明确分区，不能混入最终消息。

建议新增或扩展：

- DAG、各 Stage 状态和依赖。

不能只显示“分布式：是”。

建议指标：

- `workflow_total{template,state}`。
- `workflow_duration_seconds{template}`。
- `stage_attempt_total{type,provider,state}`。
- `stage_duration_seconds{type,provider,node}`。
- `stage_queue_seconds`、`stage_prepare_seconds`。
- `worker_lease_expired_total`。
- `stage_retry_total{reason}`。
- `workflow_partial_total`。
- `workflow_fallback_total{from,to,reason}`。
- 每个 Worker 的 accepted/completed/failed/cancelled 计数。

### 4.13 安全边界

Tailscale 网络准入可以作为主要设备信任边界，控制协议保持轻量：共享集群密钥、短期 bootstrap token 和 attempt 租约即可，不必首版引入复杂 PKI。

但仍需防止可信网络中的错误客户端破坏状态：

- Worker 只能接收能力范围内的 Stage。
- 输入和结果限制大小，schema 严格校验。
- 旧 lease 不能提交结果。
- 工具和数据源按 Stage 授权。
- 日志不写入模型密钥、数据库密码和完整敏感输入。

### 4.14 分阶段实施

#### T0：公共执行身份和本地事件日志

- 定义 workflow/stage/attempt/lease 数据结构。
- 建立本地 append-only 事件日志和幂等恢复。
- 提供 Provider registry，但先只注册本地完整推理。
- 增加显式 `execution_mode` 和准确统计。

验收：数据库离线时，主节点重启后仍能识别未完成和已完成 attempt，不重复提交结果。

#### T1：PC 完整 Worker

- 定义 Worker 能力、offer/accept/result/cancel 协议。
- 先接入 PC Worker，复用现有模型管理和集群连接。
- 实现租约、超时、迟到结果过滤和任务计数。

验收：两个 PC Worker 可以分别执行完整子任务；断开一个 Worker 只重派对应 Stage。

#### T2：固定任务图模板

- 实现双候选校验、检索增强和分块汇总模板中的至少一种。
- 建立 DAG 校验、ready Stage 队列和最终聚合。
- 首版禁止任意模型生成任务图。

验收：并行 Stage 能正确汇合；部分失败、取消和聚合重试可恢复。

#### T3：Android Full Worker

- Android 主动建立任务通道。
- 上报模型、温控、电量、充电和前台服务能力。
- 实现完整推理、进度、取消和结果提交。
- Lite 版不接收 Worker 任务。

验收：Android 失联或被系统回收后，租约过期并重派，不拖垮工作流。

#### T4：Provider 组合

- 允许 Stage 显式选择本地、完整 Worker或现有 PyTorch 层 Provider。
- 张量 Provider 可用后作为另一个 Stage 后端接入。
- 统一嵌套 Provider 的统计和取消传播。

验收：同一工作流的不同 Stage 可以使用不同引擎，但每个 Stage 的故障域和模型身份明确。

#### T5：模型辅助拆解与 Auto 调度

- 在固定模板稳定后开放模型生成草案。
- 实施图校验、预算限制和模板回退。
- Auto 调度先 shadow 运行，只记录计划，不改变执行。
- 有足够成功率和收益数据后逐步开放。

验收：不合法任务图不会执行；Auto 选择相对固定策略有可验证收益。

### 4.15 任务链待测试矩阵

| 类别 | 必测内容 | 通过条件 |
|------|----------|----------|
| 图校验 | 环、自依赖、未知 Stage、超深图、超 fan-out、预算超限 | 非法图不执行，并回退受控模板或普通推理 |
| 依赖调度 | 串行、并行、fan-in、可选分支和 fallback Stage | 只有必要依赖提交后才进入 ready，下游只触发一次 |
| Worker 竞态 | offer 后拒绝、接受超时、槽位被其他任务占用 | reservation 原子，不超出 `max_concurrency` |
| 重复消息 | 重复 offer、progress、result、cancel 和 commit ACK | 状态与计数幂等，不重复触发下游 |
| 租约过期 | Worker 失联后重派，新旧 attempt 同时返回 | 只有当前 epoch 的获胜 attempt 可以提交 |
| 主节点重启 | planning、running、committing、aggregating 各阶段重启 | 从本地日志恢复，不重复提交或丢失已提交检查点 |
| 数据库断线 | 创建、执行、完成和恢复期间数据库不可用 | 本地工作流正常；恢复后元数据幂等补写 |
| Payload | 大结果、摘要错误、非法路径、压缩炸弹、超深 JSON | 拒绝不安全载荷，不污染已提交检查点 |
| 数据权限 | `local_only` 输入尝试调度到 Android/外部 Provider | 硬过滤并记录排除原因 |
| Stage schema | Worker 返回缺字段、错版本或无法解析结果 | attempt 失败，可按策略换 Worker 重试 |
| 部分失败 | 可选分支失败、必要分支失败、聚合失败 | 分别进入 partial、fallback 或 failed，状态可解释 |
| 取消 | 排队、运行、结果提交前、聚合流式输出中取消 | 不再创建新 attempt，运行任务收到取消，迟到结果无效 |
| Android 生命周期 | 锁屏、切后台、断网、温控、电量不足、进程回收 | 正确拒绝/续租失败/重派，不拖垮工作流 |
| Provider 组合 | 本地、PC Worker、Android Worker、层/张量 Provider 混合 | Stage 边界只传业务对象，取消和统计传播正确 |
| 流式输出 | 多分支执行和最终聚合 | 中间正文不混入用户消息，仅最终 Stage 输出正文 |
| 统计 | 重试、重派、partial、fallback、实际参与节点 | workflow/Stage/attempt 三层计数一致且可追溯 |
| 质量 | 固定模板与单模型回答对照 | 至少一个目标任务集质量收益达到预设门槛 |
| 性能 | 调度开销、并行收益、P50/P95 完成时间 | 能解释额外开销，关键路径没有无意义串行化 |
| 持续运行 | 多工作流并发和至少 30 分钟负载 | 无事件、payload、租约、线程和 Worker 槽位泄漏 |

### 4.16 任务链验收门槛

任务链正式启用必须同时满足：

1. 固定模板正确性、取消、超时、重派和恢复测试通过。
2. 数据库离线不阻塞工作流。
3. Worker 迟到结果不能覆盖新 attempt。
4. Android/PC 能力差异不会导致错误 Stage 下发。
5. 任务统计能区分 workflow、Stage、attempt 和实际节点。
6. 相比单模型基线，在至少一种目标场景中提高质量、吞吐或可用设备利用率。
7. 额外调度开销和失败率在可接受范围内。

## 5. 张量拆分/张量并行方案

### 5.1 定位

张量并行把每个 Transformer 层中的矩阵、激活或 KV 按 rank 分片。每层通常需要 collective 通信，因此它只适合低延迟、高带宽且高度同构的设备组。

目标：

- 单设备装不下模型时聚合多设备内存。
- 在高速互联下提高单请求 prefill/decode 性能。
- 通过常驻模型实例服务多个请求。

非目标：

- 不让 Android、Wi-Fi 或普通 Tailscale 移动节点加入 collective group。
- 不复用 `LAYER_FORWARD`、`CHAIN_FORWARD` 或任务链 Worker 协议传 tensor shard。
- 不由 QLH 首版自行实现 all-reduce/all-gather 内核。
- 不把 llama.cpp 默认 `--tensor-split` 比例参数误认为张量并行。
- 不在任一 rank 失败后让替补节点从中间 token 接管。

### 5.2 Provider 路线

张量并行作为独立 `TensorParallelProvider`，首版优先适配成熟外部运行时，而不是在现有 Python socket 上重写 collective。

候选路线：

| 路线 | 目标环境 | 接入方式 | 当前定位 |
|------|----------|----------|----------|
| exo/MLX Provider | Apple Silicon、高速 Thunderbolt/RDMA 环境 | 外部服务 API adapter | 有匹配硬件时优先预研 |
| CUDA Tensor Provider | 同构 NVIDIA GPU、高速直连网络 | 选定运行时的外部服务 adapter | 先基准选型，再实现 |
| llama.cpp `split-mode=tensor` | 上游明确支持的本机多 GPU/实验环境 | 独立实验进程 | 能力探测，不作为跨主机主路线 |

CUDA 具体运行时不在文档阶段预定。应先使用目标模型、目标 GPU 和目标网络进行小型选型基准，再锁定一个 Provider；避免同时维护多套 collective runtime。

### 5.3 控制面架构

```text
QLH Execution Planner
        |
        v
TensorParallelProvider Adapter
  - capability inspection
  - group plan
  - instance lifecycle
  - request/cancel/metrics
        |
        v
Provider Native Control API
        |
        v
Tensor Instance
  rank0 + rank1 + ... + rankN
  native collective data plane
```

QLH 负责：

- 发现候选设备组。
- 验证模型 artifact、运行时和网络准入。
- 创建、预热、路由、排空和销毁实例。
- 请求身份、取消、超时、统计和降级。
- 将 Provider 原生指标映射为统一指标。

外部 Provider 负责：

- rank 启动与 rendezvous。
- 权重和 KV 分片。
- collective 通信。
- Provider 内部 batch、采样和缓存。
- rank 级错误检测与资源清理。

### 5.4 Tensor Group 能力模型

每个候选 rank 上报：

```json
{
  "node_id": "gpu-node-1",
  "tensor_runtime": {
    "provider": "selected_cuda_provider",
    "runtime_version": "immutable-version",
    "build_id": "...",
    "driver_version": "...",
    "collective_backend": "nccl|mlx_distributed|provider_native"
  },
  "device": {
    "accelerator_family": "...",
    "memory_total_bytes": 0,
    "memory_free_bytes": 0,
    "supported_dtypes": ["fp16", "bf16"]
  },
  "network": {
    "interface": "...",
    "direct_address": "...",
    "bandwidth_gbps": 0,
    "rtt_ms": 0,
    "collective_benchmark_id": "..."
  },
  "artifacts": ["artifact-id"]
}
```

Tensor group 的硬性不变量：

- 相同 Provider 和兼容 runtime/build。
- 相同模型 `artifact_id`、revision、tokenizer 和量化布局。
- TP degree、rank 顺序和分片规则固定。
- dtype 和计算能力满足模型要求。
- 每个 rank 有足够内存余量。
- rank 间存在 Provider 支持的直接通信路径。
- rendezvous、端口和 collective 健康检查通过。

默认要求同一加速器家族和相近显存/性能。只有 Provider 明确支持且实测通过时才允许异构 rank，否则最慢 rank 会拖慢每层 collective。

### 5.5 网络准入

在创建 group 前执行独立网络基准，而不是复用普通 HTTP ping：

- 点对点 RTT p50/p95/p99。
- 单向和双向吞吐。
- 小消息和大消息吞吐。
- Provider 原生 all-reduce/all-gather 基准。
- 30 秒以上持续 collective 的抖动和错误率。
- 是否经过中继、NAT、Tailscale DERP 或无线链路。

默认策略：

| 网络 | 默认准入 |
|------|----------|
| 同机 PCIe/NVLink | 可进入基准与静态 group |
| Apple Silicon 高速直连/RDMA | Provider 支持且实测后准入 |
| 同构 CUDA + 高速以太网/RDMA | Provider 支持且 collective 基准通过后准入 |
| 普通 1GbE | 仅实验，不自动选择 |
| Wi-Fi、移动网络、Tailscale DERP | 拒绝 tensor group |
| Android 设备 | 拒绝 tensor group |

Tailscale 可以用于控制面发现和管理，但 tensor 数据面是否允许经过 Tailscale 必须由实测决定，默认不允许。

### 5.6 模型结构与并行策略准入

“所有 rank 都有同一模型文件”仍不足以证明模型可做张量并行。Provider 必须在建组前输出经过验证的并行计划，至少说明：

| 结构 | 常见处理 | 准入检查 |
|------|----------|----------|
| Attention Q/K/V 投影 | 按输出维度做 column parallel | attention heads、KV heads 与 TP degree 的整除/复制规则 |
| Attention output 投影 | 按输入维度做 row parallel，随后 collective | Provider 是否在正确位置执行 reduce-scatter/all-reduce |
| MLP gate/up 投影 | column parallel | intermediate size、量化 block 与 TP degree 兼容 |
| MLP down 投影 | row parallel | collective dtype 与累加精度一致 |
| Embedding | 复制或 vocabulary parallel | vocab padding、token ID 路由和聚合规则 |
| LM Head | 复制或 vocabulary parallel | logits 聚合、采样所有者和 tied weights 规则 |
| Norm/RoPE | 通常复制 | 参数、位置和 dtype 在各 rank 一致 |
| KV Cache | 按 attention/KV head 或 Provider 规则分片 | GQA/MQA 下 KV head 数可能小于 TP degree |

模型准入至少校验：

- 架构在 Provider allow-list 中，不能仅凭 `AutoModel` 能加载就通过。
- hidden size、attention heads、KV heads、intermediate size 和 vocab 分片满足 Provider 约束。
- GQA/MQA 的 KV 复制或分片策略明确；不允许默认为普通 MHA。
- tied embedding/LM Head、滑动窗口、RoPE scaling、特殊 attention mask 等语义经过 Provider 支持矩阵确认。
- 量化 pack/block 布局可按目标 TP degree 切分；不能任意切开量化块。
- TP degree 变化需要独立并行计划和重新 warm-up，不能直接复用旧实例状态。
- MoE 的 expert parallel 不等于 tensor parallel；首版不把 MoE expert 路由混入 TP 计划，除非外部 Provider 完整支持并单独验收。

并行计划作为 artifact manifest 的一部分固化：

```json
{
  "tp_degree": 4,
  "strategy_revision": "provider-strategy-v1",
  "attention": "provider-defined",
  "mlp": "provider-defined",
  "embedding": "replicated|vocab_parallel",
  "lm_head": "replicated|vocab_parallel",
  "kv_cache": "provider-defined",
  "collective_dtype": "fp16",
  "validated_architecture": "..."
}
```

QLH 不解释或执行具体 tensor 切分，只校验 manifest 与实例实际报告一致。

### 5.7 模型 artifact 与分片准备

逻辑模型必须为张量 Provider 注册独立 artifact：

```json
{
  "artifact_id": "qwen-model-tp-provider-rev1",
  "logical_model_id": "qwen-model",
  "provider": "tensor_parallel",
  "format": "mlx|provider_native|safetensors_supported_layout",
  "revision": "immutable",
  "manifest_sha256": "...",
  "supported_tp_degrees": [2, 4],
  "supported_dtypes": ["fp16"],
  "architecture": "allow-listed-architecture"
}
```

准备流程：

1. 下载或转换到临时目录。
2. 校验 manifest、文件摘要和 tokenizer。
3. Provider 检查模型架构与 TP degree。
4. 各 rank 完成本地缓存或共享存储映射。
5. 所有 rank 报告 artifact ready。
6. 实例加载并执行 warm-up。
7. warm-up 正确性和 collective 健康通过后进入 `ready`。

任何 rank 未 ready，整个实例都不能接收用户请求。

### 5.8 实例生命周期

```text
discovered
  -> validating
  -> reserving
  -> launching
  -> rendezvous
  -> loading
  -> warming
  -> ready
  -> draining
  -> stopped

任意中间状态 -> failed -> cleanup -> stopped
```

关键规则：

- `instance_id` 和 `generation` 唯一标识一次完整 group 启动。
- 同一设备同一时刻默认只加入一个重模型 tensor instance。
- `ready` 前不路由请求。
- 配置或模型变化通过新 generation 重建，不在运行实例中热改 rank。
- 排空时拒绝新请求，等待或取消在途请求，再统一释放所有 rank。
- 启动失败必须清理已成功启动的部分 rank 和端口。

### 5.9 请求路由

Tensor Provider 以完整模型服务对 QLH 暴露：

```text
inspect()
plan(request, candidate_groups)
prepare(plan)
execute(attempt, on_event)
cancel(attempt_id)
drain(instance_id)
release(instance_id)
```

请求只能路由到 `ready` 实例。规划考虑：

- 请求模型和 artifact。
- 上下文与输出长度。
- 实例排队深度和 batch 策略。
- KV cache 容量。
- 预计首 token 和完成时间。
- 实例重建成本。
- 单节点完整推理和其他 Provider 基线。

首版使用静态常驻 group，不按每个聊天请求动态创建。模型加载和 rendezvous 成本通常远大于单次路由收益。

### 5.10 取消、失败与恢复

Tensor group 是整体故障域：

| 故障 | 处理 |
|------|------|
| 单个 rank 启动失败 | 整个 generation 启动失败并清理 |
| rendezvous 超时 | 终止全部 rank，释放预留资源 |
| 模型摘要不一致 | 禁止 ready，不尝试运行 |
| collective 超时或 rank 丢失 | 当前所有在途 attempt 失败，实例进入 failed |
| 单请求 Provider 错误 | 按 Provider 能力取消该请求；无法隔离时排空并重建实例 |
| 用户取消 | 通过 Provider 原生取消；确认资源释放后结束 attempt |
| 实例失败后的降级 | 从完整 messages 新建 attempt，切到本地、完整 Worker 或层 Provider |

不允许：

- 用备用 rank 从中间 token 接替失败 rank。
- 保留部分 rank 的 KV 并切换 Provider。
- 将失败实例的迟到流式 token提交给已重试请求。

### 5.11 流式输出

Provider 的 rank0 或统一网关是唯一输出源。QLH 只接收：

- `attempt_started`。
- 标准化 token/chunk 事件。
- usage 和结束原因。
- Provider/instance/rank 指标。
- 失败和取消确认。

QLH 不接收各 rank 的部分 logits 或 tensor。流式 attempt 一旦向用户提交正文，默认不自动跨 Provider 从头重试；应结束并明确提示，或由上层任务链在尚未提交 Stage 输出时重试。

### 5.12 调度与收益门槛

Tensor Provider 不因“存在多个 GPU”就自动启用。Auto Planner 至少比较：

- 最快单节点是否能装下模型。
- 单节点和 tensor group 的模型准备成本。
- 首 token、prefill tok/s、decode tok/s 和端到端时延。
- 并发吞吐和 batch 效率。
- bytes/token、collective 时间占比和最慢 rank 等待。
- 30 分钟持续负载的稳定性。
- rank 失败后的恢复时间。

建议达到以下任一收益后才允许进入候选：

1. 目标模型单节点无法装下，而 tensor group 可以稳定运行。
2. 预热后 P50 端到端时延相对最快单节点改善至少 20%。
3. 同等输出质量和延迟约束下，持续吞吐改善至少 30%。

阈值应配置化，并以目标硬件基准校准。未达到收益时保留实验入口，但不进入默认 Auto 选择。

### 5.13 可观测性

统一指标：

- `tensor_instance_state{provider,instance,generation}`。
- `tensor_instance_start_seconds`。
- `tensor_request_total{provider,state}`。
- `tensor_request_duration_seconds`。
- `tensor_first_token_seconds`。
- `tensor_prefill_tokens_per_second`。
- `tensor_decode_tokens_per_second`。
- `tensor_collective_seconds{operation}`。
- `tensor_collective_bytes_total{operation}`。
- `tensor_rank_wait_seconds{rank}`。
- `tensor_rank_failure_total{rank,reason}`。
- `tensor_instance_restart_total{reason}`。

UI 展示：

- Provider、instance、generation 和 TP degree。
- rank 到节点/设备映射。
- 模型 artifact 和 ready 状态。
- 网络基准与 collective 占比。
- 计划 rank、实际 rank 和失败 rank。
- 相对单节点基线的收益。

### 5.14 安全与运维

- 控制面沿用 Tailscale + 集群共享密钥。
- 每次 group 启动使用短期 rendezvous token，不复用长期数据库密码。
- tensor 数据端口只监听指定接口并限制 group 节点。
- Provider 进程使用独立工作目录、日志和模型缓存。
- 固定 Provider 版本、模型 revision、驱动和依赖清单。
- 升级采用新 generation 灰度启动，旧实例排空后退出。
- Provider 崩溃不能终止 QLH 主 API、本地推理或任务链控制面。

### 5.15 分阶段实施

#### X0：硬件与运行时选型基准

- 确定首个真实目标环境：Apple Silicon 集群或同构 CUDA 集群，只选一个先落地。
- 选择 1 个小模型和 1 个目标模型。
- 测单节点、多节点网络、collective、正确性和持续负载。
- 锁定候选 Provider commit/version 和许可证。

验收：得到可复现基准和 Go/No-Go 结论；没有匹配硬件时保持规划状态，不写“已支持”。

#### X1：外部 Provider Adapter

- 实现 inspect、health、model mapping、execute、cancel 和 metrics adapter。
- 首版手工启动 Provider 实例，QLH 只负责路由。
- 与现有 PyTorch socket 完全隔离。

验收：显式 `tensor_parallel` 请求走外部 Provider，实际模式和指标准确。

#### X2：静态 Tensor Group

- 实现 group 配置、rank 映射和 artifact 校验。
- 建立 readiness、warm-up、排空和统一清理。
- 禁止运行中变更 rank。

验收：任一 rank 未 ready 时不接流量；启动失败能清理全部部分资源。

#### X3：实例生命周期自动化

- 自动预留、启动、rendezvous、加载、预热和重建。
- 增加 generation fencing 和实例状态持久化。
- 主节点重启后识别并接管或清理孤儿实例。

验收：重复启动、超时和主节点重启不会产生多个冲突 group。

#### X4：故障、压力与收益验证

- 注入 rank 崩溃、collective 超时、网络抖动和取消。
- 比较最快单节点、层 Provider 和 tensor Provider。
- 做长上下文、多并发和 30 分钟以上持续负载。

验收：故障域、恢复时间和收益达到预设门槛。

#### X5：任务链与 Auto Planner 接入

- 任务链 Stage 可选择 tensor Provider。
- Auto Planner 先 shadow 比较，不自动切换。
- 指标稳定后按模型和硬件 allow-list 开放自动选择。

验收：嵌套执行统计准确；Auto 不会把普通网络或不兼容设备组成 tensor group。

### 5.16 张量并行 Go/No-Go

Go：

- 有稳定、可复现的目标硬件和高速互联。
- Provider 和模型版本可锁定。
- 正确性、取消、rank 故障和清理测试通过。
- 相比单节点获得容量或性能收益。
- Provider 可独立进程运行，不污染现有 PyTorch/llama.cpp 环境。

No-Go：

- 只能识别多个节点，collective 性能没有收益。
- 必须通过 Wi-Fi、DERP 或高抖动链路运行。
- 模型/量化布局无法在所有 rank 一致复现。
- 任一 rank 失败后存在无法清理的进程、显存或端口。
- 需要把外部 tensor 协议塞入现有 `LAYER_FORWARD`。

### 5.17 张量并行待测试矩阵

| 类别 | 必测内容 | 通过条件 |
|------|----------|----------|
| 结构准入 | MHA/GQA、Embedding、LM Head、量化布局与 TP degree | Provider 生成可复现并行计划，不支持结构明确拒绝 |
| 工件一致性 | 所有 rank 的 artifact、revision、manifest 和 tokenizer | 任一不一致时整个实例禁止 ready |
| 数值正确性 | 单节点与 TP 实例的 greedy 输出/logits 对比 | 达到 Provider 和模型约定的误差门槛 |
| Rank 映射 | TP=2/4 等受支持 degree 与设备顺序 | 实际 rank、分片和 UI 映射一致 |
| Warm-up | 首次加载、缓存命中、重建 generation | 只有 warm-up 和 collective health 通过后 ready |
| 长上下文 | Prefill、Decode、KV 增长和上下文上限 | 无 KV 错位、OOM 泄漏和输出跳变 |
| 并发与 batch | 单请求、多请求和不同长度混合 | 排队、batch 和取消互不污染 |
| 流式输出 | 正常结束、客户端断开、Provider 取消 | 只有统一网关输出，结束原因准确 |
| Rank 故障 | 启动、加载、Prefill、Decode 各阶段终止一个 rank | 当前实例和在途 attempt 整体失败并完成清理 |
| Collective 故障 | 超时、网络抖动、错误 rank、rendezvous 冲突 | 有界超时，不永久挂起主 API |
| 主节点重启 | ready、draining、failed 状态下重启 QLH | 孤儿实例被接管或清理，不重复建组 |
| 资源清理 | 重复启动/停止、取消、失败重建 | 无残留进程、显存、端口和 rendezvous 状态 |
| 收益 | 容量、P50/P95 时延、吞吐、collective 占比 | 达到配置门槛，否则不进入 Auto 候选 |
| 持续负载 | 至少 30 分钟目标并发 | 无 rank 漂移、显存增长和错误率恶化 |

## 6. 三种拆分的组合规则

### 6.1 任务链可以组合 Provider

任务链是业务层组合器，可以出现：

```text
retrieve
   -> candidate_a (Android full worker)
   -> candidate_b (PyTorch layer provider)
   -> verify      (tensor parallel provider)
   -> aggregate   (local provider)
```

组合规则：

- 每个 Stage 只选择一个 Provider。
- Provider 只接收完整 Stage 输入。
- Stage 间只传版本化业务对象。
- 不把一个 Provider 的 hidden states、KV 或 tensor shard交给另一个 Provider。
- 上层取消必须传播到当前所有运行 attempt。

### 6.2 统计规则

一个任务链 Stage 使用张量并行时，应同时统计外层和内层：

```json
{
  "execution_mode": "task_graph",
  "workflow_id": "wf_...",
  "stage_id": "verify",
  "stage_provider": "tensor_parallel",
  "tensor_instance_id": "tp_...",
  "nodes_planned": 4,
  "nodes_participated": 4,
  "fallbacks": []
}
```

不能因为外层是任务链就隐藏张量 rank，也不能因为内层用了张量并行就把整个工作流统计成一个 tensor 请求。

### 6.3 Auto Planner 初始决策顺序

1. 用户显式模式优先，并进行能力校验。
2. 普通交互且单节点可满足时，优先已预热的本地/完整 Worker。
3. 请求明确需要多阶段质量或异构协作时，选择任务链。
4. Safetensors 模型单卡装不下且 PyTorch 层测试已通过时，可选择层 Provider。
5. 目标模型存在 ready tensor instance 且达到收益门槛时，可选择 tensor Provider。
6. 任一 Provider 不 ready 时，按完整请求边界降级并记录原因。

Auto 初期必须以 shadow planning 运行：同时记录“建议计划”和“实际计划”，在没有足够数据前不改变用户请求路径。

## 7. 公共实施依赖

任务链 T0/T1 和张量 X1 依赖同一组公共能力，应优先建设：

| 公共能力 | 任务链 | 张量并行 |
|----------|--------|----------|
| Provider registry | 选择 Stage 后端 | 注册 tensor adapter |
| artifact identity | Worker 模型匹配 | rank 模型严格一致 |
| attempt/lease/fencing | Stage 重派与迟到结果 | generation 和请求隔离 |
| 本地事件日志 | 工作流恢复 | 实例生命周期恢复 |
| cancel/event API | 取消整个 DAG | 取消 Provider 请求 |
| 统一 metrics | workflow/Stage 统计 | instance/rank 统计 |
| 显式 execution mode | 防止错误显示 | 防止伪张量并行 |

公共能力完成后，两条路线可以独立推进。任务链不应等待张量 Provider；张量 Provider 也不依赖 Android Worker。

## 8. 推荐实施顺序

```text
公共执行身份、Provider registry、本地事件日志
                     |
          +----------+----------+
          |                     |
   任务链 T1-T3             张量 X0-X2
   PC/Android Worker        选型/Adapter/静态组
          |                     |
   任务链 T4 组合 <---------- Tensor Provider ready
          |
   Shadow Auto Planner
          |
   受控开放自动选择
```

优先级建议：

1. 先完成 PyTorch 层间拆分真实测试，明确当前基线是否可靠。
2. 同时建设公共 Provider 身份和事件日志。
3. 任务链先从 PC 完整 Worker和固定模板落地，再接 Android。
4. 张量并行只有在获得匹配硬件后进入 X0；否则保留文档和接口预留。
5. 两条路线各自通过验收后再进入统一 Auto Planner。

## 9. 最终验收口径

三种拆分不得共用一个模糊的“分布式正常”结论：

| 方式 | 最低验收证据 |
|------|--------------|
| PyTorch 层间拆分 | 真实从节点执行层前向、模型一致、输出正确、故障可恢复、统计准确 |
| 任务链拆分 | 多 Stage/多 Worker 实际执行、检查点与重派有效、聚合正确、部分失败可解释 |
| 张量拆分 | 多 rank collective 实际参与、模型与运行时一致、故障整体清理、容量或性能达标 |

每份测试报告必须给出实际 `execution_mode/provider`、计划和实际参与节点、模型 artifact、失败与降级、单节点基线以及最终收益。没有这些证据时，只能标记为“功能已接线”或“实验可运行”，不能标记为“分布式推理已验收”。
