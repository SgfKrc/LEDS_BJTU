## 4.2 核心架构

### 4.2.1 流水线数据流

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  主节点      │     │  从节点 1    │     │  从节点 2    │
│  Layers 0-7 │ ──→ │ Layers 8-15 │ ──→ │ Layers 16-23│
│  + Embed    │     │             │     │  + LM Head  │
└─────────────┘     └─────────────┘     └─────────────┘
   ↑                                        │
   │  prompt (text)                         │  logits → token
   │                                        ↓
   用户                               ┌─────────────┐
                                      │  主节点      │
                                      │  汇总结果    │
                                      └─────────────┘
                                            │
                                            ↓  response text
                                          用户
```

**单次推理流程 (autoregressive generation)**:

```
Step 1: 主节点收到 prompt → tokenize → input_ids
Step 2: 主节点执行 Embedding + Layer 0-7 → hidden_states[0]
Step 3: TCP 发送 hidden_states[0] → 从节点 1
Step 4: 从节点 1 执行 Layer 8-15 → hidden_states[1]
Step 5: TCP 发送 hidden_states[1] → 从节点 2
Step 6: 从节点 2 执行 Layer 16-23 + LM Head → logits
Step 7: logits → token → TCP 回传主节点
Step 8: 主节点将 token 追加到生成序列
Step 9: 以新 token 作为输入，重复 Step 2-8（直到 EOS 或 max_tokens）
```

### 4.2.2 张量传输格式

中间隐藏状态：
```python
hidden_states: torch.Tensor  # shape: (1, seq_len, 2048)
# dtype: float16 (2 bytes)
# 单 token 解码阶段: (1, 1, 2048) → 4 KB
# Prefill 阶段 (seq_len=512): (1, 512, 2048) → 2 MB
```

序列化方式：
- **小张量 (< 1MB)**: `torch.save` → `io.BytesIO` → 二进制 buffer
- **大张量 (≥ 1MB)**: `tensor.numpy().tobytes()` + shape/dtype 头 → 零拷贝传输
- 使用已有的 TCP `4字节长度头 + 数据体` 协议

## 4.3 实施步骤

### 阶段 1: ModelManager 层范围加载 `load_layer_range()`（~4 小时）

**文件**: `src/model_module.py`

**新增方法**:
```python
def load_layer_range(self, start_layer: int, end_layer: int,
                     has_embedding: bool, has_lm_head: bool,
                     model_path: str = None) -> None:
    """
    仅加载模型的指定层范围（用于分布式流水线）。

    Args:
        start_layer: 起始层编号（0-based，含）
        end_layer: 结束层编号（0-based，不含）
        has_embedding: 是否包含 Token Embedding 层
        has_lm_head: 是否包含 LM Head（输出层）

    实现方式:
        1. 加载完整模型配置（config.json）
        2. 提取 model.layers[start_layer:end_layer]
        3. 根据需要保留/丢弃 Embedding 和 LM Head
        4. 将不需要的层设为 None，释放显存
        5. 标记 self._layer_slice = (start, end)
    """
```

**关键技术点**:
- Qwen-1.8B 结构: `Embedding(2048×151936)` → `24×TransformerBlock` → `LM Head(2048×151936)`
- 每层 TransformerBlock 约 70MB (FP16)，Embedding + LM Head 各约 580MB
- 层范围加载后，每个节点的模型显存从 ~3.6GB 降至:
  - 首节点（Embedding + 8层）: ~1.1GB
  - 中间节点（8层）: ~560MB
  - 末节点（8层 + LM Head）: ~1.1GB

### 阶段 2: 单步前向传播接口 `forward_layers()`（~2 小时）

**文件**: `src/model_module.py`

**新增方法**:
```python
def forward_layers(self, hidden_states: torch.Tensor,
                   attention_mask: torch.Tensor = None,
                   past_key_values: tuple = None,
                   use_cache: bool = True) -> dict:
    """
    执行本节点负责的层范围的前向传播。

    Args:
        hidden_states: 输入隐藏状态
            - 首节点: 传入 input_ids (long tensor)
            - 后续节点: 传入上一节点的 hidden_states
        attention_mask: 注意力掩码
        past_key_values: 已缓存的 KV（用于增量解码）
        use_cache: 是否返回新的 KV 缓存

    Returns:
        {
            "hidden_states": Tensor,     # 输出的隐藏状态
            "logits": Tensor | None,     # 仅有 LM Head 的节点返回
            "past_key_values": tuple,    # 仅 use_cache=True 时返回
        }
    """
```

**关键逻辑**:
1. 如果是首节点 → `hidden_states = self.model.embed_tokens(input_ids)` → 遍历 `layers[start:end]`
2. 如果是中间/末节点 → 直接遍历 `layers[start:end]`
3. 如果是末节点 → `logits = self.model.lm_head(hidden_states)`

**KV Cache 约束**:
- 每层 Transformer 的 KV Cache 仅存在于计算该层的节点
- 不跨节点传输 KV Cache（阻止特征泄露 + 减少网络开销）
- Prefill 阶段每个节点自行构建 KV Cache
- Decode 阶段每个节点基于自己的 KV Cache 增量追加

### 阶段 3: 流水线调度引擎 `run_pipeline()`（~5 小时）

**文件**: `src/scheduler.py`

**新增方法**:
```python
def run_pipeline(self, prompt: str, max_new_tokens: int = 512,
                 temperature: float = 0.7, top_p: float = 0.9,
                 session_id: str = None) -> dict:
    """
    主节点：协调多节点流水线推理。

    流程:
        1. 获取当前分层配置 (get_layer_assignments)
        2. 向首节点发送 LAYER_FORWARD（prompt + 层范围）
        3. 首节点执行 embedding + forward → 返回 hidden_states
        4. 主节点将 hidden_states 转发给下一个节点
        5. 重复 3-4 直到末节点
        6. 末节点返回 logits → 主节点解码 token
        7. 判断 EOS / max_tokens → 继续生成或结束

    Returns:
        {"response": str, "thinking": str, "metrics": dict, ...}
    """
```

**新增 TCP 消息类型**:
```python
# 主节点 → 从节点：执行层前向传播
MSG_LAYER_FORWARD = "LAYER_FORWARD"
# payload: {
#     task_id,           # 任务唯一标识
#     hidden_states,     # 序列化后的张量（首节点为 None，用 input_ids 替代）
#     input_ids,         # 仅首节点：tokenized prompt
#     attention_mask,    # 注意力掩码
#     max_new_tokens,    # 最大生成 token 数
#     temperature,       # 温度参数
#     step,              # 当前生成步数（0=prefill, 1+ = decode）
# }

# 从节点 → 主节点：层前向传播完成
MSG_LAYER_RESULT = "LAYER_RESULT"
# payload: {
#     task_id,
#     hidden_states,     # 本节点输出（末节点为 logits）
#     node_id,           # 完成计算的节点
#     metrics: {         # 性能指标
#         time_ms,
#         memory_allocated_gb,
#         tokens_processed,
#     }
# }

# 主节点 → 从节点：流水线任务完成 / 取消
MSG_PIPELINE_DONE = "PIPELINE_DONE"
MSG_PIPELINE_ABORT = "PIPELINE_ABORT"
```

**从节点 LAYER_FORWARD 处理逻辑**:
```python
def _handle_layer_forward(self, client_id: str, msg: dict):
    """从节点接收 LAYER_FORWARD → 执行本节点层 → 返回 LAYER_RESULT"""
    data = msg["data"]
    task_id = data["task_id"]
    step = data["step"]  # 0 = prefill, 1+ = decode

    # 反序列化输入
    if step == 0:
        input_ids = torch.tensor(data["input_ids"])
        hidden_states = None  # 首节点从 input_ids 开始
    else:
        hidden_states = deserialize_tensor(data["hidden_states"])

    # 执行前向传播
    result = self._model_manager.forward_layers(
        hidden_states=hidden_states,
        input_ids=input_ids,
        attention_mask=data.get("attention_mask"),
        past_key_values=self._kv_cache.get(task_id) if step > 0 else None,
    )

    # 缓存 KV（用于下一次 decode step）
    if result.get("past_key_values"):
        self._kv_cache[task_id] = result["past_key_values"]

    # 序列化输出
    self._tcp_client.send_message({
        "type": "LAYER_RESULT",
        "task_id": task_id,
        "hidden_states": serialize_tensor(result["hidden_states"]),
        "logits": serialize_tensor(result.get("logits")),
        "node_id": NODE_ID,
        "metrics": { ... },
    })
```

### 阶段 4: 自动回退机制（~1 小时）

当从节点不可用或模型未加载层范围时，自动回退到全层主节点模式：

```python
def run_pipeline_safe(self, prompt, **kwargs):
    """带自动回退的流水线推理"""
    try:
        # 检查所有节点是否在线且已加载层范围
        if not self._all_pipeline_nodes_ready():
            logger.warning("部分节点未就绪，回退到全层主节点模式")
            return self._run_full_model_inference(prompt, **kwargs)
        return self.run_pipeline(prompt, **kwargs)
    except (TCPTimeout, ConnectionError) as e:
        logger.error(f"流水线节点通信失败: {e}，回退到全层主节点模式")
        self._mark_node_degraded(failed_node_id)
        return self._run_full_model_inference(prompt, **kwargs)
```

## 4.4 技术难点与对策

### 难点 1: 首 Token 延迟（Time to First Token）

**问题**: 多节点串联导致首 Token 延迟 = Σ(各节点 prefill 时间 + 网络传输时间)  
**预估**: 3 节点 × (200ms prefill + 50ms 传输) = ~750ms（vs 单节点 ~300ms）  
**对策**:
- 使用 `torch.save` + `io.BytesIO` 零拷贝序列化（< 5ms）
- 预连接 TCP（保持长连接，避免每次握手）
- 小 prompt（< 128 tokens）时自动回退到单节点

### 难点 2: KV Cache 一致性

**问题**: 每个节点独立维护自己的 KV Cache，切换 session 时需全部清空  
**对策**:
- KV Cache 用 `task_id` 索引，不依赖 session
- 流水线任务完成后主节点广播 `PIPELINE_DONE`，各节点清理对应的 KV Cache
- 增加 `MSG_CLEAR_KV` 消息用于手动清理

### 难点 3: 内存管理

**问题**: 每节点同时缓存多个 task 的 KV Cache 可能 OOM  
**对策**:
- 限制并发流水线任务数 = 1（同类型推理串行执行）
- 不同用户的请求排队，单流水线复用
- 设置 KV Cache 总量上限（PagedKV 的 `MAX_PAGE_NUM`）

### 难点 4: 异构设备性能差异

**问题**: 强节点等弱节点，流水线气泡（bubble）导致吞吐量下降  
**对策**:
- `compute_layer_assignment()` 已实现按算力比例分配层数
- 弱节点分配到更少的层，强节点分配到更多的层
- 引入 micro-batch pipeline（未来优化项）：将单次推理拆成多个 micro-batch

## 4.8 后续扩展方向（v0.3.0+）

1. **Micro-batch Pipeline**: 将 prompt 拆成多个 micro-batch，减少流水线气泡
2. **动态负载均衡**: 推理过程中根据实时 GPU 利用率动态调整层分配
3. **张量压缩传输**: 对 hidden_states 做 INT8 量化后再传输（~4KB → ~2KB）
4. **异步流水线**: 不同请求的 prefill/decode 阶段交叠执行，提升吞吐
5. **多备份节点**: 同一层范围部署在 2+ 节点，节点故障时立即切换
6. **图转树拓扑编排**: 集群 ≥ 5 节点时引入最大带宽生成树 + DFS 约束搜索替代简单比例分配（详见 [图算法](图算法.md)）

## 4.9 低成本辅助改进（与流水线同步实现）

以下两个改进改动量小、独立性强，建议在流水线实施的阶段 3（流水线调度引擎）中一并完成，
为后续图转树拓扑编排储备数据。

### 改进 1: RTT 心跳采集（~30 分钟）

**目标**: 在心跳机制中增加往返延迟采集，让 `NodeInfo` 拥有实时网络延迟数据。

**当前状态**:
- `tcp_comm.py` 的 `_heartbeat_loop` 仅单向发送心跳包
- 主节点仅知道"多久没收到心跳"（`last_heartbeat` 差值），不知道实际 RTT

**改动**:

1. **心跳回显协议** — 主节点收到 `HEARTBEAT` 后立即回复 `HEARTBEAT_ACK`（含服务端时间戳）

2. **TCPClient 增加 RTT 测量** — 客户端发送心跳时记录 `t_send`，收到 ACK 后计算 `rtt = time.time() - t_send`

3. **NodeInfo 新增字段**:
   ```python
   @dataclass
   class NodeInfo:
       # ... 现有字段 ...
       avg_rtt_ms: float = 0.0      # 滑动平均 RTT（指数加权）
       last_rtt_ms: float = 0.0     # 最近一次 RTT
       bandwidth_mbps: float = 0.0  # 预留：链路带宽估算（未来 iperf 探测填充）
   ```

4. **主节点广播时附带 RTT** — `NODE_LIST_SYNC` 和 `NODE_UPDATE` 中包含 `avg_rtt_ms`

**文件**: `src/tcp_comm.py`（新增 `HEARTBEAT_ACK` 消息类型 + RTT 计算）、`src/scheduler.py`（NodeInfo 字段 + 广播附带）

**价值**:
- 前端 AdminPanel 可展示节点间延迟
- 为 `_compute_node_weight()` 增加网络质量维度
- 为图转树算法的「链路带宽/延迟」输入做准备

---

### 改进 2: 显存约束分层（~1 小时）

**目标**: `compute_layer_assignment()` 增加显存下限检查，防止单节点被分配到装不下的层数。

**当前状态**:
- 按算力比例分配层数，不检查单节点 VRAM 是否足够
- Qwen-1.8B 每层 ~70MB (FP16)，3 节点各 8 层绰绰有余
- 但未来换更大模型（7B 级别，每层 ~300MB）会出问题

**改动**:

1. **新增配置**:
   ```python
   # config.py
   MIN_VRAM_PER_LAYER_MB = 70   # 单层最低显存（Qwen-1.8B FP16）
   # 不同量化精度下的修正系数
   LAYER_VRAM_FACTOR = {
       "fp16": 1.0,   # 70 MB/layer
       "int8": 0.55,  # 38 MB/layer
       "int4": 0.35,  # 25 MB/layer
   }
   ```

2. **`compute_layer_assignment()` 增加显存检查**:
   ```python
   def _check_vram_constraint(self, node_id: str, start: int, end: int,
                               has_embedding: bool, has_lm_head: bool) -> bool:
       """检查节点是否有足够显存承载分配的层范围"""
       vram_available = self._get_node_vram_mb(node_id)
       layers_count = end - start
       vram_needed = layers_count * LAYER_VRAM_PER_MB
       if has_embedding:
           vram_needed += EMBEDDING_VRAM_MB
       if has_lm_head:
           vram_needed += LM_HEAD_VRAM_MB
       return vram_available >= vram_needed * 1.1  # 10% 余量
   ```

3. **不满足时的回退策略**:
   - 将超额层重新分配给 VRAM 最充裕的节点
   - 若所有节点都不满足 → 发出警告 + 回退到全量主节点模式

**文件**: `src/scheduler.py`（新增 `_check_vram_constraint`）、`src/config.py`（新增常量）

**价值**:
- 换 7B 模型时不会静默 OOM
- Android 手机节点加入时自动减层（手机可用 RAM 仅 2-3GB）

## 5.1 限制 1：无 KV Cache — ✅ 已完成 (2026-06-14)

### 实施方案

**关键技术决策**:
- `forward_layers()` 使用 **DynamicCache** 作为 KV cache 载体（兼容 Qwen2 SDPA/Flash Attention）
- 分布式节点**临时修改 `self_attn.layer_idx`** 为本地索引（0..N-1），使 DynamicCache 按连续本地索引存储
- KV cache **不跨节点传输** — 各节点独立维护 `_kv_cache: dict[task_id, tuple]`
- 返回值 `past_key_values` 为 **tuple of (key, value)**，兼容旧接口

### ~~当前行为~~ → 已修复

每个 decode step 将**完整序列**（prompt + 所有已生成 token）重新发送给首节点，
各节点从头计算所有层的 hidden_states。

### 成本分析

| 指标 | 全序列重计算（当前） | KV Cache（目标） |
|------|---------------------|-------------------|
| decode step 计算量 | O(seq_len × layers) ≈ O(n²) | O(1 × layers) = O(1) |
| 通信量 (hidden_states) | (1, seq_len, 2048) FP16 | (1, 1, 2048) FP16 |
| 100 token 输出总通信 | ~800KB × 100 = 80MB | ~4KB × 100 = 400KB |
| 500 token 输出总通信 | ~4MB × 500 = 2GB | ~4KB × 500 = 2MB |
| 首 Token 延迟 | 无额外影响 | 无额外影响 |
| 每 Token 延迟 | 随 seq 增长线性恶化 | 恒定 |

### 改进方案

**阶段 A: 单节点 KV Cache（~3 小时）**

在 `forward_layers()` 中增加 `past_key_values` 支持：

```python
def forward_layers(self, input_ids=None, hidden_states=None,
                   attention_mask=None, position_ids=None,
                   past_key_values=None, use_cache=True):
    """
    新增参数:
        past_key_values: tuple of (key, value) per layer, 或 None
        use_cache: 是否返回新的 KV cache
    """
    with torch.no_grad():
        # ... embedding ...
        
        new_kv = [] if use_cache else None
        for i, layer in enumerate(transformer.layers):
            layer_output = layer(
                hidden_states,
                attention_mask=causal_mask,
                position_ids=position_ids,
                position_embeddings=position_embeddings,
                past_key_value=past_key_values[i] if past_key_values else None,
                use_cache=use_cache,
                cache_position=cache_position,
            )
            hidden_states = layer_output[0]
            if use_cache:
                new_kv.append(layer_output[1])  # (key, value)
        
        return {
            "hidden_states": hidden_states,
            "logits": logits if has_lm_head else None,
            "past_key_values": tuple(new_kv) if use_cache else None,
        }
```

**阶段 B: 分布式 KV Cache 管理（~2 小时）**

- 每个节点维护 `self._kv_cache: dict[task_id, past_key_values]`
- Prefill (step 0): 各节点自行构建 KV cache
- Decode (step 1+): 主节点仅发送**最后一个 token** 的 `input_ids`
  - 首节点取 `input_ids (1,1)` + 缓存的 KV → forward → 输出 (1,1) hidden_states + 新 KV
  - 中间节点取 hidden_states (1,1) + 缓存 KV → forward → 输出 + 新 KV
  - 末节点取 hidden_states + 缓存 KV → forward → logits
- Pipeline 完成/取消时广播 `PIPELINE_DONE` 清理各节点缓存

**阶段 C: PagedKV 分页管理（~2 小时，可选）**

- 复用 [config.py](../src/config.py) 已有的 `PAGE_SIZE=128` / `MAX_PAGE_NUM=256` 配置
- 引入 vLLM 风格的逻辑分页，避免 KV cache 碎片化
- 优先级低 — Qwen-1.8B 的 KV cache 较小（24层 × 2 × 8KB/token ≈ 384KB/token, 4K seq ≈ 1.5GB）

## 5.2 限制 2：串行节点通信 — 主节点中转

### 当前行为

```
Master → Worker1 (LAYER_FORWARD)
Master ← Worker1 (LAYER_RESULT)
Master → Worker2 (LAYER_FORWARD, with Worker1's hidden_states)
Master ← Worker2 (LAYER_RESULT)
Master → Worker3 (LAYER_FORWARD, with Worker2's hidden_states)
Master ← Worker3 (LAYER_RESULT, with logits)
```

每个 step 有 2×N 次网络传输（N = 从节点数），所有数据经主节点中转。

### 成本分析

| 场景 | 当前中转模式 | 直连模式（目标） |
|------|-------------|-----------------|
| 3 节点、1 token decode | 主↔W1 + 主↔W2 + 主↔W3 = 6 次传输 | W1→W2 + W2→W3 + W3→主 = 3 次传输 |
| 每 step 主节点带宽消耗 | 2 × N × 4KB | 0（主节点仅收最终 logits） |
| 延迟贡献 | Σ(2×RTT_i) | RTT_12 + RTT_23 + RTT_3M |

### 改进方案

**方案 A: 链式直连（推荐，~4 小时）**

1. **节点间直连** — 每个从节点同时是下游节点的 TCP 服务端
   - Worker1 开放一个临时端口，Worker2 连接 Worker1
   - Worker2 开放一个临时端口，Worker3 连接 Worker2
   
2. **流水线令牌传递** — 主节点仅启动链条，后续节点自动接力：
   ```
   Master: 发送 LAYER_FORWARD(task_id, input_ids) → Worker1
   Worker1: forward() → 直连发送 LAYER_FORWARD(task_id, hidden_states) → Worker2
   Worker2: forward() → 直连发送 LAYER_FORWARD(task_id, hidden_states) → Worker3
   Worker3: forward() → 发送 LAYER_RESULT(task_id, logits) → Master
   ```

3. **节点发现与连接图** — 主节点在 `push_layer_config_to_clients()` 时附带拓扑信息：
   ```python
   # 主节点发送给 Worker1 的 layer_config 中包含下游节点信息
   {
       "start_layer": 0, "end_layer": 8, ...
       "downstream": {"node_id": "client2", "host": "192.168.1.3", "port": 18889}
   }
   ```

**方案 B: 环形拓扑（备选，~3 小时）**

- 末节点也将 hidden_states 发回首节点（用于下一 token 的 decode）
- 适合节点数 ≥ 4 的场景

### 文件变更

| 文件 | 变更 |
|------|------|
| `src/tcp_comm.py` | TCPClient 支持同时连接多个 peer（上游 + 下游） |
| `src/scheduler.py` | `run_pipeline()` 改为仅发送首节点、等待末节点；新增拓扑发现 |
| `src/config.py` | 新增 `PIPELINE_PEER_PORT_RANGE` |

## 5.3 限制 3：单任务串行 — PIPELINE_MAX_CONCURRENT = 1

### 当前行为

同一时刻仅允许 1 个流水线推理任务。多用户并发请求时，后续请求需排队等待。

### 适用场景分析

- Qwen-1.8B 规模小（24 层，每层 70MB），三层流水线单次推理 ~2-5 秒
- 单用户场景（个人使用）：影响小
- 多用户场景（团队共用）：排队延迟 = (N_requests - 1) × avg_inference_time

### 改进方案

**方案 A: 请求队列 + 快速切换（~2 小时）**

```python
class PipelineQueue:
    """流水线请求队列 — FIFO + 超时"""
    
    def __init__(self):
        self._queue = asyncio.Queue(maxsize=10)
        self._current_task = None
    
    async def submit(self, prompt, **kwargs) -> dict:
        """提交推理请求，等待结果"""
        future = asyncio.Future()
        await self._queue.put((prompt, kwargs, future))
        return await future
    
    async def _worker(self):
        """后台协程：逐个处理队列中的请求"""
        while True:
            prompt, kwargs, future = await self._queue.get()
            try:
                result = self._run_pipeline(prompt, **kwargs)
                future.set_result(result)
            except Exception as e:
                future.set_exception(e)
```

**方案 B: Micro-batch Pipeline（~5 小时，远期）**

- 将多个请求的 prefill/decode 阶段交叠执行
- 请求 A 在 Worker1 计算时，请求 B 可以进入 Worker2
- 需要更复杂的任务调度器和流水线气泡填充算法

**方案 C: 多模型实例（~1 小时，大显存设备）**

- 在显存充裕的主节点（≥ 8GB VRAM）上运行独立模型实例
- 流水线 + 单机实例并行服务不同请求

### 推荐优先级

| 改进 | 优先级 | 理由 |
|------|--------|------|
| KV Cache | 🔴 最高 | 性能影响最大 — 当前 O(n²) 计算量在长回复场景不可接受 |
| 请求队列 | 🟡 中 | 多用户场景的可用性问题，单用户影响小 |
| 节点直连 | 🟢 低 | 6 次 vs 3 次传输的差异对 4KB 级别的 decode 通信影响不大；prefill 阶段 2MB 传输有收益 |
