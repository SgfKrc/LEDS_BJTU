## 1. 当前架构概述

### 1.1 节点角色定位

```
┌──────────────────────────────────────────────────┐
│                   PC 节点（参与推理）                │
│  ┌─────────┐   ┌─────────┐   ┌─────────┐         │
│  │ Master  │──▶│ Slave 1 │──▶│ Slave 2 │  ...    │
│  │ (主节点) │   │ (从节点) │   │ (从节点) │         │
│  │ PyTorch │   │ PyTorch │   │ PyTorch │         │
│  │ 层0-7   │   │ 层8-15  │   │ 层16-23 │         │
│  └─────────┘   └─────────┘   └─────────┘         │
│       ▲             层拆分流水线                     │
│       │  HTTP                                     │
├───────┼──────────────────────────────────────────┤
│       │           Android 节点（不参与层拆分）       │
│  ┌────┴────┐          ┌──────────────┐            │
│  │ 全有 Mode│          │ 全无 Mode     │            │
│  │ 本地推理  │          │ HTTP 瘦客户端  │            │
│  │ llama.cpp│         │ 零计算        │            │
│  │ 完整模型  │          │ 转发给 Master │            │
│  └─────────┘          └──────────────┘            │
└──────────────────────────────────────────────────┘
```

- **PC 节点**：PyTorch 引擎，参与层拆分流水线，可担任 master 或 client
- **Android 节点**：llama.cpp 引擎，**只能担任 client**，不参与层拆分。全有=本地跑完整模型，全无=HTTP 瘦客户端。将来探索 llama.cpp 层拆分

### 1.2 两条推理路径共存（互不通信）

```
                   POST /api/chat
                        │
         ┌──────────────┴──────────────┐
         ▼                              ▼
┌─────────────────┐          ┌─────────────────────┐
│ 路径 A: 从节点转发 │          │ 路径 B: 流水线编排    │
│ role == "client" │          │ role == "master"    │
│                 │          │ engine == "pytorch" │
│ Slave → INFER_  │          │                     │
│ FORWARD → Master│          │ Master → Slave1     │
│                 │          │   → Slave2 → Slave3  │
│ Master 本地跑    │          │   → Master          │
│ 全部24层        │          │                     │
│                 │          │ 链式拓扑 + KV Cache  │
│ 从节点贡献: 0层  │          │ 从节点各跑分配层      │
└─────────────────┘          └─────────────────────┘
```

## 2. 不完善之处

### P0 — 结构性问题

#### 2.1 路径 A 从未接入流水线

[`handle_infer_forward()`](src/scheduler.py#L3474) 的 docstring 自述：

> TODO: 真正的分布式流水线（模型层拆分到多个节点并行推理）
> 当前先实现「全部层都在主节点」模式，让从节点转发可用。

**现状**：从节点（client）发送 `INFER_FORWARD` 到主节点后，主节点在**本地**跑全部 24 层。流水线（路径 B）完全被绕过。从节点只是把 prompt 转发了一下，主节点自己全干了。

**影响**：
- 主节点过载 — 所有从节点的请求都压在主节点一台机器上
- 从节点形同虚设 — 除了转发没做任何推理
- 路径 A 和路径 B 是两套独立代码，维护成本高

#### 2.2 缺少节点类型 (`node_type`)

`NodeInfo` 只有 `role: master | client`，没有区分设备平台。

**Android 角色定位**（全有或全无）：

| 模式 | 描述 | 推理方式 | 参与流水线层拆分 |
|------|------|---------|-----------------|
| **全有** (Mode A) | Android 本地跑完整模型 | llama.cpp 本地推理，不依赖 PC | ❌ 不参与 |
| **全无** (Mode B) | Android 瘦客户端 | HTTP 请求转发给 PC 主节点 | ❌ 不参与 |
| **将来** | 探索 llama.cpp 层拆分 | 可能参与流水线 | ❓ 远期探索 |

**结论**：Android 现阶段**不参与任何层拆分流水线**。只有 PC 节点（主/从）参与分布式推理。

因此 `node_type` 应区分：

| node_type | 可担任 role | 参与层拆分 | 使用引擎 | 
|-----------|------------|--------|----------|
| `pc` | master / client | ✅ | PyTorch CUDA |
| `android` | client 仅 | ❌ | llama.cpp（本地全量 或 不跑） |

**影响**：如果 Android 节点注册到集群时未标记 `node_type`，会被 `compute_layer_assignment()` 当作 PC 节点分配 Transformer 层，但它用 llama.cpp 做不到层拆分。Android 全有节点应注册为 `node_type=android`，主节点只记录其存在（用于状态展示和将来的独立请求分发），不将其纳入流水线。

#### 2.3 流水线仅限 PyTorch 引擎

[`api_server.py` 路由逻辑](src/api_server.py#L1535-L1538)：

```python
if (... and model_manager._engine_type == "pytorch"):
    pipeline_result = scheduler.run_pipeline_safe(...)
```

主节点用 llama.cpp(GGUF) 时，即使有从节点在线，也走本地推理。所有从节点闲置。

#### 2.4 CHAIN_FORWARD 无回退

链式转发（从节点间直连）失败时，当前代码只做了一层处理：

```python
# scheduler.py:3819-3822 — 当前逻辑
ok = self._send_chain_forward(chain_next["node_id"], chain_data)
if not ok:
    # 直连失败 → 直接向主节点报错，整个 step 失败
    self._send_layer_result("master", task_id, error=f"链式转发到 {chain_next['node_id']} 失败")
```

**缺少的回退层级**：

| 层级 | 策略 | 当前 |
|------|------|------|
| L1 | Slave1 → Slave2 直连 | ✅ 已实现 |
| L2 | Slave1 → **Master 中转** → Slave2 | ❌ 缺失 |
| L3 | Master 中转也失败 → **寻找其他可用从节点**替代 | ❌ 缺失 |
| L4 | 所有替代节点不可用 → 主节点全量回退 | ✅ 已有（整个流水线回退） |

完整回退链路应该是：
1. 尝试 Slave1 → Slave2 直连
2. 失败 → 退回 `Slave1 → Master → Slave2`（主节点做 relay，流水线不中断）
3. Master → Slave2 也失败 → Master 扫描其他在线 PC 从节点，选一个显存足够的替代 Slave2
4. 所有替代方案穷尽 → 整个流水线回退到主节点全量推理

### P1 — 功能缺失

#### 2.5 无流式输出 (SSE/Streaming)

当前 `/api/chat` 返回完整 `ChatResponse`。用户等全部 token 生成完才看到结果。

| 场景 | 当前体验 | 流式体验 |
|------|---------|---------|
| 生成 512 token (~30s) | 30s 白屏 → 一次性显示 | 逐 token 显示，< 1s 看到首 token |
| 移动端弱网 | 超时风险高 | 渐进渲染，超时也能看到部分结果 |

**需要**：SSE endpoint `/api/chat/stream`，`run_pipeline()` 每个 decode step 产出一个 token 时推送。

#### 2.6 缺少流水线就绪检查

`run_pipeline()` 直接开始推理，没有预先验证：
- 每个从节点的层范围是否加载成功
- 模型文件是否存在
- 节点是否真的处于 ONLINE 状态（而不只是心跳没超时）

任一步失败才回退，浪费了 prefill 计算。

#### 2.7 并发问题

| 路径 | 并发控制 | 问题 |
|------|---------|------|
| 路径 A | 无 — 每个 INFER_FORWARD 直接 `Thread.start()` | GPU OOM 风险 |
| 路径 B | PipelineQueue FIFO 串行 | 正确但无超时控制 |
| 跨路径 | 无协调 | A 和 B 可能同时跑，争抢 GPU |

#### 2.8 无模型分发机制

从节点需要自己提前准备好模型文件。主节点推送 `LAYER_CONFIG` 时假设模型已存在。

缺少：
- 主节点推送配置前检查从节点模型状态
- 局域网内模型文件分发（从主节点 rsync / HTTP 下载）
- 模型校验（SHA256 确保从节点模型与主节点一致）

---

### P2 — 体验与健壮性

#### 2.9 错误恢复粒度粗

`run_pipeline_safe()` 的自动回退是「全部或全无」：
- 一个节点出错 → 整个流水线回退到主节点全量推理
- 没有节点级 bypass（跳过故障节点，其余节点继续）

#### 2.10 缺少分布式推理指标

前端 / API 缺少：
- 每个节点的每层耗时（prefill 耗时 / decode 耗时 / 网络传输时间）
- KV Cache 命中率 / 大小
- 流水线气泡时间（节点等待上游数据的时间）

#### 2.11 跨节点安全

当前 TCP 连接无任何认证：
- 任何知道 IP:PORT 的客户端都能注册为从节点
- 恶意节点可注入伪造的 hidden_states，操纵推理结果
- 无 TLS 加密，hidden_states（含 prompt 语义信息）明文传输

### 阶段 2：CHAIN_FORWARD 多级回退（~3h）【P0】

**目标**：从节点间直连失败时，逐级回退，最大化流水线存活率。

**改动**：

1. `scheduler.py` — `_send_chain_forward()` 增加三级回退：
   ```python
   def _send_chain_forward(self, target_node_id: str, data: dict,
                           chain_remaining: list = None) -> tuple[bool, str]:
       """
       Returns: (success, fallback_level)
         - (True, "direct")       — 直连成功
         - (True, "master_relay") — 主节点中转成功
         - (True, "alt_node")     — 替代节点成功
         - (False, "exhausted")   — 所有方案失败
       """
       # L1: 直连
       if self._try_direct_send(target_node_id, data):
           return True, "direct"
       
       logger.warning(f"直连 {target_node_id} 失败，尝试 L2 主节点中转")
       
       # L2: 主节点中转
       if self._try_master_relay(target_node_id, data):
           return True, "master_relay"
       
       logger.warning(f"主节点中转 {target_node_id} 也失败，尝试 L3 寻找替代节点")
       
       # L3: 寻找替代从节点
       alt_node = self._find_alternative_slave(target_node_id, chain_remaining)
       if alt_node:
           logger.info(f"找到替代节点 {alt_node}，重建链式拓扑")
           if self._try_direct_send(alt_node, data):
               return True, "alt_node"
       
       # L4: 所有方案穷尽
       return False, "exhausted"
   ```

2. `scheduler.py` — 新增 `_find_alternative_slave()`：
   ```python
   def _find_alternative_slave(self, failed_node_id, chain_remaining):
       """从在线 PC 从节点中找到显存足够替代 failed_node 的节点"""
       failed_layers = self._node_layer_configs.get(failed_node_id, {})
       required_layers = failed_layers.get("layers_count", 0)
       
       for nid, info in self.nodes.items():
           if nid == failed_node_id:
               continue
           if nid in [r.get("node_id") for r in (chain_remaining or [])]:
               continue  # 已在链中，不能重复
           if not info.is_available():
               continue
           if info.node_type != "pc":
               continue  # 只选 PC 节点
           if self._can_handle_layers(nid, required_layers):
               return nid
       return None
   ```

3. `scheduler.py` — `_handle_layer_forward()` 中调用处改为多级回退：
   ```python
   ok, level = self._send_chain_forward(chain_next["node_id"], chain_data, chain_remaining)
   if not ok:
       # L4 穷尽，才向主节点报错
       self._send_layer_result("master", task_id, error=...)
   ```

**文件**: `src/scheduler.py`

### 阶段 4：流水线就绪检查（~1.5h）【P1】

**目标**：推理前验证所有节点就绪，避免浪费 prefill。

1. `scheduler.py` — 新增 `_verify_pipeline_readiness()`：
   ```python
   def _verify_pipeline_readiness(self, pipeline_nodes):
       for node in pipeline_nodes:
           # 1. 检查心跳新鲜度（< 10s）
           if time.time() - node.last_heartbeat > 10:
               return False, f"{node.node_id} 心跳超时"
           # 2. 检查层范围是否已加载
           cfg = self._node_layer_configs.get(node.node_id)
           if not cfg:
               return False, f"{node.node_id} 未收到分层配置"
       return True, "ok"
   ```

### 阶段 5：并发安全（~2h）【P1】

**目标**：防止路径 A 和路径 B 同时占用 GPU。

**改动**：

1. `scheduler.py` — 统一推理入口加锁：
   ```python
   class Scheduler:
       _inference_lock = threading.Lock()
       
       def run_pipeline_safe(self, prompt, **kwargs):
           if not self._inference_lock.acquire(timeout=0.1):
               return {"error": "推理引擎正忙，请稍后重试"}
           try:
               return self.run_pipeline(prompt, **kwargs)
           finally:
               self._inference_lock.release()
   ```

2. `handle_infer_forward()` — 改为入队而非直接启动线程

**文件**: `src/scheduler.py`
