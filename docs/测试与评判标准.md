# 测试与评判标准

> 状态：现行
> 更新日期：2026-07-14
> 适用范围：本地推理、当前 PyTorch 层流水线以及后续独立执行 Provider

## 1. 测试记录的必要信息

每次结果必须同时记录：

- 逻辑模型、模型工件哈希、格式、量化方式、tokenizer 和上下文长度；
- 节点设备画像，包括 CPU、GPU/NPU、内存、操作系统和运行引擎；
- 实际执行模式和参与节点，不能只写“分布式：是/否”；
- 网络路径、RTT、吞吐、丢包/重连情况以及是否经过 Tailscale；
- Prompt token、生成 token、并发数、随机参数和是否流式输出；
- 是否发生重试、降级、节点替换或模型重新加载。

设备可以是台式机、工作站、服务器、笔记本、Android 手机或平板。测试分组按能力和执行模式建立，不按设备外形建立。

## 2. 当前对照组

| 组别 | 执行方式 | 用途 |
|------|----------|------|
| 本地基线 | 单节点完整模型、默认优化 | 正确性与最终降级基线 |
| 单机优化 | 不同量化、编译和 KV 策略 | 测量内存、速度和质量变化 |
| PyTorch 层流水线 | 同一模型工件、不同层分配与节点路径 | 验证容量、吞吐和分层合理性 |
| 故障与降级 | 注入断连、超时、模型不一致、节点崩溃 | 验证失败边界和恢复能力 |
| 异构客户端 | Android 本地 GGUF、Android 远程 PC | 验证端到端产品路径，不宣称参与层拆分 |

完整 Worker、任务链、张量并行或 GGUF Stage 落地后应分别增加独立对照组，不能并入“层流水线”统计。

## 3. 核心指标

### 3.1 正确性

- 层覆盖连续、无重叠、无缺层；Embedding 和 LM Head 所有权正确。
- 流水线与完整模型在确定性设置下达到约定的 token/logit 一致性。
- 参与者使用同一认证模型工件；不兼容节点在执行前被排除。
- 用户看到的流式文本不重复、不乱序，失败 attempt 的迟到结果被丢弃。

### 3.2 性能

- 首 token 延迟、逐 token 延迟、端到端时延和 tokens/s；
- 单请求延迟与多请求吞吐分别统计；
- CPU/GPU/内存/显存占用和模型加载时间；
- 每个节点的计算时间、等待时间、传输字节和慢节点占比；
- 编排、模型同步、层加载与 ready 等准备时间单独统计。

### 3.3 稳定性与容错

- 节点发现失败、数据库不可用、TCP 断开和模型同步失败均不能拖垮本地推理入口；
- 未 ready 节点不能收到层任务；取消/超时后释放 KV、锁、租约和队列状态；
- 故障恢复必须标明是同 attempt 续跑、整次重试还是降级重启；
- 长时间运行检查内存增长、线程泄漏、重复注册和任务统计漂移。

### 3.4 输出质量

- 固定问题集上的任务正确率、格式遵循率和人工偏好；
- 不同量化/Provider 的输出差异与拒答、幻觉情况；
- 质量结论必须绑定具体模型工件，不能推广到同名的其他格式或量化版本。

## 4. 自动化与真实环境验收

- 单元与组件测试：`tests/test_model_module.py`、`tests/test_scheduler.py`、`tests/test_tcp_comm.py`、`tests/test_graph_orchestrator.py`。
- 分布式逻辑回归：`tests/test_distributed_inference.py`。
- 多进程和真实模型路径：`tests/simulation/`，执行约束见[分布式推理仿真测试计划](分布式推理仿真测试计划.md)。
- PC 安装包和 Android release 产物需要在对应真实设备上做启动、模型加载、联网、断网和升级回归。

Mock 测试只能证明状态机和错误分支，不能替代真实模型、真实 TCP 连接与异构设备的端到端结论。

## 5. 展示要求

前端和报告至少展示 `execution_mode`、`provider`、`attempt_id`、参与节点、实际层段/Stage、回退原因和各节点任务计数。只有确实进入相应执行 Provider 的任务才能计入其分布式统计。
