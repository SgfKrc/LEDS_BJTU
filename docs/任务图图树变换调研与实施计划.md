# 任务图图树变换调研与实施计划

> 状态：规划（调研结论已形成，尚未实现）
>
> 更新日期：2026-07-14
>
> 适用范围：任务链/DAG 的语义优化、调度分析和 Stage 级容错；不适用于 PyTorch 层流水线的通信拓扑编排

## 1. 结论

任务图是任务链拆分的核心数据结构，但 QLH **不应把逻辑任务 DAG 直接转换成普通树作为执行真相**。

原因是任务 DAG 中常见共享前驱和汇合节点。把 DAG 展开成树会复制共享子图，通常增加计算、模型调用和 payload 传输；从 DAG 中任取生成树又会删除真实依赖，破坏结果语义。当前[图算法](图算法.md)中的“通信图转最大带宽生成树”只用于缩小 PyTorch 层流水线的候选路径搜索空间，不能直接套到任务链。

有必要借鉴图树算法思想，但应拆成三个彼此隔离的方向：

1. **执行 DAG 等价优化**：裁剪不可达 Stage、安全传递约简、公共子任务合并、Stage 融合和共享 payload 引用，用于减冗余。
2. **树形分析视图**：从 DAG 派生支配树、后支配树、关键路径骨架或有界树分解，用于优先级、取消域、检查点和规划分析；它们不替代执行 DAG。
3. **容错 attempt 图**：在逻辑 Stage 之外受控增加重试、fallback 和推测执行 attempt，用租约、epoch 和唯一提交者提高鲁棒性；不能随意给逻辑 DAG 加边。

推荐的长期结构是：

```text
LogicalTaskDAG（用户/模板语义，唯一事实来源）
        |
        +--> OptimizedTaskDAG（保持语义的约简与融合）
        |
        +--> AnalysisForest（支配树、关键路径、可选树分解）
        |
        +--> ExecutionAttemptGraph（重试、fallback、推测执行）
                         |
                         +--> ProviderTopologyGraph（节点与网络选择）
```

近期优先级应是执行 DAG 优化框架和 attempt 容错边界，而不是通用树分解。当前 `dual_candidate` 只有三个 Stage，规划耗时远小于模型推理耗时；近期真正的性能收益仍来自多 Worker 并行和 payload 共享。

## 2. 必须区分的“图”和“树”

| 名称 | 顶点/边含义 | 是否参与执行语义 | 用途 |
|------|-------------|------------------|------|
| 逻辑任务 DAG | Stage 与业务依赖 | 是，唯一事实来源 | 表达输入、依赖、权限、失败策略 |
| 优化任务 DAG | 等价改写后的 Stage 与依赖 | 是，执行前冻结 | 减少无效工作和调度开销 |
| 通信拓扑图 | Provider/节点与链路 | 否 | 节点选择和网络路径估算 |
| 最大带宽生成树 | 通信拓扑的候选子图 | 否 | PyTorch 层流水线候选路径编排 |
| 支配树/后支配树 | DAG 中“所有路径必经”关系 | 否，派生视图 | 检查点、失败域、汇合屏障分析 |
| 树分解 | 由相关 Stage 组成的 bag 树 | 否，派生视图 | 复杂约束下的动态规划 |
| attempt 图 | Stage 的具体执行尝试和切换关系 | 是，但不改变 Stage 依赖 | 重试、fallback、推测执行和提交仲裁 |

这些结构必须使用不同的类型、ID 命名空间和持久化字段。不得把 Provider 拓扑边误当作 Stage 依赖，也不得把失败重试写成逻辑 DAG 的回边。

## 3. 为什么不能直接 DAG 转树

### 3.1 展开成树会复制共享子图

典型菱形任务图：

```text
              +--> candidate_a --+
shared_input -+                   +--> aggregate
              +--> candidate_b --+
```

树要求每个非根节点只有一个父节点。若按路径展开，`shared_input` 或 `aggregate` 必须被复制。前者会重复检索/预处理，后者会改变汇合语义。对模型 Stage 而言，复制一次可能意味着一次完整推理，远比少量图调度开销昂贵。

正确做法是保留 fan-out/fan-in，并让两个下游引用同一个不可变 `payload_ref`，而不是复制 payload 或 Stage。

### 3.2 生成树会丢业务依赖

通信图的生成树只需要保持节点连通；任务 DAG 的每条边可能表示：

- 下游必须读取的特定输入；
- schema、数据作用域或权限传递；
- join/barrier 条件；
- 失败传播、允许 partial 或 fallback 的策略；
- provenance 和审计关系。

删除一条边即使仍保持图论可达，也不一定保持这些语义。因此不能对任务 DAG 使用普通最小/最大生成树。

### 3.3 树分解不是“把执行图变成树”

树分解把多个相关顶点放进相互重叠的 bag，并要求包含每个顶点的 bag 连通。它适合在低 treewidth 图上做规划动态程序，但：

- bag 可能随 treewidth 增大而迅速变大；
- 它只应作为规划器内部分析结构；
- 它不减少模型推理本身的计算量；
- 当前固定模板太小，引入它的实现和验证成本高于收益。

因此树分解只保留为后期、有基准触发条件的实验项。

## 4. 可落地的 DAG 减冗余方案

### 4.1 规范化和裁剪

优化前先完成确定性规范化：

- 校验无环、唯一 Stage ID、依赖存在、最终 Stage 可达；
- 规范化 schema、配置、模型 artifact、采样参数和数据作用域；
- 从最终输出反向裁剪不可达 Stage；
- 对条件分支只裁剪在规划时已确定且无副作用的死分支；
- 保存优化前图、语义摘要和变换记录。

这对应成熟任务图系统常见的 `cull` 思路。它对模型辅助生成的较大任务草案有价值，对当前三个固定 Stage 通常是 no-op。

### 4.2 安全传递约简

图论传递约简可以在保持可达关系的前提下删除冗余边，例如 `A -> B -> C` 已存在时删除纯依赖边 `A -> C`。

QLH 必须使用“带语义的传递约简”，只有同时满足以下条件才能删边：

- 被删除边不提供独有 `input_ref` 或字段映射；
- 不承载额外 schema、权限、数据作用域和 provenance；
- 不改变 join、失败传播、partial、fallback 或取消语义；
- 替代路径上的 payload 可完整、确定地传递；
- 变换后通过结构与语义等价测试。

普通图库的传递约简通常只处理节点和可达性，甚至不会自动保留节点/边属性，不能直接对生产任务图原地应用。

### 4.3 公共子任务合并

两个 Stage 只有具有相同语义指纹且允许共享时才能合并。建议指纹至少包括：

```text
stage_fingerprint = hash(
  stage_type,
  normalized_input_refs_and_digests,
  input_and_output_schema_versions,
  logical_model_id,
  artifact_id_and_revision,
  provider_requirements,
  normalized_generation_or_tool_config,
  data_scope,
  side_effect_class,
  determinism_and_share_policy
)
```

默认允许合并：确定性、纯函数、输入不可变、输出可复用的 `transform` 或检索后处理 Stage。

默认禁止合并：

- 有外部副作用的工具调用；
- 依赖当前时间、随机数或可变外部状态的 Stage；
- 不同租户、会话或数据作用域的 Stage；
- 未锁定 artifact/revision 的模型调用；
- 需要独立多样性的随机采样候选。

当前 `dual_candidate` 的 `candidate_a` 和 `candidate_b` 使用不同指令，且目标就是获得独立候选，绝不能因基础 messages 相同而合并。即使两个 prompt 完全相同，只要采样策略要求独立随机结果，也必须保留两个 Stage。

### 4.4 Stage 融合

可把连续、廉价、同 Provider、同权限域的纯变换 Stage 合并，以减少序列化、排队和远程往返。

不得跨越以下边界融合：

- 模型或 Provider 切换；
- checkpoint 或结果提交边界；
- 独立重试、超时、取消或 fallback 边界；
- 安全域、租户或数据作用域边界；
- 外部副作用；
- 需要单独观测和计费的 Stage。

融合会降低独立重试能力和放置灵活性，因此必须记录 `original_stage_ids`，并允许通过开关回退未融合图。

### 4.5 共享 payload 引用

对当前工程最实用的减冗余不是改图形状，而是让 fan-out 分支共享不可变输入：

- 大 messages、文档和检索结果只提交一次；
- 下游传递 `payload_ref + sha256 + schema_version`；
- Worker 按权限和依赖读取，不在控制消息中复制正文；
- 以内容摘要去重本地 blob，但不得跨数据作用域泄露缓存命中信息；
- 引用计数或 TTL 负责最终清理。

这能减少主从节点传输和内存复制，同时保留原始 DAG 语义。

## 5. 树形分析视图

### 5.1 支配树与后支配树

若从唯一入口出发的每条到达 Stage `N` 的路径都经过 `D`，则 `D` 支配 `N`。反向从唯一出口分析可得到后支配关系。

对任务链的价值：

- 支配节点适合作为检查点或失败影响域的边界；
- 后支配节点可识别必经的 verify/aggregate 提交屏障；
- 支配子树可辅助级联取消和资源回收；
- 结合 Stage 成本可决定哪些结果值得持久化。

它是分析结果，不改变依赖，也不能据此跳过 Stage。

### 5.2 关键路径骨架

以预计执行时长、排队、传输和 prepare 成本作为权重，对 DAG 计算最长完成路径。调度器优先处理关键路径上的 ready Stage，并把非关键分支放入剩余并发预算。

历史实测不足时必须保守：使用 Stage 类型基线和区间估计，不能把不稳定的单次耗时当成长期节点评分。每次计划同时记录预计和实际关键路径，供 shadow planning 校准。

### 5.3 有界树分解的触发条件

仅在同时满足以下条件时进入实验：

- 已支持模型辅助生成任务图，真实图规模明显超过固定模板；
- Stage 数、跨分支约束和 Provider 组合使规划耗时成为可测瓶颈；
- 基线规划器在目标 P95 时限内无法找到足够好的计划；
- 实测图的 treewidth 较低，bag 规模受配置上限约束；
- 超限或超时可立即回退普通 DAG 启发式调度。

当前建议触发阈值：真实生产图 `stage_count >= 32` 且规划 P95 超过 50 ms，再通过基准决定是否继续；阈值是实验门槛，不是性能承诺。

## 6. 从“树”扩展为鲁棒执行图

鲁棒性不应通过修改逻辑依赖实现，而应为每个逻辑 Stage 建立独立 attempt 组：

```text
Logical Stage S
      |
      +--> attempt S/epoch-1/node-a (primary) -----+
      |                                            |
      +--> attempt S/epoch-1/node-b (speculative) -+--> commit one winner
      |                                            |
      +--> attempt S/epoch-2/node-c (retry) --------+
```

### 6.1 重试和 fallback

- attempt 失败或租约过期后，增加 `lease_epoch` 并创建新 `attempt_id`；
- 重试从 Stage 输入或已提交 checkpoint 开始，不接续另一 Provider 的半成品状态；
- 旧 epoch 的迟到结果必须被 fencing 拒绝；
- fallback Provider 必须重新通过 artifact、schema、权限和数据作用域硬过滤；
- 逻辑 DAG 保持无环，重试关系只存在于 attempt 图或事件日志中。

### 6.2 推测执行

只对纯函数、可取消、结果可仲裁且成本允许的慢 Stage 启动第二 attempt：

- 原 attempt 超过动态 P95/截止时间风险阈值；
- 有空闲且故障域不同的兼容 Provider；
- 预计尾延迟收益大于额外模型调用和传输成本；
- 同一 Stage 仍只允许一个原子提交赢家；
- 赢家提交后取消其他 attempt，并统计浪费算力。

不能对外部副作用 Stage 做无幂等保护的推测执行，也不应在资源已经饱和时用副本进一步加剧排队。

### 6.3 冗余分支与降级结果

允许模板显式描述“主分支 + fallback 分支”或“多个候选 + 最少成功数”，但必须定义：

- `min_successful_inputs` 和 deadline；
- 哪些错误允许 partial；
- 降级结果的 schema 和用户可见标记；
- 未完成分支的取消和清理；
- aggregate 是否能区分缺失、失败与被取消。

这种受控冗余能增强鲁棒性；任意增加依赖边只会制造死锁、重复执行和不可解释的失败传播。

## 7. 建议的数据边界

### 7.1 图版本与优化记录

```json
{
  "logical_graph_version": 1,
  "logical_graph_digest": "sha256:...",
  "optimizer_version": "task-dag-opt-v1",
  "optimized_graph_digest": "sha256:...",
  "optimization_trace": [
    {
      "rule": "cull_unreachable",
      "affected_stage_ids": ["unused_stage"],
      "reason": "not_reachable_from_final"
    }
  ]
}
```

每个优化 Stage 保存 `original_stage_ids`。工作流查询和故障报告应能从优化后 ID 追溯到用户/模板看到的逻辑 Stage。

### 7.2 语义边

边不能只是 `(source, target)`，至少需要表达：

```json
{
  "source_stage_id": "retrieve",
  "target_stage_id": "analyze",
  "input_bindings": {
    "documents": "retrieve.output.documents"
  },
  "schema_version": 1,
  "data_scope": "trusted_tailnet",
  "join_policy": "required",
  "failure_policy": "fail_target"
}
```

只有完整比较这些语义后，才能判断边是否冗余。

### 7.3 attempt 提交状态

建议补充：

- `attempt_group_id`：同一 Stage 的 primary/speculative/retry 组；
- `lease_epoch`：单调 fencing 版本；
- `commit_state`：`candidate|validating|winner|rejected|cancelled`；
- `winner_attempt_id`：Stage 的唯一提交者；
- `superseded_by`：旧 attempt 被哪个新 attempt 取代；
- `late_result_rejected` 和拒绝原因；
- `speculation_trigger`、额外成本和节省的尾延迟。

## 8. 优化流水线

建议采用固定、可审计顺序：

```text
validate_and_normalize(logical_dag)
  -> cull_unreachable
  -> safe_common_subtask_elimination
  -> semantic_transitive_reduction
  -> safe_stage_fusion
  -> materialize_shared_payload_refs
  -> validate_semantic_invariants
  -> freeze(optimized_dag)
  -> build_dominator_and_critical_path_views
  -> schedule_stage
  -> expand_execution_attempts_after_provider_selection
```

关键约束：

- 每条规则必须独立开关、版本化、可回退；
- 优化器只能在工作流开始前改写图，执行中不得原地改变已冻结依赖；
- 动态 fallback 通过 attempt/事件状态机表达，不原地重写逻辑图；
- 优化失败时回退已校验的原始 DAG，不应让整个请求失败；
- shadow 模式先记录建议优化，不改变实际执行。

## 9. 对当前 `dual_candidate` POC 的判断

当前结构是：

```text
candidate_a --+
              +--> aggregate
candidate_b --+
```

它没有可删除的传递边，也没有应合并的公共 Stage。近期可做的只有：

- 保持两个候选的独立采样与独立 attempt；
- 未来有多个 Worker 后并行执行两个 ready 候选；
- 将公共 messages 保存一次并通过只读引用传给两个候选；
- 把 `aggregate` 视为后支配/提交屏障，只有满足模板的成功输入数才运行；
- 为候选失败定义重试、fallback 或 partial 的明确策略；
- 使用 winner fencing 防止取消/重派后的迟到结果污染聚合。

因此，现在实现“通用图转树”不会加速该 POC，反而会增加错误面。应先完成 PC Full Worker、ready Stage 并行调度、租约和结果提交，再引入图优化器 shadow 模式。

## 10. 分阶段实施计划

### G0：术语、不变量与可观测快照

- 在类型层分开 Logical DAG、Optimized DAG、Analysis View、Attempt Graph 和 Provider Topology。
- 为 Stage、语义边、图摘要和优化 trace 定义稳定 schema。
- 保存优化前后快照，但查询接口只返回安全摘要，不泄露候选正文。
- 建立图规模、边数、规划耗时、payload bytes 和关键路径基线。

验收：任何执行记录都能说明执行的是哪个逻辑图和优化器版本；不同类型的边不能混用。

### G1：shadow 优化器框架

- 实现纯函数式 `optimize(graph) -> graph + trace` 接口。
- 先只做规范化、不可达裁剪建议和共享 payload 规划。
- 实际执行仍使用原图，对比两者结构与预计成本。
- 任一规则异常只禁用该次优化，不影响请求执行。

验收：连续压力下优化器异常不会中断任务链；trace 可复现，输入不变时输出摘要稳定。

### G2：安全约简与 payload 共享

- 上线已验证的不可达裁剪。
- 实现带边语义的传递约简，默认关闭，按模板逐个准入。
- 将大输入 fan-out 改为不可变 `payload_ref`。
- 保留原图回退开关和优化前后对照指标。

验收：菱形图只传一次共享 payload；有独有字段/权限/失败策略的边不会被删除；输出与原图基线等价。

### G3：公共子任务合并与安全融合

- 引入 Stage 语义指纹和 `determinism/share_policy`。
- 首先只开放纯 `transform` Stage 合并与融合。
- LLM 生成、工具副作用和跨作用域 Stage 默认禁止。
- 对融合后的日志、取消、计费和 provenance 做映射。

验收：重复纯变换只执行一次；随机候选仍独立；融合 Stage 失败能准确映射回原始 Stage。

### G4：支配树和关键路径调度

- 为单入口/单出口图生成支配与后支配视图；多入口/出口使用虚拟节点分析。
- 用历史区间估算关键路径，在 shadow 模式比较 ready queue 顺序。
- 验证后用于优先级、检查点建议和级联取消，不改变依赖语义。

验收：关键路径调度降低或不恶化目标工作流 P95；分析失败自动回退普通拓扑顺序。

### G5：容错 attempt 图

- 完成 lease、epoch、fencing、幂等提交和迟到结果拒绝。
- 支持同 Stage 的 retry 与 Provider fallback。
- 在严格准入和成本预算下实验推测执行。
- UI 区分逻辑 Stage 数、attempt 数、实际参与 Provider 和浪费成本。

验收：任一 Worker 断线只影响对应 attempt；重派后旧结果不能提交；推测执行永远只有一个赢家。

### G6：有条件的树分解实验

- 仅在真实大图达到第 5.3 节触发条件后实施。
- 对比普通 DAG 启发式、支配树/关键路径方案和有界 treewidth 规划。
- 限制 bag 大小、规划时间和内存，超限立即回退。

验收：必须在真实任务图上稳定改善计划质量或规划时延，否则 No-Go 并移除运行时依赖。

## 11. 测试矩阵

| 场景 | 必须验证的性质 |
|------|----------------|
| 菱形共享前驱 | 不复制共享 Stage，两个分支读取同一已授权 payload |
| 带冗余纯依赖边 | 安全约简后可达性和输出不变 |
| 带独有 input binding 的“传递边” | 不得删除 |
| 不可达纯 Stage | 可裁剪且无输出变化 |
| 不可达副作用 Stage | 模板校验应拒绝或显式处理，不能静默优化掉审计事实 |
| 两个相同纯 transform | 指纹相同时只执行一次 |
| 两个随机 LLM 候选 | 不合并、不共享随机状态 |
| 跨数据作用域 Stage | 不合并缓存，payload 不越权 |
| 融合链中间 Stage 超时 | 失败、取消和原始 ID 映射准确 |
| attempt 租约过期后重派 | 新 epoch 可提交，旧结果被拒绝 |
| primary 与 speculative 同时完成 | 只有一个 winner，统计不重复 |
| fallback Provider 不兼容 artifact | 调度前拒绝，不进入执行 |
| 优化器抛异常/超时 | 回退原始 DAG，工作流继续 |
| 分析树构建失败 | 使用普通 ready queue，不影响正确性 |
| 数据库离线 | 本地日志仍能恢复图版本、attempt 和提交者 |

还需使用随机 DAG 做属性测试：优化前后最终输出、必需输入、权限约束、失败传播和 side-effect 次数保持约定等价，而不只比较节点数或可达性。

## 12. 指标与 Go/No-Go

### 12.1 减冗余指标

- 原始/优化后 Stage 数与语义边数；
- 避免的 Stage 执行次数和模型调用数；
- payload 内联/传输字节和缓存命中；
- 规划 P50/P95/P99、优化器超时和回退率；
- 关键路径预计值与实际值误差；
- 端到端首结果和完成时延。

### 12.2 鲁棒性指标

- retry、fallback、speculative attempt 数；
- lease 过期、迟到结果拒绝和重复提交拦截数；
- Worker 故障后的恢复时间；
- partial/fallback 输出比例；
- 推测执行节省的尾延迟与额外 token、算力和网络成本；
- 实际参与节点和 Provider，不把仅在线节点计入。

### 12.3 Go/No-Go

执行 DAG 优化 Go：语义等价测试通过，目标工作负载确实减少模型调用、payload 或调度开销，且具备一键回退。

容错 attempt 图 Go：租约与唯一提交先通过故障注入，再开放 retry/fallback；推测执行还必须证明尾延迟收益超过额外成本。

树分解 Go：仅在大图规划已经成为瓶颈、真实 treewidth 受控且相对基线有稳定收益时采用。否则保持 No-Go。

## 13. 调研依据

- [Dask Task Graph Optimization](https://docs.dask.org/en/stable/optimize.html)：任务图常见的裁剪、融合、内联和重写规则。QLH 只借鉴优化类别，不直接复用其数据模型。
- [NetworkX `transitive_reduction`](https://networkx.org/documentation/stable/reference/algorithms/generated/networkx.algorithms.dag.transitive_reduction.html)：传递约简保持 DAG 的可达关系；其文档也提示结果默认不携带节点/边数据，说明生产图不能忽略边语义。
- [Aho, Garey, Ullman: The Transitive Reduction of a Directed Graph](https://doi.org/10.1137/0201008)：有向图传递约简的经典定义与算法基础。
- [Lengauer and Tarjan: A Fast Algorithm for Finding Dominators in a Flowgraph](https://doi.org/10.1145/357062.357071)：支配关系和高效算法基础。
- [MapReduce: Simplified Data Processing on Large Clusters](https://research.google/pubs/mapreduce-simplified-data-processing-on-large-clusters/)：任务重执行、backup task 和结果提交等分布式容错思想的参考。
- [Resilient Distributed Datasets](https://www.usenix.org/conference/nsdi12/technical-sessions/presentation/zaharia)：基于 lineage 重算丢失分区的容错参考。
- [三种分布式拆分细化实施方案](三种分布式拆分细化实施方案.md)：QLH 任务链的 Stage、Attempt、Provider、租约和验收边界。
- [图算法](图算法.md)：当前 PyTorch 层流水线通信拓扑的最大带宽生成树方案；与本文任务 DAG 优化严格分离。

外部系统和论文只提供算法与架构参考。QLH 的边携带业务 payload、权限和失败语义，是否可约简、融合或重算必须由本项目自己的 schema、故障注入和等价测试决定。

## 14. 最终决策

1. 保留逻辑任务 DAG 作为唯一执行语义来源，不实施通用“DAG 展开成执行树”。
2. 实施“DAG 等价优化 + 树形分析视图 + 容错 attempt 图”的三层方案。
3. 先完成 Full Worker、并行 ready Stage、lease/fencing 和 payload 引用，再做优化器 shadow 模式。
4. 传递约简、公共子任务合并和融合全部采用语义白名单，LLM 随机候选默认不合并。
5. 树分解延期到真实大任务图出现规划瓶颈后再评估；当前不增加运行时复杂度。
