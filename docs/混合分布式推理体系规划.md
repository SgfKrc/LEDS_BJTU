# 混合分布式推理体系规划

> 调研日期：2026-07-14
>
> 状态：架构规划与技术预研
>
> 范围：PC PyTorch/Safetensors、PC 与 Android llama.cpp/GGUF、完整推理 Worker、任务链、层间拆分和张量并行
>
> 目标：不再把所有设备强行塞进一条层间流水线，而是提供多种可独立部署、由统一控制面选择的分布式执行体系

---

## 1. 结论摘要

最终系统应成为**统一控制面 + 多种独立执行体系**的混合架构，而不是继续扩展一条万能流水线。

建议保留和建设五类执行方式：

| 执行方式 | 切分边界 | 主要模型/运行时 | 定位 |
|---|---|---|---|
| 本地完整推理 | 单节点内部 | PyTorch、llama.cpp/GGUF | 基线与最终降级 |
| 完整推理 Worker | prompt/messages -> 完整结果 | PC/Android 任意兼容引擎 | 多请求吞吐与异构设备利用 |
| 任务链/DAG 拆分 | 文本、JSON、检索结果等业务对象 | 可跨模型、跨引擎 | 异构分布式主路线 |
| 层间拆分流水线 | hidden states + KV/position 状态 | 当前 QLH PyTorch；可选独立 GGUF stage runtime | 模型容量扩展、低带宽跨机 |
| 张量并行 | 单层内 tensor shard + collective | 独立 MLX/CUDA/GGML tensor runtime | 低延迟，需要高速互联和同构设备 |

关键决策：

1. **原版 llama.cpp/GGUF 的跨主机层间拆分不作为 QLH 主路线。** 单进程 CPU/GPU offload 和同机多 GPU split 已有支持，但跨主机主要依赖仍处于 POC 的 GGML RPC，不能等同于稳定的分布式层流水线。
2. **Mesh-LLM 证明 GGUF 跨机层段可以实现，但代价是另一套运行时。** 它维护 llama.cpp fork、把 GGUF 预处理为逐层 package，并运行 Skippy stage 协议；这不是给原版 llama.cpp 加一个 CMake 开关。
3. **exo 证明张量并行和拓扑感知自动编排可以产品化，但当前技术重心是 MLX/Apple Silicon。** 它不是 GGUF/Android/Windows PC 的直接替代品，适合先作为外部执行引擎研究和接入。
4. **张量并行必须独立于现有层间流水线。** 它的通信是每层多次 collective，对低延迟、高带宽互联要求远高于逐层 activation handoff；不能复用当前 `LAYER_FORWARD` 协议。
5. **任务链是混合体系的上层组合器。** 一个 Stage 可以调用现有 PyTorch 层流水线，另一个 Stage 可以调用 exo、Mesh-LLM、Android 完整 Worker 或本地 llama.cpp，但 Stage 之间只传稳定业务对象。
6. **统一的是控制语义，不是内部 tensor ABI。** 统一任务身份、模型身份、租约、取消、指标和结果；各执行体系保留自己的模型格式、通信协议和容错边界。

## 2. 术语与边界

### 2.1 三种“拆分”不是同一件事

| 名称 | 拆分对象 | 每个生成 token 的通信特征 | 适合网络 | 主要限制 |
|---|---|---|---|---|
| 层间拆分 | 连续 Transformer 层段 | 每个 stage 边界传 activation；KV 通常留在层所有者 | 普通局域网可研究 | 最慢 stage 限制整体；运行时/模型语义必须一致 |
| 张量并行 | 每层矩阵和 KV/tensor 分片 | 每层需要 all-reduce/all-gather 等 collective | NVLink、RDMA、TB5、快速 PCIe/以太网 | 对延迟和带宽极敏感，通常要求同构设备 |
| 任务链拆分 | 推理或工具调用的业务 Stage | Stage 边界传文本/JSON/文档 | Wi-Fi、Tailscale 也可用 | 不是单次模型前向的容量聚合 |

完整推理 Worker 不拆模型，而是把完整请求分配给某台设备。它能增加集群吞吐和设备利用率，但不能让多个小内存节点共同装下一个超大模型。

### 2.2 `tensor-split` 不一定是张量并行

llama.cpp 中需要区分：

- `--tensor-split 3,1`：比例参数。在默认 `--split-mode layer` 下，它控制模型层/权重如何按比例放到设备，并不表示每层都被 tensor parallel。
- `--split-mode layer`：不同设备拥有连续或按规则分配的层，属于 pipeline/layer parallel。
- `--split-mode tensor`：真正把每层权重和 KV 跨设备分片，通过 meta device 和 collective 协作；当前上游仍标记为 experimental。
- 已废弃的 `row`：旧的部分 tensor split 路径，不应作为新方案。

因此文档和 UI 不能只看到 `tensor_split` 字段就显示“张量并行”。执行模式必须由 `split_mode` 和实际 backend 能力共同确定。

### 2.3 单机异构 offload 不等于跨机分布式

llama.cpp 在一个进程内可以让部分层运行于 CPU、部分层运行于 GPU，也可以使用同一主机上的多个 backend device。该路径共享进程地址空间、模型上下文和 backend scheduler，和通过网络连接多台设备不是同一种可靠性模型。

跨机后新增的问题包括：

- 权重分发和远端缓存。
- 协议版本和 GGML op 一致性。
- 网络断开时计算图中间状态的处置。
- KV cache 的远端所有权和恢复。
- 任务租约、取消、幂等和迟到结果。
- 远端服务的认证、输入限制和资源隔离。

不能因为单机 CPU/GPU split 可用，就推导跨机 split 同样可用。

## 3. 当前 QLH 基线

### 3.1 现有执行路径

| 路径 | 模型格式 | 设备 | 分布式方式 |
|---|---|---|---|
| PC CUDA | Safetensors/PyTorch | 主节点 + PC 从节点 | 自研连续层段流水线 |
| PC CPU/集显 | GGUF/llama.cpp | 单机 | 本地完整推理 |
| Android Full | GGUF/llama.cpp | Android 单机 | 本地完整推理 |
| Android Lite/Thin | 无本地模型 | 请求转发 PC | 不执行模型 |

当前 `scheduler.py` 的层协议绑定 PyTorch 模型语义，不能直接接收 llama.cpp/GGUF stage。当前 Android 也上报 `pipeline_worker=false`。

### 3.2 当前层间流水线的保留价值

现有 PyTorch 层间流水线仍应保留：

- 与项目当前核心实验目标一致。
- 可在普通 PC 节点间按层分配 Safetensors 模型。
- 相比 tensor parallel，跨节点 collective 次数少，对普通网络更友好。
- 已有模型同步、层配置 ACK、任务取消、迟到结果过滤和回退基础。

它不应承担以下职责：

- Android GGUF 节点的接入。
- exo/MLX tensor parallel 的内部通信。
- Mesh-LLM/Skippy stage 的内部协议。
- 业务任务 DAG 的编排。

## 4. GGUF 与原版 llama.cpp 的分布式判断

### 4.1 可直接采用的能力

- 单节点 CPU 完整推理。
- 单节点 GPU offload，通过 `n_gpu_layers` 控制 GPU 层数。
- 同一进程可见的多 GPU `split-mode=layer`。
- 支持架构和后端满足条件时，实验性 `split-mode=tensor`。
- GGML RPC 暴露远端 CPU/GPU backend device，并由主运行时调度远端计算。

### 4.2 为什么跨机层间拆分很可能走不通

“走不通”不是指完全无法生成一次结果，而是指很难达到 QLH 正式产品需要的可维护性和收益：

1. 原版 llama.cpp 不提供面向 QLH 的稳定“载入第 N-M 层 -> 接收 activation -> 返回 activation”应用 API。
2. GGML RPC 是 backend 级远程图执行，不是带任务租约、重试和业务错误分类的集群协议。
3. 锁定版本上游明确把 RPC 标记为 proof-of-concept、脆弱且不安全。
4. RPC 与 llama.cpp/GGML op 和协议版本高度耦合，升级要求两端严格一致。
5. Android 生命周期、Wi-Fi 抖动和普通 Tailscale RTT 不适合托管远程计算图的关键 tensor step。
6. 当模型本来能在某台设备完整运行时，跨机 activation/tensor 往返可能比本机 CPU/GPU offload 更慢。
7. 当模型无法在任一设备完整运行时，还要解决远端权重驻留、KV 状态和断线后的重新 materialize。

因此原版 GGML RPC 仅保留实验价值，不直接改造成正式 QLH GGUF 层流水线。

### 4.3 llama.cpp 原生 tensor parallel 的限制

锁定版本的 `docs/multi-gpu.md` 将 `split-mode=tensor` 标记为实验性，并说明：

- 每层需要多次跨 GPU reduction，互联速度是核心瓶颈。
- 主要对 NVIDIA CUDA 多 GPU 给出性能预期，其他 backend 不保证。
- 要求 flash attention 可用。
- KV cache 暂时要求非量化类型。
- 不是所有模型架构都在 allow-list，DeepSeek2 等 MoE/hybrid 架构被明确排除。
- auto-fit 在 tensor 模式不可用，需要人工控制 context、并发和 GPU layers。

这条路线适合先研究**同一台 PC 的多 GPU**或具有 NCCL/RCCL/P2P 的紧耦合集群，而不是 Android + 普通 PC 的松耦合网络。

## 5. exo 调研

### 5.1 项目定位

[exo](https://github.com/exo-explore/exo) 是独立的分布式推理系统。调研时上游仓库：

- commit：`b5375f8cee4368d09e1ce96a56b9f81fb0bc81aa`
- 版本：`0.3.70`
- 许可证：Apache-2.0
- 核心运行时：MLX、MLX Distributed 和 exo 自身的拓扑/实例编排

其官方说明包含：

- 自动设备发现和拓扑感知 placement。
- pipeline parallel 和 tensor parallel。
- Tensor parallel 使用 collective，并强调 Apple Silicon + Thunderbolt RDMA 场景。
- OpenAI、Claude、Responses 和 Ollama 兼容 API。
- 模型实例预览、创建、就绪等待和删除生命周期。

### 5.2 对 QLH 有价值的设计

- 把 topology、placement 和 execution instance 分离。
- 在部署前预览 placement，而不是边生成边猜测节点。
- tensor parallel 与 pipeline parallel 是明确不同的 sharding 类型。
- 模型实例具有异步创建和 ready gate。
- 控制面可以在无算力节点运行，worker/runner 独立。
- 使用统一 API 暴露一个独立执行集群。

这些模式可用于改进 QLH 的执行计划和状态机，即使不引入 exo 代码。

### 5.3 不能直接采用的原因

- 当前主力模型是 MLX/Hugging Face 模型卡，不是本项目的 GGUF 路径。
- 上游最强支持面是 Apple Silicon/macOS；README 对 Linux GPU 的定位仍偏开发中。
- 项目依赖 Python 3.13、Rust bindings 和一组定制 MLX 依赖，直接嵌入当前 Windows 打包环境成本高。
- 没有 Android 产品路径。
- 其 tensor parallel 的收益依赖高速互联；普通 Wi-Fi/Tailscale 集群不具备同样条件。

### 5.4 建议接入方式

exo 应先作为**独立 Tensor/Pipeline Provider**，通过 OpenAI 兼容 API 接入任务链：

```text
QLH TaskGraph Stage
    -> ExoProvider
    -> exo /v1/chat/completions
    -> 标准化 StageResult
```

不复制 exo 的 MLX tensor 协议，也不让 exo 节点注册为 QLH `pytorch_layer_worker`。

适合验证的硬件是现有或可获得的 Apple Silicon 多机集群；没有该硬件时只做 API adapter 和设计借鉴，不承诺性能复现。

## 6. Mesh-LLM 调研

### 6.1 同名澄清

本文讨论的是 [Mesh-LLM/mesh-llm](https://github.com/Mesh-LLM/mesh-llm)，不是同名的 3D Mesh 生成论文项目。

调研时上游仓库：

- commit：`c39cb3b06c658aac1379f04c82572ed26acff1d8`
- workspace 版本：`0.72.1`
- 仓库元数据许可证：Apache-2.0；workspace 声明 `MIT OR Apache-2.0`
- 核心实现：Rust 控制面 + Skippy/llama.cpp stage runtime + GGUF layer package

### 6.2 Skippy 实际做法

Mesh-LLM 不是直接拿一个完整 GGUF 调用原版 llama.cpp RPC。其 split 路线包括：

1. 把完整/分片 GGUF 预处理为 layer package。
2. package 保存 `model-package.json`、共享 metadata、embedding、output head 和逐层 GGUF 文件。
3. coordinator 根据节点内存和拓扑分配连续层范围。
4. 每个节点只下载和 materialize 自己需要的层。
5. 下游 stage 先启动，全部 ready 后才发布 stage 0 路由。
6. stage runtime 传递 activation frame，并维护模型家族相关的 KV/recurrent state 策略。

这证明 GGUF 层间拆分可实现，但前提是：

- 自定义 GGUF 分层制品格式。
- 自定义 llama.cpp fork/stage API。
- 独立协议、拓扑规划、cache 和 correctness certification。
- 按模型家族维护 parity/例外矩阵。

这恰好说明它不是当前 `llama.cpp + add_subdirectory()` 的自然延伸。

### 6.3 与张量拆分的关系

Skippy 是连续层段流水线，不是单层内 tensor parallel。它与 QLH 当前 PyTorch 层流水线在“按层段”概念上相似，但模型制品、运行时、activation/KV 协议和模型支持矩阵完全不同。

Mesh-LLM 还包含完整模型路由和实验性 Mixture-of-Agents，但这些属于完整 Worker/任务编排层，也不是张量并行。

### 6.4 对 QLH 的可用价值

- Windows、Linux、macOS 以及多种 CPU/GPU backend 的独立 release 思路。
- GGUF layer package、immutable revision、SHA-256 和原子 materialization。
- stage readiness、下游先启动和 stage 0 最后发布。
- 模型家族 certification，而不是宣称“任意 GGUF 都能分层执行”。
- 远端节点只下载所需层，解决每个从节点先下载完整模型的问题。
- 独立 OpenAI 兼容 API，适合作为 sidecar provider。

### 6.5 风险与接入结论

- 上游 README 仍把自身定义为 experimental distributed-systems software。
- 需要维护 llama.cpp fork 和大量 Rust/native 组件。
- layer package 必须预处理，不能直接复用当前 QLH 单文件 GGUF 发布流程。
- 当前发布矩阵未列 Android；不能据 PC ARM/Linux 支持推导 Android APK 可用。
- 模型 family support、wire dtype 和 KV/recurrent state 存在细粒度例外。

建议先把 Mesh-LLM 作为**独立 GGUFStageProvider sidecar**评估：QLH 负责认证、任务链和统一 UI，Mesh-LLM 负责其内部 layer package 与 stage transport。验证成功前不 fork 它的代码进 `src/scheduler.py`，也不替换当前 vendored llama.cpp。

## 7. 三种核心分布式体系比较

具体数据结构、控制协议、故障边界、验收门槛与分阶段实施见[三种分布式拆分细化实施方案](三种分布式拆分细化实施方案.md)。其中 PyTorch 层间拆分不再扩写实现方案，只保留真实多设备待测试清单。

| 维度 | QLH PyTorch 层流水线 | 任务链/DAG | 独立张量并行 | GGUF/Skippy 层段 |
|---|---|---|---|---|
| 切分单位 | Transformer 连续层段 | 业务 Stage | 单层 tensor shard | GGUF 连续层 package |
| 模型格式 | Safetensors/PyTorch | 任意 | MLX/CUDA/GGML 专用制品 | 预处理 GGUF layer package |
| 设备同构要求 | 中 | 低 | 高 | 中 |
| 网络要求 | 中 | 低 | 极高 | 中 |
| 单模型容量聚合 | 是 | 否 | 是 | 是 |
| 单请求低延迟潜力 | 中 | 取决于工作流 | 高速互联下高 | 中 |
| 多请求吞吐 | 可配合 microbatch | 高 | 高 | 可配合 batch |
| Android 近期可用 | 否 | 是 | 否 | 未验证 |
| 当前成熟度 | 已有实现，继续修正 | 待实现 | 外部引擎预研 | 外部 sidecar 预研 |
| 失败恢复边界 | pipeline attempt | Stage attempt | 整个 tensor instance/request | Skippy stage/session |

没有一种模式对所有场景最优。调度器必须根据模型、目标、设备和网络选择执行体系。

## 8. 目标混合架构

### 8.1 统一控制面与隔离数据面

```text
API / Android / PC UI
          |
          v
Request Coordinator
  - task_id / attempt_id
  - model identity
  - cancellation / lease
  - policy / accounting
          |
          v
Execution Planner
  +-- LocalProvider
  +-- FullWorkerProvider
  +-- TaskGraphProvider
  +-- PyTorchLayerProvider
  +-- TensorParallelProvider (exo 或未来 CUDA provider)
  +-- GGUFStageProvider (Mesh-LLM/Skippy sidecar)
          |
          v
各 Provider 自己的数据面与进程边界
```

统一控制面只定义：

- 请求、Stage、attempt 和 lease 身份。
- 模型逻辑 ID、artifact ID、revision 和摘要。
- 能力发现、准入、健康和容量。
- 开始、进度、取消、完成、失败和超时状态。
- 用户可见输出和标准 metrics。

禁止统一控制面尝试定义一个跨 PyTorch、MLX 和 GGML 通用的内部 tensor 格式。

### 8.2 Provider 接口

建议抽象：

```python
class ExecutionProvider:
    def inspect(self) -> ProviderCapabilities: ...
    def plan(self, request, candidates) -> ExecutionPlan: ...
    def prepare(self, plan) -> PrepareResult: ...
    def execute(self, attempt, on_event) -> ExecutionResult: ...
    def cancel(self, attempt_id) -> None: ...
    def release(self, plan_id) -> None: ...
```

`prepare()` 必须和 `execute()` 分开，允许：

- PyTorch workers 先完成 layer config ACK。
- exo instance 先创建并 await ready。
- Mesh-LLM stages 先 materialize 且全部 ready。
- Android Worker 先确认模型已加载、温控和电量准入。

### 8.3 模型制品身份

同一个逻辑模型可以有多个不可混用的 artifact：

```json
{
  "logical_model_id": "qwen-1.8b-chat",
  "artifacts": [
    {
      "artifact_id": "qwen-1.8b-safetensors-int4",
      "format": "safetensors",
      "provider": "pytorch_layer",
      "sha256": "..."
    },
    {
      "artifact_id": "qwen-1.8b-q4km-gguf",
      "format": "gguf",
      "provider": "local_or_full_worker",
      "sha256": "..."
    },
    {
      "artifact_id": "qwen-1.8b-skippy-package",
      "format": "gguf_layer_package",
      "provider": "gguf_stage",
      "revision": "immutable-ref",
      "manifest_sha256": "..."
    },
    {
      "artifact_id": "qwen-mlx-4bit",
      "format": "mlx",
      "provider": "tensor_parallel_exo",
      "revision": "..."
    }
  ]
}
```

逻辑模型名称相同不代表输出数值、tokenizer 行为或量化误差完全一致。任务链可以跨 artifact，但单个 tensor/层执行 attempt 只能使用同一 provider 认证过的 artifact 集合。

## 9. 执行模式选择策略

### 9.1 显式模式

API 和实验面板应支持：

```text
auto
local
full_worker
task_graph
pytorch_layer
tensor_parallel
gguf_stage
```

用户选择显式模式时，系统应验证能力并返回准确失败原因，不能静默换成另一模式后仍显示原模式参与。

### 9.2 Auto 决策顺序

建议按以下约束选择，而不是只比较一个节点分数：

1. 根据逻辑模型筛选存在且通过认证的 artifact/provider。
2. 根据请求目标筛选：低延迟、批量吞吐、超大模型容量、离线或异构利用。
3. 检查 provider ready 状态和模型驻留，避免请求到达后才大规模下载。
4. 根据网络矩阵判断是否允许 tensor parallel 或层段跨机。
5. 估算端到端完成时间，包括 prepare、prefill、decode、网络和排队。
6. 选择收益超过最快单节点基线且满足可靠性门槛的计划。

推荐策略：

| 场景 | 首选 | 次选 |
|---|---|---|
| 模型单机可装下、交互低延迟 | 最快本地/完整 Worker | PC 层流水线 |
| 多个独立请求 | 完整 Worker 池 | task graph fan-out |
| 异构 Android + PC 协作 | 任务链 | 完整 Worker 池 |
| 普通局域网、Safetensors 超出单卡 | PyTorch 层流水线 | 本地 CPU/offload |
| Apple Silicon + TB5/RDMA | exo tensor provider | exo pipeline provider |
| 高速同构 CUDA 多 GPU | 独立 tensor provider | PyTorch/llama layer mode |
| GGUF 超出单机内存 | Mesh-LLM sidecar 实验 | 更大单机/更小量化 |
| Wi-Fi/Tailscale 移动节点 | 完整 Worker/任务链 | 不选 tensor parallel |

### 9.3 收益门槛

分布式不应只以“成功使用多节点”为成功。至少比较：

- 首 token 延迟。
- prefill tok/s 和 decode tok/s。
- 端到端 tokens/s。
- 峰值内存与各节点驻留。
- 模型准备时间和缓存命中率。
- 网络 bytes/token、RTT 和重传。
- 30 分钟持续吞吐和温控。
- 节点失败后的恢复时间和重复输出风险。

若某分布式模式没有容量收益，且端到端性能持续低于最快单节点，应默认不选。

## 10. 容错与降级边界

### 10.1 只在 attempt 边界切换 Provider

不能把执行到一半的 MLX tensor attempt、GGUF stage attempt 或 PyTorch hidden-state attempt交给另一 provider 继续。失败后只能：

1. 取消并清理原 attempt。
2. 判断是否已经向用户输出不可撤回 token。
3. 在允许重放时创建新的 attempt。
4. 从 prompt/messages 或最近业务 checkpoint 重启。

### 10.2 流式输出约束

- 未输出 token：可安全切换 provider 重试。
- 已输出 token：默认不得从头换引擎重放，避免重复或语义跳变。
- 任务链 Stage：只有 Stage 输出尚未提交给下游时可无感重试。
- 外部副作用 Stage：必须依赖幂等键或人工确认。

### 10.3 Provider 故障域

| Provider | 最小故障域 | 降级方式 |
|---|---|---|
| PyTorch 层流水线 | 整个 pipeline attempt | 清理 KV，重新规划或本地完整重试 |
| exo tensor | 整个 model instance/request | 等待实例恢复或换 provider 重启 |
| Mesh-LLM stage | 整个 split session/request | 由 sidecar 恢复；失败则换 provider 重启 |
| Android 完整 Worker | 单个 Worker attempt | 租约过期后重派 |
| 任务链 | 单个 Stage attempt | 从 checkpoint 重试或降级分支 |

## 12. 分阶段实施路线

本节只保留混合体系的总体顺序。任务链与张量并行的可执行阶段、依赖和验收条件以[三种分布式拆分细化实施方案](三种分布式拆分细化实施方案.md)为准。

### H0：明确当前边界

- 现有 PyTorch 层流水线继续独立运行。
- llama.cpp/GGUF 继续本地完整推理。
- Android 不加入 PyTorch layer assignment。
- 统一文档中的 layer、tensor、task-chain 术语。

### H1：Provider 网关与模型 artifact 清单

- 建立 `ExecutionProvider` 接口和 provider registry。
- 模型配置从单路径扩展为逻辑模型 + 多 artifact。
- 统一 `task_id/attempt_id/lease/cancel/result`。
- 先把现有本地推理和 PyTorch 层流水线包装为两个 provider，保持行为不变。

验收：显式模式选择和实际统计一致，旧 API 默认行为不回归。

### H2：完整 Worker 与任务链

- 实现 Android/PC 完整 Worker 池。
- 实现业务 `TaskGraphScheduler`。
- 支持 Stage checkpoint、并行分支、重派和聚合。
- 验证双候选生成 + 主节点评审流程。

验收：异构节点通过文本/JSON 边界协作，任一 Worker 失败不拖垮整条工作流。

### H3：exo 外部 Provider 实验

- 只实现 OpenAI API adapter、健康检查、模型映射和 metrics。
- 在有 Apple Silicon 集群时分别测 pipeline 与 tensor placement。
- 不修改当前 PyTorch socket 协议。

验收：QLH 可以把一个完整 Stage 交给 exo，并准确显示其 provider/placement；无对应硬件时不宣称性能收益。

### H4：Mesh-LLM GGUF Stage Provider 实验

- 独立进程/容器部署固定版本 Mesh-LLM。
- 使用 immutable layer package revision。
- 先选 Qwen dense 小模型做 correctness 和断线测试，再考虑大模型容量验证。
- QLH 只通过 API 和管理接口控制，不引入其 llama.cpp fork。

验收：多个 PC 节点只下载所需层并完成 GGUF split 推理；失败不会污染现有 QLH llama.cpp/PyTorch 状态。

### H5：独立 TensorParallelProvider

- 根据硬件决定 exo/MLX 或同构 CUDA provider。
- 要求高速互联基准和模型架构 allow-list。
- 独立模型实例生命周期、collective health 和进程隔离。
- 不将 Android/Wi-Fi 节点纳入 tensor group。

验收：相同模型与请求下，端到端延迟或容量相对单节点达到预设收益，节点故障可终止并清理整个 instance。

### H6：Auto planner

- 基于 artifact、目标、ready、网络和历史指标选择 provider。
- 先 shadow planning，只记录建议而不改变执行。
- 数据稳定后逐步开放自动选择。

## 13. Go/No-Go 条件

### 13.1 Mesh-LLM

Go：

- 目标 Windows/Linux PC backend 有可复现 release。
- 目标模型 family 在上游 certification 矩阵中通过。
- 固定 layer package revision 可离线复现。
- 相比单节点确实解决容量问题，且持续生成稳定。

No-Go：

- 必须把 fork 直接合并进当前 vendored llama.cpp 才能运行。
- 模型 family 依赖未认证补丁。
- stage 失败无法通过 sidecar API 清理。
- 普通网络下性能和稳定性不满足基线。

### 13.2 exo/tensor parallel

Go：

- 有匹配的 Apple Silicon/RDMA 或同构高速 GPU 环境。
- tensor placement 的端到端收益通过实测。
- 模型卡和模型 revision 可锁定。
- 作为独立 provider 时取消、健康和指标可观测。

No-Go：

- 试图用 Wi-Fi Android 节点加入 collective group。
- 只有“多节点被识别”，没有性能或容量收益。
- 要求把 MLX tensor 适配到现有 PyTorch `LAYER_FORWARD`。

### 13.3 原版 GGML RPC

默认 No-Go。只有在隔离实验中完成认证、安全封装、断线清理和收益验证后，才重新评估；即使通过，也作为独立 GGML provider，而不是改造当前 PyTorch layer provider。

## 15. 最终形态

```text
统一 API / UI / 权限 / 任务身份 / 模型目录 / 指标
                       |
                  Execution Planner
                       |
      +----------------+----------------+----------------+
      |                |                |                |
PyTorch 层间流水线   任务链/DAG       张量并行 Provider   GGUF Stage Provider
Safetensors/PC       跨引擎组合       exo/MLX 或 CUDA     Mesh-LLM/Skippy
      |                |                |                |
      +---------- 完整 Worker / 本地推理作为基础与降级 ----------+
```

QLH 的目标不是选出唯一的“分布式真理”，而是让不同拆分方式拥有清楚的适用边界、独立故障域和可验证收益。现有层间拆分继续服务 PyTorch PC 集群；任务链连接 PC 与 Android 等异构节点；张量并行服务高速同构集群；GGUF 层段仅通过经过验证的独立运行时提供。统一控制面负责正确选择和如实展示实际执行路径。
