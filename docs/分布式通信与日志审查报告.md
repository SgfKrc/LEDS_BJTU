## 分布式通信

### C1 (Medium) — 注册被拒后连接未关闭 + 泄露给调度器

**文件**: [tcp_comm.py:946-1051](src/tcp_comm.py#L946-L1051) + [tcp_comm.py:908-914](src/tcp_comm.py#L908-L914)

**描述**: `_handle_registration` 在认证失败或角色验证失败时发送 rejection ACK，但返回 `temp_id`（如 `pending_54321`）。`_handle_client` 收到返回值后：

1. 将 `client_id` 设为 `temp_id`
2. 调用 `self.on_message(client_id, msg)` — 调度器收到 `msg_type="register"` 的消息，`client_id="pending_54321"`
3. 调度器的 `_on_tcp_message` 执行 `register_node(node_id="pending_54321", ...)`，将一个垃圾 ID 写入节点表
4. **连接未关闭**，被拒客户端可持续发送消息（仅受 socket timeout 限制）

**影响**: 
- 调度器 `self.nodes` 被污染（`pending_54321` 条目）
- 被拒客户端占用连接资源，可继续发送消息
- 攻击者可通过反复连接-被拒来填充节点表

**修复方向**: 被拒后应 (a) 跳过 `on_message` 回调，或 (b) 在 `_handle_registration` 中抛出特定异常由 `_handle_client` 捕获后 `break` 关闭连接。

### C2 (Medium) — `TCPServer.clients` 字典无线程同步

**文件**: [tcp_comm.py:802](src/tcp_comm.py#L802)

**描述**: `self.clients: dict[str, ClientConn]` 被多个线程并发访问且无锁保护：

| 线程 | 操作 | 位置 |
|------|------|------|
| `_handle_client`（每客户端一个） | `del self.clients[client_id]` | L936 |
| `_heartbeat_loop` | `del self.clients[cid]` | L1136 |
| 主线程（通过 `send_to_client`） | `client_id not in self.clients` + `self.clients[client_id].sock` | L1081-1085 |
| 主线程（通过 `broadcast`） | `list(self.clients.keys())` + `send_to_client` | L1092-1096 |
| 主线程（通过 `get_client_ids`） | `list(self.clients.keys())` | L1146 |

**竞态场景**:
- `_heartbeat_loop` 在 L1136 执行 `del self.clients[cid]` 的同时 `send_to_client` 在 L1081 检查 `client_id not in self.clients` → 检查通过，但 L1085 访问时条目已被删除 → `KeyError`
- `_handle_client` finally 块和 `_heartbeat_loop` 同时对同一客户端执行 `del` → 第二次 `KeyError`（当前未捕获，会导致线程崩溃）

**影响**: 低概率竞态崩溃，在大规模集群或高频率连接断开时更易触发。

**修复方向**: 将 `self.clients` 操作包装在 `threading.Lock` 中，或使用 `client_id in self.clients` 后用 `.get()` 安全访问。

### C3 (Low) — `_on_tcp_message` 无 `else` 分支处理未知消息类型

**文件**: [scheduler.py:2004-2165](src/scheduler.py#L2004-L2165)

**描述**: 消息分发链（`if/elif/...`）无 `else` 分支。收到未知消息类型时静默忽略，无日志。排查协议不匹配或版本不一致问题时缺少线索。

**修复方向**: 添加 `else: logger.debug(f"未知消息类型: {msg_type}, client={client_id}")`

## 日志功能

### L1 (Medium) — `traceback.print_exc()` 替代 `logger.error(..., exc_info=True)`

**文件**: [scheduler.py:1522](src/scheduler.py#L1522), [4479](src/scheduler.py#L4479), [5496](src/scheduler.py#L5496), [5587](src/scheduler.py#L5587)

**描述**: 4 处使用 `import traceback; traceback.print_exc()` 输出异常堆栈。`print_exc()` 默认输出到 `sys.stderr`，不进入日志文件。在打包部署环境（PyInstaller）中 stderr 可能不可见，导致崩溃后无堆栈信息可供排查。

**实例**:
```python
# scheduler.py:5496 — _process_queued_pipeline_task 异常处理
except Exception as e:
    logger.error(f"队列任务流水线推理失败: {e}，回退到全模型推理")
    import traceback
    traceback.print_exc()  # ← 进 stderr，不进日志文件
```

**修复**: 替换为 `logger.error(f"队列任务流水线推理失败: {e}", exc_info=True)`

---

### L2 (Medium) — `tcp_comm.py` 错误日志缺少 `exc_info=True`

**文件**: [tcp_comm.py:860](src/tcp_comm.py#L860), [925](src/tcp_comm.py#L925), [944](src/tcp_comm.py#L944), [1142](src/tcp_comm.py#L1142), [1376](src/tcp_comm.py#L1376)

**描述**: 所有 `logger.error(f"...{e}")` 仅输出异常消息，不含堆栈。TCP 通信问题（如 `ConnectionError`、`OSError`）通常需要调用栈才能定位根因。

**实例**:
```python
logger.error(f"Accept 异常: {e}")           # ← 无堆栈
logger.error(f"消息回调异常: {e}")          # ← 无堆栈
```

**修复**: 添加 `exc_info=True`。

---

### L3 (Low) — 调度器 `_on_tcp_disconnect` 无日志

**文件**: [scheduler.py:2167-2176](src/scheduler.py#L2167-L2176)

**描述**: 节点断连是重要的运维事件，但调度器的 `_on_tcp_disconnect` 没有任何日志输出。TCPServer 在 `_handle_client` finally 中有日志，但调度器层面的状态变更（node 标记 offline、deregister、通知其他节点）完全静默。

**修复**: 添加 `logger.info(f"🔌 节点 {client_id} 断开连接，已标记 offline")`

### L4 (Low) — `pipeline_pause`/`pipeline_resume` 仅 DEBUG 级别日志

**文件**: [scheduler.py:2154-2160](src/scheduler.py#L2154-L2160)

**描述**: 这两个 handler 目前是协议预留，若将来实际使用，`logger.debug` 在默认 INFO 级别下不可见。流水线暂停/恢复是运维操作，应至少 INFO 级别记录。

**修复**: 升为 `logger.info`（或在 handler 中添加状态变更时才 info）。

### L5 (Low) — `detect_lan_ip`/`get_mac_addresses` 多层回退无声失败

**文件**: [tcp_comm.py:272-408](src/tcp_comm.py#L272-L408), [411-550](src/tcp_comm.py#L411-L550)

**描述**: IP/MAC 检测函数使用多层回退策略，每层有 `except Exception: pass`。当高层策略失败时无 DEBUG 日志，排查"为什么检测到了回退 IP"时缺少信息。

**影响**: 低 — 当前有最终 `logger.warning` 兜底，但中间策略失败不可见。

---

### L6 (Low) — `_process_loop` 启动/停止无日志

**文件**: [scheduler.py:676-746](src/scheduler.py#L676-L746)

**描述**: `PipelineQueue._process_loop` 是核心工作线程。启动时无日志，停止时仅一行 `logger.info("流水线队列工作线程已退出")`。启动的沉默使得从日志中无法判断 worker 何时开始处理任务。

- ✅ 注册消息在 TCPServer 层做了 client_id/role 白名单校验
