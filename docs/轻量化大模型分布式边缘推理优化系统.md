# 一、项目总览 & 整体架构（框架首页）
## 1.1 项目背景与目标
1. 解决问题：单台普通笔记本显存/算力不足，无法流畅运行大语言模型
2. 实现方案：将 Qwen-1.8B 按 Transformer 层拆分（读取硬件条件动态配置拆分），部署到多台边缘笔记本，组成分布式推理集群
3. 三大单机优化：INT4/INT8 模型量化、算子融合、轻量化 PagedAttention 分页KV缓存
4. 通信方案：主从架构 + TCP 长连接 + 自定义应用层协议
5. 交付成果：可运行分布式推理系统 + 带性能对照的可视化平台

## 1.2 整体拓扑架构
### 硬件拓扑（3 台边缘设备）
- 主节点（Server）：承载模型前半部分网络层，TCP 服务端，统一调度、接收前端请求
- 从节点 1（Client）：承载模型中间层，TCP 客户端，转发中间特征张量
- 从节点 2（Client）：承载模型后半部分网络层，TCP 客户端，输出最终推理结果

### 数据流走向（固定流水线）
`前端交互 → 主节点计算 → 中间特征张量(TCP) → 从节点1计算 → 中间特征张量(TCP) → 从节点2 → 结果回传前端`

### 软件分层架构（自上而下）
1. 应用层：可视化交互 & 性能监控平台
2. 调度层：主从节点任务调度、指令分发、状态管理
3. 通信层：TCP 长连接、粘包处理、心跳检测、张量序列化/反序列化
4. 推理层：模型加载、量化、算子融合、分页KV缓存、模型前向计算
5. 基础依赖层：Python / PyTorch / Transformers / bitsandbytes / Streamlit

## 1.3 核心技术栈清单
| 模块 | 技术/库 | 用途 |
| ---- | ---- | ---- |
| 模型加载推理 | PyTorch、Transformers | 加载 Qwen 模型、前向推理 |
| 模型量化 | bitsandbytes / AutoGPTQ | INT4 / INT8 离线量化 |
| 算子融合 | torch.compile | 自动合并连续算子，减少显存IO |
| KV缓存 | 自研轻量化分页KV | 替代传统连续KV，优化显存碎片 |
| 网络通信 | Python socket | TCP 主从通信、长连接、心跳 |
| 数据序列化 | torch.save / torch.load | 张量转字节流跨设备传输 |
| 可视化界面 | Streamlit | 对话交互 + 实时性能监控 + 对照实验 |

# 二、核心概念&技术原理文档（编码+答辩共用）
> 开发时遇到原理问题直接查阅，同时作为答辩知识储备

## 2.1 算子与算子融合
1. 核心连续算子组合：`LayerNorm → Linear → GELU`
2. 算子定义：深度学习最小原子计算单元，串行执行产生大量显存读写
3. 融合原理：多算子合并为一个执行单元，中间结果保留在GPU寄存器，**不落地全局显存**
4. 项目实现方式：`torch.compile()` 自动融合，无手动CUDA开发
5. 优势：不改变计算逻辑、无精度损失、降低IO、提升推理速度

## 2.2 Q / K / V 与 KV 缓存
1. Q(Query)：当前 Token 专属查询向量，每轮重新计算，**不缓存**
2. K(Key)：历史特征索引标签，一次计算、反复复用
3. V(Value)：历史语义真实数据，与K一一绑定
4. KV缓存作用：自回归解码阶段避免重复计算历史特征，大幅降低算力
5. 传统KV问题：整块连续显存，易产生内存碎片
6. 分页KV（本项目自研简化版）
   - 原理：借鉴操作系统内存分页，拆分固定大小内存页 + 页表映射
   - 功能：动态分配、自动回收、降低显存碎片、提升边缘设备显存利用率
   - 约束：**仅本地节点维护，不跨设备传输**

## 2.3 模型量化
1. 定义：FP32/FP16 浮点权重 → INT8/INT4 整数存储，模型结构不变
2. 类型：后训练量化 PTQ（本项目使用，无需重新训练）
3. 精度取舍：以**微小精度损失**换取显存降低、推理提速
4. 项目配置：权重使用 INT4 量化，中间特征、KV缓存保留 FP16
5. 作用：适配低配边缘笔记本，降低模型静态显存占用

## 2.4 边缘设备 & 分布式流水线推理
1. 边缘设备：本项目指普通学生笔记本，算力/显存有限、网络不稳定
2. 流水线并行规则
   - 单条请求：层间强数据依赖，**严格串行**，无法并行
   - 多条并发请求：不同请求在不同节点同时运行，实现流水线并行
3. 分层策略：根据三台设备性能**自适应分配模型层**，非平均拆分
4. 数据依赖：下一级节点必须接收上一级中间隐藏特征张量才能计算

## 2.5 通信协议（TCP）
1. 底层协议：TCP（可靠传输、丢包重传、保序，适配特征张量传输）
2. 拓扑：主从架构、长连接不断开
3. 数据包格式：`4字节长度头 + 数据体`，解决 TCP 粘包/拆包
4. 传输数据分类
   - 控制数据：指令、心跳、节点状态、文本问答（JSON 字符串）
   - 推理数据：模型中间隐藏特征张量（二进制字节流）
5. 约束：Q/K/V、KV缓存**不跨节点传输**

# 三、模块拆分 & 功能设计（核心编码框架）
按功能拆分为 **5 大独立模块**，解耦开发，可单人/多人分工编写

## 3.1 模型模块 `model_module.py`
### 功能职责
1. 原版 Qwen-1.8B 模型加载
2. 模型离线量化 / 加载量化后模型
3. 开启自动算子融合 `torch.compile`
4. 模型层级拆分（按配置分配给主/从节点）
5. 模型前向推理入口

### 对外接口
- `load_model(model_path, quant_type)`：加载模型，指定 FP16 / INT8 / INT4
- `split_model(layer_config)`：根据配置拆分模型层
- `model_forward(input_ids)`：单步前向推理，输出中间特征/最终结果

### 配置项
- 量化精度：`["fp16", "int8", "int4"]`
- 分层配置：各节点负责的起止层编号

## 3.2 分页KV缓存模块 `paged_kv_cache.py`
### 功能职责
1. 内存页管理、页表映射
2. KV 动态分配、追加写入
3. 会话结束页面回收、清空缓存
4. 适配 Prefill / Decode 双阶段

### 对外接口
- `__init__(page_size)`：初始化页大小、空闲页池、页表
- `append_kv(new_k, new_v)`：追加新Token的KV
- `get_all_kv()`：读取全部历史KV
- `clear()`：清空当前会话缓存，回收所有页面

## 3.3 网络通信模块 `tcp_comm.py`
### 功能职责
1. 主节点（服务端）监听、从节点（客户端）连接
2. 长连接维持、心跳检测、断线重连
3. 封包/解包（长度头 + 数据体）
4. 张量序列化 / 反序列化
5. 控制指令、特征张量收发

### 对外接口
- `start_server(ip, port)`：启动TCP服务端（主节点）
- `connect_client(server_ip, port)`：客户端连接服务端（从节点）
- `send_data(data)`：通用发送接口（自动封包）
- `recv_data()`：通用接收接口（自动解包、还原张量/字符串）
- `heartbeat_loop()`：心跳线程

## 3.4 调度控制模块 `scheduler.py`
### 功能职责
1. 节点状态管理（空闲/忙碌/离线）
2. 推理任务分发、流程启停
3. 异常捕获、错误上报
4. 流水线数据流调度控制

### 对外接口
- `init_nodes()`：初始化所有从节点状态
- `start_infer_task(prompt)`：启动一轮完整推理任务
- `stop_infer_task()`：强制停止推理、重置流水线

## 3.5 可视化&评测模块 `web_ui.py`（Streamlit）
### 功能职责
1. 对话交互窗口（输入提问、展示回答）
2. 优化开关：一键开启/关闭 量化、算子融合、分页KV、分布式
3. 实时性能监控：显存占用、CPU使用率、推理时延、Token生成速度
4. 可视化图表：显存变化折线图、多方案对比柱状图
5. 实验数据记录与导出

### 页面分区
1. 交互区：聊天历史、输入框、功能开关
2. 实时指标区：数字卡片展示各项性能数据
3. 图表区：动态曲线、对比统计图
4. 日志区：节点通信日志、推理日志

# 五、运行流程说明（全链路时序）
## 5.1 系统初始化流程
1. 读取全局配置 `config.py`
2. 主节点启动 TCP 服务端，监听端口
3. 两台从节点启动客户端，主动连接主节点，注册节点信息
4. 所有节点加载模型、执行量化、开启算子融合、初始化分页KV缓存
5. 主节点调度器进入就绪状态，等待前端请求

## 5.2 单次推理完整流程
1. 前端接收用户提问，下发至主节点调度器
2. **Prefill 阶段**：主节点处理输入文本，计算前半层 → 生成中间特征 → TCP 发送至从节点1
3. 从节点1接收张量，计算中间层 → 转发特征至从节点2
4. 从节点2计算最后层，生成结果，本轮所有节点**本地追加KV至分页缓存**
5. **Decode 循环阶段**：逐Token生成，重复「计算→传张量→追加KV」流程
6. 生成结束：清空本轮KV缓存，节点回归空闲状态
7. 最终结果回传至前端展示，同步记录所有性能指标

## 5.3 异常处理流程
1. 网络断开：心跳检测失效 → 标记节点离线、尝试自动重连
2. 推理报错：停止任务、清空KV、重置流水线、前端提示错误
3. 显存溢出：日志告警、任务终止

1. 统一使用 Python 标准缩进：4 空格
2. 变量命名：见名知意，使用小写+下划线（例：`kv_cache`、`layer_feature`）

2. 核心函数：说明**输入、输出、逻辑、调用位置**
