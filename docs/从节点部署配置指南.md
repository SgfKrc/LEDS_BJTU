# 从节点首次连接自动部署计划

**创建日期**: 2026-07-11  
**更新日期**: 2026-07-12  
**适用范围**: Windows / Linux PC 从节点、Android Full 从节点  
**目标**: 从节点首次接入 Tailscale 子网后，自动完成主节点发现、配置下发、节点登记和后续启动配置持久化。

---

## 1. 当前问题

现有部署说明以手工 `.env` 配置为主，但项目代码还没有完整实现自动部署流程，实际存在几个缺口：

| 问题 | 当前状态 | 影响 |
|------|----------|------|
| 首次连接自动部署 | 已完成基础代码实现 | 从节点可通过主节点 bootstrap 获取密钥并持久化 |
| Android 从节点 | 已完成基础代码接入 | Android Full 可调用 bootstrap 后再做 HTTP presence 登记 |
| 安全模型过重 | 文档强调手工密钥与复杂认证 | 与 Tailscale 子网可信边界重复，增加部署成本 |
| 平台差异 | 未区分 PC 与 Android | `.env` 适合 PC，不适合 Android DataStore / 应用私有存储 |
| 运维边界 | 未明确只做首次连接 | 容易扩展成复杂的远程配置管理系统 |

本计划只解决“首次连接自动部署”。已部署节点的后续配置变更、批量管控、证书轮换、复杂审批流不纳入本阶段。

### 1.1 当前代码审查结论与实现状态

改造前结论：代码已经有“自动发现 / 自动连接”，但没有“首次连接自动部署”。升级后从节点断联的直接原因是 `QLH_CLUSTER_SECRET` 改为只从环境变量 / `.env` 读取，而从节点安装包不携带 `.env`，导致 TCP 注册 HMAC 校验失败。

当前实现状态：已补齐基础版首次连接自动部署。PC 从节点在缺少集群密钥时，`connect_to_master()` 会先调用主节点 `/api/bootstrap/first-connect`，保存 `node_config.json`，再继续原 TCP/HMAC 注册。Android 端已补充 bootstrap API 客户端和 DataStore 持久化字段，启动时会尝试 bootstrap 后再进行 presence 登记。

| 链路 | 当前实现 | 缺口 |
|------|----------|------|
| 主节点发现 | `scheduler._auto_connect_on_startup()` 启动后调用 `discover_master()` | 若无 DB 且无主节点地址，仍需用户输入一次主节点 Tailscale IP |
| 主动连接 | `scheduler.connect_to_master()` 创建 `TCPClient` 并连接主节点 | 已增加缺密钥时先 bootstrap |
| TCP 注册 | `TCPClient.connect()` 注册消息始终携带 `auth=build_auth_signature()` | 已改为运行时读取最新密钥 |
| 主节点认证 | `TCPServer._handle_register()` 调用 `verify_auth_signature()` | 主节点密钥为空时明确拒绝，bootstrap API 会生成密钥 |
| 配置读取 | `config.py` 读取 `.env` / 环境变量 / `node_config.json` | 仍需实机验证安装包路径 |
| Bootstrap API | 已实现 `/api/bootstrap/first-connect` | 目前是基础版，复杂发现/二维码不做 |
| Android | Android 端已有 bootstrap DTO、DataStore 字段、启动 bootstrap 尝试 | 尚未新增专门首次连接 UI |

关键代码位置：

- `src/config.py`: 启动时先应用 `node_config.json`，再读取 `QLH_CLUSTER_SECRET` 等配置。
- `src/node_config.py`: 负责 `node_config.json` 的读写、bootstrap 响应持久化、主节点密钥生成。
- `src/bootstrap.py`: 负责受信 CIDR 判断和从节点 first-connect 请求。
- `src/scheduler.py`: `_auto_connect_on_startup()` 已实现自动发现和自动连接。
- `src/scheduler.py`: `discover_master()` 先查数据库，失败后才回退 `CLIENT_MASTER_HOST`。
- `src/scheduler.py`: `connect_to_master()` 在密钥缺失时先 bootstrap，再建立 TCP 连接。
- `src/tcp_comm.py`: `build_auth_signature()` / `verify_auth_signature()` 运行时读取当前 `CLUSTER_SECRET`。
- `src/api_server.py`: 已实现 `/api/bootstrap/first-connect`、`/api/cluster/connect`、`/api/cluster/discover`、`/api/cluster/android/register`。
- `android/.../SettingsDataStore.kt`: 已保存 `bootstrapped`、`cluster_id`、`master_tcp_port`、`model_manifest_url`。
- `android/.../MainViewModel.kt`: 启动和连接测试成功后会尝试 `ensureAndroidBootstrap()`，再做 Android presence 登记。

### 1.2 升级后从节点全部断联的原因

升级前如果集群密钥是硬编码或随代码分发，从节点可以直接完成 HMAC 注册。升级后密钥被移动到 `.env` / 环境变量：

```text
主节点: QLH_CLUSTER_SECRET=真实密钥
从节点: QLH_CLUSTER_SECRET 缺失，实际为空字符串
从节点: 使用空字符串生成 HMAC
主节点: 使用真实密钥验证 HMAC
结果: HMAC 签名不匹配，从节点注册被拒绝
```

同时，安装包不包含 `.env` 是正确的安全行为，但当前代码没有补上“首次连接时通过可信 Tailscale 网络自动下发部署配置”的替代流程，所以自动连接链路在认证阶段中断。

短期止血方案：

1. 手工把主节点相同的 `QLH_CLUSTER_SECRET` 写入所有 PC 从节点本地 `.env`。
2. 如果从节点依赖数据库自动发现，还需要补齐 `QLH_DB_*`；否则手工配置 `CLIENT_MASTER_HOST` / `QLH_MASTER_HOST` 等主节点地址。
3. 重启从节点后，现有自动连接逻辑会重新生效。

已完成的长期修复方向：

- 保留 `.env` 不进安装包的安全策略。
- 增加 `/api/bootstrap/first-connect`。
- 在 Tailscale 可信来源内，由主节点首次下发 `cluster_secret` 和主节点连接配置。
- 从节点本地持久化配置，后续启动不再请求 bootstrap。

仍需验证：

- Windows 安装包模式下 `node_config.json` 路径是否符合预期。
- Linux 包装启动器使用 `QLH_API_PORT` 后的端口检查和浏览器打开流程。
- Android 真机首次连接 Tailscale 主节点后的 DataStore 持久化和 presence 登记。

## 2. 目标原则

### 2.1 网络信任边界

默认安全假设：

> 能通过 Tailscale 子网所有者审核并接入该 Tailscale 子网的设备，视为可信设备。

因此首次连接阶段不再设计复杂的二次认证、管理员预授权列表、二维码审批、设备指纹审批等流程。

最低安全约束只保留：

1. 主节点自动部署接口只绑定或只接受 Tailscale 来源地址。
2. 拒绝非 Tailscale / 非受信子网来源。
3. 禁止从节点声明 `role=master`。
4. 禁止从节点使用保留 `node_id=master`。
5. Android 节点只能作为 `client`，默认不参与 Transformer 层间拆分。

### 2.2 部署体验

首次接入流程应变成：

1. 用户把设备加入 Tailscale 子网。
2. 安装并启动从节点应用。
3. 从节点自动发现主节点或用户输入一次主节点 Tailscale 地址。
4. 从节点请求首次部署配置。
5. 主节点返回运行配置。
6. 从节点本地持久化配置。
7. 从节点自动注册上线。

用户不需要复制 `.env`，不需要手工获取 `QLH_CLUSTER_SECRET`。

### 2.3 只做首次连接

首次部署成功后：

- PC 从节点写入本地 `.env` 或专用 `node_config.json`。
- Android 写入 `SettingsDataStore`。
- 后续启动直接读取本地配置并重连。
- 若用户要重新部署，提供“清除节点配置 / 重新接入”操作，不自动覆盖已有配置。

## 3. 节点类型定义

| 节点类型 | 平台 | 当前能力 | 本阶段目标 |
|----------|------|----------|------------|
| PC Worker | Windows / Linux | TCP 注册，参与层间拆分 | 首次自动获取主节点地址、端口、集群密钥、节点 ID |
| Android Full | Android | 本地 llama.cpp 推理、HTTP presence 登记、模型下载能力 | 首次自动获取主节点地址、API 端口、节点 ID、模型清单配置 |
| Android Lite | Android | thin client | 可复用 Android Full 的首次连接配置，但不作为 Worker |

边界说明：

- Android 本阶段默认仍是 `node_type=android`、`role=client`、`pipeline_worker=false`。
- Android 不参与 Transformer 层间拆分，避免把移动端误接入 PC Worker 调度。
- 后续如果实现 Android 普通版完整推理 Worker，应走“任务级完整推理”路径，不走层间拆分路径。

## 4. 首次连接自动部署流程

### 4.1 总体流程

```text
从节点启动
  |
  |-- 检查本地是否已有部署配置
  |     |
  |     |-- 有：直接按本地配置连接主节点
  |     |
  |     `-- 无：进入首次连接自动部署
  |
  |-- 发现或输入主节点 Tailscale 地址
  |
  |-- POST /api/bootstrap/first-connect
  |     payload: node_id / node_type / hostname / platform / capabilities
  |
  |-- 主节点检查来源 IP 是否来自 Tailscale / 受信子网
  |
  |-- 主节点返回部署配置
  |
  |-- 从节点持久化配置
  |
  `-- 从节点注册上线
```

### 4.2 主节点返回配置

建议返回结构：

```json
{
  "status": "ok",
  "cluster": {
    "cluster_id": "qlh-default",
    "master_api_host": "100.x.y.z",
    "master_api_port": 8000,
    "master_tcp_host": "100.x.y.z",
    "master_tcp_port": 8888,
    "cluster_secret": "server-generated-secret"
  },
  "node": {
    "node_id": "pc-hostname-a1b2",
    "role": "client",
    "node_type": "pc",
    "pipeline_worker": true
  },
  "android": {
    "presence_interval_seconds": 30,
    "pipeline_worker": false,
    "model_manifest_url": "http://100.x.y.z:8000/api/models/downloadable"
  }
}
```

说明：

- `cluster_secret` 仍可保留，用于兼容当前 TCP HMAC 注册逻辑，但它不再由用户手工分发。
- PC 从节点拿到配置后，后续 TCP 注册仍可走现有 HMAC。
- Android presence 接口可不使用 HMAC，只依赖 Tailscale 来源地址校验。
- Android 返回段仅在 `node_type=android` 时使用。

## 5. PC / Linux 从节点实施计划

### 5.1 新增首次部署模块

建议新增：

| 文件 | 职责 |
|------|------|
| `src/bootstrap.py` | 首次连接客户端逻辑：发现主节点、请求配置、写入本地配置 |
| `src/bootstrap_server.py` 或 `api_server.py` 内路由 | 主节点首次连接接口 |
| `src/node_config.py` | 统一读取 `.env` / `node_config.json` / 环境变量 |

### 5.2 从节点启动逻辑

在 `scheduler.start()` 或更早的应用启动阶段增加：

1. 如果 `NODE_ROLE=client` 且本地没有有效主节点配置，进入 bootstrap。
2. 优先尝试自动发现主节点：
   - 读取用户输入过的 `master_api_host`。
   - 扫描 Tailscale 网段内可用主节点接口（可作为 P1，首次先允许用户输入）。
   - 查询共享数据库不是首选，因为新从节点不应依赖数据库密码。
3. 调用主节点 `/api/bootstrap/first-connect`。
4. 写入本地配置。
5. 继续执行现有 `_auto_connect_on_startup()`。

### 5.3 本地配置格式

PC 端推荐优先写 `node_config.json`，再由配置加载层映射到运行参数，减少直接修改 `.env` 的副作用。

```json
{
  "bootstrapped": true,
  "cluster_id": "qlh-default",
  "node_id": "pc-hostname-a1b2",
  "node_type": "pc",
  "master_api_host": "100.x.y.z",
  "master_api_port": 8000,
  "master_tcp_host": "100.x.y.z",
  "master_tcp_port": 8888,
  "cluster_secret": "server-generated-secret",
  "created_at": "2026-07-12T00:00:00+08:00"
}
```

兼容策略：

- 环境变量优先级最高。
- `node_config.json` 第二。
- `.env` 第三。
- 默认值最后。

### 5.4 需要修改的现有代码

| 位置 | 修改点 |
|------|--------|
| `src/config.py` | 增加从 `node_config.json` 读取主节点地址、端口、`CLUSTER_SECRET` 的逻辑 |
| `src/api_server.py` | 增加 `/api/bootstrap/first-connect` 接口 |
| `src/tcp_comm.py` | 注册失败信息保持清晰；HMAC 缺失时提示“请先完成首次连接部署” |
| `src/scheduler.py` | 从节点启动前触发 bootstrap；主节点继续拒绝 `role=master` 和 `node_id=master` |
| 安装包脚本 | 首次启动不再要求 `.env`，只要求 Tailscale 已连接 |

## 7. 主节点接口设计

### 7.1 `POST /api/bootstrap/first-connect`

用途：首次接入设备请求部署配置。

请求字段：

| 字段 | 必填 | 说明 |
|------|------|------|
| `node_id` | 否 | 客户端已有稳定 ID 时上报；缺失则主节点生成 |
| `node_type` | 是 | `pc` / `android` |
| `hostname` | 否 | 设备名 |
| `platform` | 否 | `windows` / `linux` / `android` |
| `app_variant` | 否 | Android `full` / `lite` |
| `capabilities` | 否 | 设备画像 |

主节点处理：

1. 获取 `request.client.host`。
2. 判断来源是否是 Tailscale 地址或配置允许的受信子网。
3. 生成或规范化 `node_id`。
4. 写入节点记录，初始状态可为 `offline` 或 `bootstrapped`。
5. 返回主节点 API/TCP 地址、端口、集群密钥、节点行为策略。

### 7.2 Tailscale 来源判断

最低实现：

- 允许 `100.64.0.0/10`。
- 允许配置项 `QLH_TRUSTED_BOOTSTRAP_CIDRS` 扩展额外网段。
- 如果主节点部署在反向代理后面，明确只信任直接连接，不解析任意 `X-Forwarded-For`。

建议配置：

```bash
QLH_BOOTSTRAP_ENABLED=true
QLH_BOOTSTRAP_REQUIRE_TAILSCALE=true
QLH_TRUSTED_BOOTSTRAP_CIDRS=100.64.0.0/10
```

### 7.3 错误响应

| 场景 | 状态码 | 响应 |
|------|--------|------|
| 非 Tailscale 来源 | 403 | `source network is not trusted` |
| `node_type` 非法 | 400 | `unsupported node_type` |
| `node_id=master` | 400 | `reserved node_id` |
| Android 声明 master | 400 | `android cannot be master` |
| bootstrap 未启用 | 404 或 403 | `bootstrap disabled` |

## 8. 安全与边界

本阶段不做：

- 设备指纹审批。
- 一次性邀请码。
- 二维码配对。
- mTLS。
- 管理员手动批准每台设备。
- 复杂证书轮换。

本阶段必须做：

- Tailscale / 受信子网来源限制。
- 主节点角色保留字保护。
- Android 不可成为 master。
- Android 默认 `pipeline_worker=false`。
- 日志记录首次部署事件：来源 IP、node_id、node_type、hostname、时间。

风险接受：

- 同一 Tailscale 子网内设备可请求首次部署配置。
- 该风险由 Tailscale 子网所有者的接入审核承担。
- 如果设备被移出 Tailscale 子网，后续无法访问主节点接口。

## 9. 分期实施计划

### P0：PC 从节点首次自动部署

目标：Windows / Linux 从节点不再手工复制 `.env`。

任务：

1. 增加主节点 `/api/bootstrap/first-connect`。
2. 增加 Tailscale 来源校验。
3. 增加 PC 端 `node_config.json` 读写。
4. 从节点启动时无配置则进入 bootstrap。
5. bootstrap 成功后复用现有 TCP 注册和 HMAC 逻辑。

验收：

- 新 PC 从节点无 `.env` 可首次启动。
- 输入主节点 Tailscale IP 后自动写入配置。
- 重启后不再触发 bootstrap。
- 主节点管理界面显示该从节点在线。

### P1：Android Full 首次自动部署

目标：Android Full 接入 Tailscale 后可自动登记到主节点。

任务：

1. Android 增加首次连接页或设置项。
2. Android 调用 `/api/bootstrap/first-connect`。
3. Android 保存主节点配置到 `SettingsDataStore`。
4. bootstrap 成功后调用现有 Android presence 登记。
5. 设置页增加“清除部署配置”。

验收：

- 新安装 Android App 无需预置服务器地址。
- 输入主节点 Tailscale IP 后自动保存配置。
- 主节点管理界面显示 Android 节点在线。
- Android 节点 `pipeline_worker=false`。

### P2：主节点发现优化

目标：减少首次输入成本，但不扩大安全模型。

任务：

1. PC 端尝试扫描或读取候选主节点。
2. Android 端可保存最近成功主节点。
3. 主节点返回 `cluster_id`，避免误连多个集群。

验收：

- 同一设备重装后能优先提示最近主节点。
- 多个候选主节点时要求用户选择一次。

| 场景 | 预期 |
|------|------|
| PC 首次启动，无配置 | bootstrap 成功并注册上线 |
| PC 第二次启动 | 不再请求 bootstrap，直接重连 |

### 10.3 手工验收

1. 主节点连接 Tailscale 并启动。
2. 新 Windows 从节点加入同一 Tailscale 子网。
3. 删除从节点本地 `.env` / `node_config.json`。
4. 启动从节点，输入主节点 Tailscale IP。
5. 确认自动注册成功。
6. 新 Android Full 加入同一 Tailscale 子网。
7. App 输入主节点 Tailscale IP。
8. 确认主节点管理界面显示 Android 在线。

## 11. 临时兜底方案

在自动部署实现前，仍可临时使用手工 `.env`：

```bash
QLH_NODE_ROLE=client
QLH_MASTER_HOST=100.x.y.z
QLH_MASTER_PORT=8888
QLH_CLUSTER_SECRET=<主节点当前密钥>
```

该方案只作为过渡，不再作为推荐部署路径。
