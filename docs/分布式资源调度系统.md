# 分布式资源调度系统 — MLFQ 请求队列 + 图算法层编排

> **适用范围**: QLH v0.1.7+ 分布式推理集群  
> **相关源码**: `scheduler.py` (PipelineQueue/MLFQ) · `graph_orchestrator.py` (图算法)  
> **前置阅读**: [图算法.md](图算法.md) (最大带宽生成树 + DFS 路径搜索推导)

## 一、两套调度系统的分工

QLH 分布式推理有两套**独立但协同**的调度系统，解决不同层面的问题：

| 维度 | MLFQ 请求队列 | 图算法层编排 |
|------|--------------|-------------|
| **调度对象** | 用户推理请求（prompt） | 模型层（Transformer Layer） |
| **目标** | 降低交互请求等待时间 | 找到最优流水线节点序列 |
| **输入** | 请求 token 数 + 等待时间 | 节点算力/显存/网络拓扑 |
| **输出** | 下一条执行的 task_id | 每个节点分配的层区间 |
| **触发时机** | 每次用户发消息 | 节点上下线 / 手动 override |
| **算法** | MLFQ 三级反馈队列 + FIFO 兼容 | 最大带宽生成树 + DFS 路径搜索 |
| **代码位置** | `scheduler.PipelineQueue` | `graph_orchestrator.GraphOrchestrator` |

**关系**：层编排决定了"谁负责哪些层"（静态拓扑），请求队列决定了"谁先推理"（动态调度）。二者解耦——换层编排策略不影响队列调度。

## 二、MLFQ 三级反馈队列

### 2.1 为什么需要 MLFQ

分布式推理场景中，请求类型差异极大：

- **短交互**（"翻译这句话"）→ 期望 <2s 响应
- **中等生成**（"写一封邮件"）→ 可接受 5-10s
- **长批量**（"分析这篇论文"）→ 分钟级完成即可

简单 FIFO 的问题是：**一个 1024 token 的长请求会堵住后面所有短交互**——这就是"队头阻塞"（Head-of-Line Blocking）。

### 2.2 三级队列设计

```
         ┌──────────────────────────────────┐
         │          MLFQ 调度器              │
         │                                  │
  ┌──────▼──────┐  ┌──────▼──────┐  ┌──────▼──────┐
  │  Q0 交互级   │  │  Q1 普通级   │  │  Q2 批量级   │
  │ ≤128 tokens │  │ ≤512 tokens │  │ >512 tokens │
  │  最高优先级  │  │  中优先级    │  │  最低优先级  │
  └─────────────┘  └─────────────┘  └─────────────┘
```

| 队列 | 条件 | 定位 | max_new_tokens |
|------|------|------|----------------|
| **Q0** | 交互级 | 聊天、翻译、简单问答 | ≤128 |
| **Q1** | 普通级 | 邮件撰写、中等生成 | 129-512 |
| **Q2** | 批量级 | 长文分析、批量处理 | >512 |

### 2.3 调度规则

1. **严格优先级**：Q0 > Q1 > Q2。只要 Q0 非空，永远从 Q0 取。
2. **同级 SJF**（Shortest Job First）：同队列内按 `max_new_tokens` 升序排列，短请求优先。
3. **老化提升**（Aging）：长时间等待的请求自动升级到更高优先级队列：
   - Q1 等待 >60s → 提升到 Q0
   - Q2 等待 >120s → 提升到 Q1
   - 任意请求等待 >300s → 直接提升到 Q0（**饥饿防护**）
4. **流水线独占**：同时只执行 1 个推理任务（GPU 独占），其余排队。
5. **抢占预留**（Phase 2）：支持在 decode 阶段中断 Q2 长请求执行 Q0 短请求，短请求完成后再恢复长请求（KV cache 保存/恢复）。

### 2.4 FIFO 兼容模式

当 `strategy="fifo"` 时，三级队列退化为**非严格 FIFO**：

- 所有请求进入同一队列（按 `max_new_tokens` 分级但不分队列）
- 同级内保持入队顺序（FIFO）
- Q0 整体仍在 Q1/Q2 之前出队（跨级优先级保留）

### 2.5 配置参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `strategy` | `"mlfq"` | 调度策略 (`mlfq` / `fifo`) |
| `q0_max_tokens` | 128 | Q0 入队阈值 |
| `q1_max_tokens` | 512 | Q1 入队阈值 |
| `aging_q1_to_q0` | 60s | Q1→Q0 老化时间 |
| `aging_q2_to_q1` | 120s | Q2→Q1 老化时间 |
| `aging_max_wait` | 300s | 饥饿防护上限 |

## 三、图算法智能层编排

### 3.1 问题描述

分布式流水线推理需要决定：**N 个节点各负责 Transformer 的哪些层？**

简单方案是按算力权重比例分配（`_simple_weight_assignment`）。但这忽略了**网络拓扑**：

- 两个强算力节点之间可能只有 WiFi 连接（带宽有限）
- 一个中等算力 + 千兆以太网的路径可能总延迟更低

图算法编排的目标：**找到一条节点序列，使得流水线总通信延迟最小，同时每个节点的显存足够容纳分配给它的层**。

### 3.2 算法流程

```
  全连接通信图                   最大带宽生成树              DFS 最优路径
  ┌───┐                         ┌───┐                      A ─→ C ─→ D
  │A  │──100Mbps──│B  │         │A  │──100Mbps──│C  │      Layer 0-7   8-15  16-23
  │   │──30Mbps ──│   │         │   │──30Mbps ──│   │      emb ✓   mid   lm_head ✓
  └───┘           └───┘         └───┘           └───┘
    │               │             │               │
    │80Mbps   50Mbps│     ──→     │80Mbps         │         B (剔除: 显存不足)
    │               │             │               │
  ┌───┐           ┌───┐         ┌───┐
  │C  │──60Mbps──│D  │         │D  │──60Mbps──(孤立)     A,C,D = 流水线路径
  └───┘           └───┘         └───┘
      (原图: n² 条边)            (生成树: n-1 条边)            (DFS: 1 条路径)
```

**Step 1 — 构建全连接通信图**：每对节点之间估算带宽 + 延迟，构建加权无向图。

**Step 2 — 最大带宽生成树**：使用 Kruskal 算法（按带宽降序），保留 n-1 条高带宽边，剔除低带宽冗余连接。时间复杂度 O(n² log n)。

**Step 3 — DFS 路径搜索**：在生成树上从主节点出发 DFS，寻找满足每个节点显存约束的最优路径。选择标准：总通信延迟最小。

**Step 4 — 层分配**：沿最优路径按 VRAM 比例分配层数，保证：
- embedding 层分配给首节点（`has_embedding=True`）
- LM Head 分配给末节点（`has_lm_head=True`）
- 每节点至少 1 层
- 区间连续无间隙

### 3.3 回退链

图编排器设计了三层降级，确保任何情况下系统都能工作：

```
GraphOrchestrator.orchestrate()
  │
  ├── 成功 → DFS 最优路径 + VRAM 比例层分配
  │
  ├── DFS 无可行路径 → _fallback_weight_assignment()     ← 内部回退
  │     （忽略网络拓扑，纯 VRAM 比例分配）
  │
  └── 顶层异常 → scheduler._simple_weight_assignment()  ← 外部回退
        （忽略网络拓扑和 VRAM，纯算力权重分配）
```

### 3.4 启用条件

- 节点数 > `GRAPH_ORCHESTRATOR_THRESHOLD`（默认 5）
- ≤5 节点时直接用简单权重分配（开销更低，效果相近）

## 四、两套系统的配合

```
用户发送 "翻译: hello"
        │
        ▼
┌──────────────────────────────────────┐
│         PipelineQueue.enqueue()       │  ← MLFQ 分级
│  分类: max_new_tokens=64 → Q0 交互级  │
└──────────────────────────────────────┘
        │
        ▼ (Q0 是最高优先级，立即调度)
┌──────────────────────────────────────┐
│       scheduler.run_pipeline()       │
│                                      │
│  ┌────────────────────────────┐      │
│  │ GraphOrchestrator 层分配     │      │  ← 图算法编排（仅在拓扑变化时重算）
│  │ A: Layer 0-7  (emb)        │      │
│  │ C: Layer 8-15              │      │
│  │ D: Layer 16-23 (lm_head)   │      │
│  └────────────────────────────┘      │
│                 │                    │
│    ┌────────────┼────────────┐       │
│    ▼            ▼            ▼       │
│  Node A      Node C      Node D      │  ← LAYER_FORWARD 链式传递
│  emb→L0-7   L8-15→      L16-23→     │
│            hidden_states   logits    │
│                 │                    │
│    ◄────────────┴────────────►       │
│         KV Cache 增量更新            │
└──────────────────────────────────────┘
        │
        ▼
  返回推理结果 + metrics
```

关键点：
- **层编排是惰性的**：只在节点注册/注销/手动 override 时重新计算，不在每次请求时执行
- **MLFQ 是实时的**：每次 enqueue 都根据请求特征分级
- **两者解耦**：层编排改变不影响已在队列中的请求；队列策略切换不影响已分配的层

## 五、模型校验与安全

层编排过程中，`push_layer_config_to_clients()` 会校验从节点的模型 SHA256：

| 场景 | 处理 |
|------|------|
| 主节点 SHA256 ✓ + 从节点 SHA256 ✓ + 一致 | 正常分配层 |
| 主节点 SHA256 ✓ + 从节点 SHA256 ✓ + 不一致 | **拒绝**该节点参与本次编排 |
| 主节点 SHA256 ✓ + 从节点无 SHA256 | **降级放行**（旧版本兼容），但记录 WARNING |
| 主节点无 SHA256 | 跳过所有校验（无模型文件/开发环境） |

此外，`override_layer_assignments()` 拒绝将 Android 节点纳入层分配（Android 无 PyTorch，不支持 `forward_layers`）。
