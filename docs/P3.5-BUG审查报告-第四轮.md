## 🔴 Critical

### C1 — `client.js` `setQueueStrategy` 发送原始对象而非 JSON 字符串 ✅已修复

**文件**: [frontend/src/api/client.js:497-501](frontend/src/api/client.js#L497-L501)  
**描述**: `setQueueStrategy` 的 `body` 是 `{ strategy }`（JS 普通对象），未使用 `JSON.stringify`。`request()` 函数不会自动序列化 body，直接将 options 展开传给 `fetch`。浏览器会将 `{ strategy: "mlfq" }` 强转为字符串 `"[object Object]"`，FastAPI 收到后 JSON 解析失败 → 422/400 错误。策略切换功能完全不可用。

**对比**: 同文件中所有其他 POST 调用（`loadModel`、`switchModel`、`sendMessage`、`updateMaxNodes` 等）均使用 `JSON.stringify`。

**复现**:
```js
// 当前代码
body: { strategy }             // → fetch 收到 "[object Object]"

// 正确写法（与其他所有 POST 调用一致）
body: JSON.stringify({ strategy })
```

**影响**: 策略切换下拉框完全不工作，但 UI 会乐观更新显示新策略（与实际不一致）。

## 🟡 Medium

### M1 — `get_queue_detail()` 未暴露 `q0_max_tokens` / `q1_max_tokens`，前端依赖硬编码回退 ✅已修复

**文件**: 
- [src/scheduler.py:642-672](src/scheduler.py#L642-L672) — `get_queue_detail()` 的 `aging_params` 仅含 `q1_to_q0_s`、`q2_to_q1_s`、`max_wait_s`
- [frontend/src/components/AdminPanel.jsx:1879-1881](frontend/src/components/AdminPanel.jsx#L1879-L1881) — 访问 `queueDetail?.aging_params?.q0_max_tokens`

**描述**: 前端 Q0/Q1/Q2 队列描述标签（`≤128tk` / `≤512tk` / `>512tk`）通过 `queueDetail?.aging_params?.q0_max_tokens || 128` 获取阈值。但后端 `get_queue_detail()` 的 `aging_params` 中没有 `q0_max_tokens` 和 `q1_max_tokens` 字段。前端回退到硬编码默认值 `128`/`512`。当前后端默认值恰好匹配，但若 `config.py` 中的 `PIPELINE_Q0_MAX_TOKENS` 或 `PIPELINE_Q1_MAX_TOKENS` 被修改，前端显示与实际分级阈值不一致。

**修复方向**: 在 `get_queue_detail()` 的 `aging_params` 中添加 `q0_max_tokens` 和 `q1_max_tokens` 字段。

### M2 — `_apply_aging` "绝对上限"段可在单次调度中二次提升任务 ✅已修复（文档）

**文件**: [src/scheduler.py:543-553](src/scheduler.py#L543-L553)  
**描述**: `_apply_aging` 分三步执行：(1) Q2→Q1 提升，(2) Q1→Q0 提升（排除刚从 Q2 提升的），(3) "绝对上限"遍历 Q1+Q2 强制置顶 Q0。问题：步骤 (1) 刚提升到 Q1 的任务，若其等待时间同时超过 `aging_max_wait`（300s），会在步骤 (3) 被再次提升到 Q0。这在单次调度周期内实现了 Q2→Q0 的越级提升，与 docstring 声明的"每次调用仅提升一级"不一致。

**实际影响**: 低。行为本身合理（等待 >300s 理应得到最高优先级），仅是设计文档一致性问题。`just_promoted` 集合的排除逻辑仅在步骤 (2) Q1→Q0 中生效，未覆盖步骤 (3) 的绝对上限检测。

**修复方向**: 若严格遵循"每次仅一级"，在步骤 (3) 中排除 `just_promoted` 中的 task_id；或更新 docstring 明确说明绝对上限可越级。

### M3 — `_run_full_model_inference` 丢弃 `_stream_callback` ✅已修复

**文件**: [src/scheduler.py:5602-5638](src/scheduler.py#L5602-L5638)  
**描述**: 当队列 worker 的回退路径（流水线节点不可用、引擎不兼容等）调用 `_run_full_model_inference` 时，`**kwargs` 中的 `_stream_callback` 被静默丢弃。该方法签名接受 `**kwargs` 但不使用它。调用链为:

```
_process_queued_pipeline_task → _run_full_model_inference(prompt, **kwargs)
                                 ↑ _stream_callback 在 kwargs 中但被忽略
```

**影响**: 回退到全模型推理时，流式回调不触发。用户在前端看不到逐 token 输出，只能等完整响应返回。仅影响分布式节点全部离线时的降级场景。

### L2 — `cancel_task` 使用 O(n²) 线性搜索 + `deque.remove` ✅已修复（文档注释）

**文件**: [src/scheduler.py:586-616](src/scheduler.py#L586-L616)  
**描述**: `cancel_task` 用嵌套 for 循环遍历三级队列查找任务，找到后用 `deque.remove(task)` 删除。`deque.remove()` 内部是 O(n) 线性扫描，叠加外层循环总体 O(n²)。对于 100 容量的队列影响可忽略，但如果未来扩展队列容量，可能成为瓶颈。

**影响**: 当前 `PIPELINE_QUEUE_MAX_SIZE=100` 时可忽略。
