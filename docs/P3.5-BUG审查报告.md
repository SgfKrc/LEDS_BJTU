## Critical

### C1 — `_queue` 属性每次返回新 deque，无锁保护，变异静默丢失 ✅已修复

**文件**: [scheduler.py:256-264](src/scheduler.py#L256-L264)  
**描述**: `_queue` property 每次创建新 deque 副本。外部代码对返回值的任何修改（`.append()`, `.popleft()`）都会静默丢失。未加锁，并发读取可能得到不一致快照。  
**修复**: 添加详细 docstring 警告说明只读语义。已验证无外部代码直接访问 `._queue`。

---

### C2 — `run_pipeline` 中 `logits` 变量遮蔽导致类型分支穿透 ✅已修复

**文件**: [scheduler.py:4871-4875](src/scheduler.py#L4871-L4875)（原有代码）  
**描述**: 当 `logits_data` 不是 `bytes` 类型时（如已是 Tensor），`isinstance` 分支跳过，`logits = logits.to(device)` 读取外层作用域的 `logits`。Step 0 时 `logits=None`，触发 `AttributeError`。  
**修复**: 
```python
if isinstance(logits_data, bytes):
    logits = deserialize_tensor_fast(logits_data).to(device=device)
elif torch is not None and isinstance(logits_data, torch.Tensor):
    logits = logits_data.to(device=device)
else:
    step_error = f"未知 logits 类型: {type(logits_data).__name__}"
```

## High

### H1 — FIFO 模式非严格全局 FIFO ✅已修复

**文件**: [scheduler.py:405-417](src/scheduler.py#L405-L417)  
**描述**: `_get_next_task()` 在 FIFO 模式下按 Q0→Q1→Q2 优先级弹出，而非严格全局入队顺序。短请求（Q0）始终优先于长请求（Q2），与文档声称的"统一先进先出"不一致。  
**修复**: 更新 docstring 明确说明 FIFO 模式为"级内 FIFO，Q0→Q1→Q2 优先级"。

---

### H2 — Q1 任务超绝对上限时未置顶 Q0 ✅已修复

**文件**: [scheduler.py:462-473](src/scheduler.py#L462-L473)  
**描述**: Q1 任务等待超过 `_aging_max_wait` 时，常规老化将其 `append` 到 Q0 队尾；绝对上限检查仅遍历 Q1/Q2，此时任务已在 Q0 中被错过，不会触发 `appendleft` 置顶。  
**修复**: Q1→Q0 老化时检查 `now - t.created_at > self._aging_max_wait`，超过时使用 `appendleft` 置顶。

---

### H3 — `task_data is None` 哨兵检查死代码 ✅已修复

**文件**: [scheduler.py:564](src/scheduler.py#L564)  
**描述**: `QueueTask.to_task_data()` 始终返回 dict，永不为 None。该哨兵检查永远不会触发，但若未来添加真正的哨兵对象可能被漏过。  
**修复**: 移除死代码。

## Medium

### M1 — SJF 排序中间态数据丢失风险 ✅已修复

**文件**: [scheduler.py:488-495](src/scheduler.py#L488-L495)  
**描述**: `q.clear()` 和 `q.extend(q_sorted)` 之间存在短暂窗口——队列为空。若 extend 抛异常则数据丢失。

### M5 — `_process_loop` 与 `run_pipeline_safe` 之间的 TOCTOU 竞态 ✅已修复

**文件**: [scheduler.py:581-596](src/scheduler.py#L581-L596)  
**描述**: `_process_loop` 在持有锁时 pop 任务（L581-583），释放锁后（L584），在重新获取锁设置 `_current_task_id`（L593-596）之前存在竞态窗口。期间 `run_pipeline_safe` 可看到 `is_busy=False` + `queue_size=0`，设置 `__reserved__` 并获取 `_inference_lock` 直接执行——队列任务被插队。  
**修复**: 将 `_current_task_id` 赋值移入 pop 所在的同一锁块内，实现原子化 pop+mark busy。

### M6 — `run_pipeline` logits 错误分支代码重复 ✅已修复

**文件**: [scheduler.py:4891-4918](src/scheduler.py#L4891-L4918)  
**描述**: "logits 键缺失/为 None" 分支（L4901-4905）直接执行 `_broadcast_pipeline_abort` + `return`，而"未知 logits 类型"分支（L4898-4900）仅设置 `step_error` 后落入共享中止路径（L4907-4918）。两条错误路径做相同的事但代码路径不同，维护困难。  
**修复**: 统一为"设置 `step_error` → 落入共享中止路径"模式，消除重复的 abort+return。

### L6 — `Scheduler.__init__` 直接设置 `PipelineQueue` 私有属性 ✅已修复

**文件**: [scheduler.py:726-737](src/scheduler.py#L726-L737)  
**描述**: `Scheduler` 在 `PipelineQueue` 构造后直接对 `_strategy`、`_q0_max_tokens` 等私有属性赋值，绕过构造函数。若未来 `PipelineQueue` 添加属性 setter/验证逻辑会静默失效。  
**修复**: 调度参数提升为 `PipelineQueue.__init__` 的构造函数关键字参数，`Scheduler` 通过构造函数传参。

### L7 — `is_busy` 在实际执行开始前返回 True ✅已修复

**文件**: [scheduler.py:594](src/scheduler.py#L594)（与 M5 同源）  
**描述**: `_current_task_id` 设置在 `_process_fn` 调用之前，导致 `is_busy` 在推理锁竞争期间过早返回 True——外部请求看到"忙碌"但实际尚未开始推理。  
**修复**: 随 M5 修复自然解决——`_current_task_id` 现在在 pop 时原子设置，紧邻实际执行。

### L8 — 抢占统计死字段 + `PIPELINE_PREEMPT_ENABLED` 未导入 ✅已修复

**文件**: [scheduler.py:252-255](src/scheduler.py#L252-L255) + [config.py:135](src/config.py#L135)  
**描述**: `_preempt_count`、`_preempt_total_overhead_ms`、`_last_preempt_time` 声明后从未更新；`PIPELINE_PREEMPT_ENABLED` 在 config.py 中定义但 scheduler.py 未导入。抢占逻辑属于二期实施。  
**修复**: 添加 `PIPELINE_PREEMPT_ENABLED` 导入（标注"二期协同抢占"），统计字段添加注释"二期实施，参数预留"。
