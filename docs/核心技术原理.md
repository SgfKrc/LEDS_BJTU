# 核心技术原理

> 状态：现行
> 更新日期：2026-07-14
> 定位：说明当前多引擎与分布式推理的概念边界；实现细节以源码和专项文档为准。

## 1. 推理引擎与模型工件

QLH 当前有两条主要推理引擎路径：

| 引擎 | 模型工件 | 主要设备 | 当前分布式能力 |
|------|----------|----------|----------------|
| PyTorch/Transformers | Safetensors | CUDA PC，也可按配置回退 CPU | PC 连续层段流水线 |
| llama.cpp | GGUF | CPU/集显 PC、Android | 当前为单节点完整推理 |

同一个逻辑模型名称不代表工件相同。一次 PyTorch 层流水线必须使用经过校验的同模型、同架构、同 tokenizer 和兼容运行时；GGUF 量化文件不能直接加入 `LAYER_FORWARD` 流水线。

## 2. 量化与算子优化

- PyTorch 路线可以按硬件和模型能力选择 FP16、INT8 或 INT4；bitsandbytes 量化主要面向 CUDA。
- llama.cpp 使用 GGUF 量化，例如 Q4_K_M、Q5_K_M 或 Q8_0。
- `torch.compile` 只适用于满足当前门控条件的 PyTorch/CUDA 路径，不是所有模型和设备的无条件优化。
- 量化会改变内存需求、速度和输出误差，评测必须记录模型工件和量化方式，不能只记录模型显示名称。

## 3. Q、K、V 与 KV Cache

自回归注意力会为历史 token 保存 Key/Value，避免每生成一个 token 都重新计算完整历史。QLH 的 PyTorch 层流水线遵循 KV 所有权本地化：

1. 每个节点只维护自己层段产生的 KV cache。
2. 节点边界主要传 hidden states、位置和任务控制信息，不在每一步搬运全部 KV cache。
3. Prefill 处理完整输入并建立缓存；Decode 通常只推进新 token。
4. 任务完成、取消、失败或超时后必须清理对应缓存，迟到结果不能复活已结束任务。

`paged_kv_cache.py` 提供通用分页缓存结构；具体模型前向中的 `past_key_values` 生命周期由模型模块和调度器共同管理。不能把“存在分页缓存类”写成所有引擎都已经使用同一种 PagedAttention 实现。

## 4. 三种分布式拆分

| 方式 | 拆分对象 | 节点间传输 | 适合场景 |
|------|----------|------------|----------|
| 层间拆分 | 连续 Transformer 层段 | hidden states 与控制状态 | 同模型容量扩展，普通局域网可研究 |
| 任务链拆分 | 检索、生成、校验等业务 Stage | 文本、JSON、文档 | 异构设备、不同模型和不同引擎协作 |
| 张量并行 | 单层矩阵/tensor shard | 高频 collective | 同构加速器与高速低延迟互联 |

完整推理 Worker 不拆模型，而是把整个请求交给一台能独立加载模型的设备。以上方式的通信、失败恢复和性能模型不同，不能共用一个“分布式：是/否”字段替代真实执行信息。

## 5. PyTorch 层流水线约束

- 单条生成链具有逐层数据依赖；多节点不必然降低单请求延迟，主要价值可能是装下更大模型或提高多请求吞吐。
- 分层依据是实测计算能力、可用内存、Embedding/LM Head 成本和网络状态，不按节点数平均，也不按“主/从”角色先验地让主节点少算。
- 图算法只负责候选节点与路径编排；最终层数还需满足模型结构、内存和首末段职责约束。
- TCP 的可靠传输不等于应用级可恢复。系统仍需任务身份、超时、取消、ready ACK、幂等和迟到结果过滤。

详细协议见[分布式推理流水线实施计划](分布式推理流水线实施计划.md)，多 Provider 路线见[混合分布式推理体系规划](混合分布式推理体系规划.md)。
