## 1. 当前状态：FIFO + 单道串行

### 1.1 核心组件

```
PipelineQueue (deque + RLock + _process_loop 后台线程)
     │
     ├── _inference_lock (Lock, 非阻塞)  → GPU 互斥
     └── pipeline_queue._lock (RLock)    → 入队原子检查
```

### 1.2 现行策略

```
请求到达顺序:  A(1024tk) → B(64tk) → C(256tk)
执行顺序:      A(50s)    → B(3s)   → C(12s)
B 的总响应时间: 53s（等了 50s 才跑了 3s）
```

纯 FIFO，`PIPELINE_MAX_CONCURRENT = 1`，无优先级无抢占。短请求 B 排在长请求 A 后面时，实际只需 3s 的计算却要等 50s——**护航效应（convoy effect）**。

### 1.3 类比

| OS 概念 | 本系统对应 |
|---------|-----------|
| CPU burst | token 生成阶段（纯计算，~50-100ms/token） |
| I/O burst | 节点间 hidden_states 传输（~5-20ms/step） |
| 进程/线程 | 推理请求（PipelineQueue 中的 task） |
| 就绪队列 | `deque` 中的待执行任务 |
| 上下文切换 | KV cache checkpoint → restore（~100-500ms） |
| 时间片 | 当前无（一次性跑到完成或 max_tokens） |

## 2. 目标调度模型：三级反馈队列 (MLFQ) + 协同抢占 + 老化防饥饿

### 2.1 整体架构

仿照 OS 多级反馈队列 (Multi-Level Feedback Queue)，按请求的 **max_tokens**（对应用户设定的生成长度上限）分为三个优先级队列：

```
                        新请求到达
                            │
                    ┌───────┴───────┐
                    │  按 max_tokens │
                    │  分配到对应级   │
                    └───────┬───────┘
                            │
         ┌──────────────────┼──────────────────┐
         ▼                  ▼                  ▼
   ┌──────────┐      ┌──────────┐      ┌──────────┐
   │  Q0: 交互 │      │  Q1: 普通 │      │  Q2: 批量 │
   │ ≤128 tok │      │ ≤512 tok │      │ >512 tok │
   │ 策略: SJF │      │ 策略: SJF │      │ 策略: SJF │
   │ 可抢占    │      │ 不可抢占  │      │ 不可抢占  │
   └──────────┘      └──────────┘      └──────────┘
         │                  │                  │
         │    ┌─────────────┤                  │
         │    │  老化提升    │                  │
         │    ▼             ▼                  │
         │ 等待 > 60s     等待 > 120s           │
         │ 提升到 Q0      提升到 Q1             │
         │                                      │
         └──────────────────────────────────────┘
                            │
                    调度器选择下一个任务:
                    优先 Q0 → Q1 → Q2
                    同级内 SJF 排序
```

### 2.2 三级队列定义

| 队列 | max_tokens | 标签 | 典型场景 | 调度策略 | 可抢占 | 初始时间片 |
|------|-----------|------|---------|---------|--------|-----------|
| **Q0 (交互)** | ≤ 128 | `interactive` | 简短问答、翻译、补全 | SJF | ✅ 可抢占 Q1/Q2 | 无限制（短到不需要） |
| **Q1 (普通)** | ≤ 512 | `normal` | 常规对话、代码生成 | SJF + 老化 | ❌ | 无限制 |
| **Q2 (批量)** | > 512 | `batch` | 长文生成、报告、分析 | SJF + 老化 | ❌ | 无限制 |

**为什么 Q0 无时间片限制**：≤128 token 的请求本身只需 ~5-8s，切分毫无必要。Q0 的特权是"可以插队"而非"被切碎"。

### 2.3 调度算法伪代码

```python
def schedule_next(self) -> Optional[Task]:
    """从三级队列中选择下一个要执行的任务。"""
    self._apply_aging()          # 1. 老化提升（饥饿保护）
    self._apply_sjf_sorting()    # 2. 每级队列内部 SJF 排序

    # 3. 严格优先级：Q0 → Q1 → Q2
    for queue in [self._q0, self._q1, self._q2]:
        if queue:
            return queue.popleft()
    return None


def _apply_aging(self):
    """老化提升：等待过久的请求逐级上浮。"""
    now = time.time()
    # Q2 → Q1: 等待超过 AGING_Q2_TO_Q1 秒
    aged_up = [t for t in self._q2
               if now - t.created_at > AGING_Q2_TO_Q1_SECONDS]
    for t in aged_up:
        self._q2.remove(t)
        self._q1.append(t)
        t.priority_level = 1
        logger.info(f"老化提升 Q2→Q1: {t.task_id} (等待 {now - t.created_at:.0f}s)")

    # Q1 → Q0: 等待超过 AGING_Q1_TO_Q0 秒
    aged_up = [t for t in self._q1
               if now - t.created_at > AGING_Q1_TO_Q0_SECONDS]
    for t in aged_up:
        self._q1.remove(t)
        self._q0.append(t)
        t.priority_level = 0
        logger.info(f"老化提升 Q1→Q0: {t.task_id} (等待 {now - t.created_at:.0f}s)")


def _apply_sjf_sorting(self):
    """每级队列内部按预估剩余时间升序排列。"""
    for q in [self._q0, self._q1, self._q2]:
        q_sorted = sorted(q, key=lambda t: t.estimated_remaining_seconds())
        q.clear()
        q.extend(q_sorted)
```

### 2.4 老化参数

| 参数 | 默认值 | 含义 |
|------|--------|------|
| `AGING_Q1_TO_Q0_SECONDS` | 60 | Q1 请求等待 > 60s → 提升到 Q0 |
| `AGING_Q2_TO_Q1_SECONDS` | 120 | Q2 请求等待 > 120s → 提升到 Q1 |
| `AGING_MAX_WAIT_SECONDS` | 300 | 绝对上限：任何请求等待 > 300s → 直接置顶 |

**老化效果**: 一个 1024 token 的 Q2 请求，初始排在所有 Q0/Q1 后面。等待 2 分钟后升到 Q1，再等 1 分钟升到 Q0。5 分钟时如果还没排到，强制置顶。**没有任何请求会永久饥饿**。

## 3. 协同抢占机制

### 3.1 为什么是"协同"而非"强制"

LLM 推理与 CPU 进程不同——推理状态是分布在多个 GPU 节点上的 KV cache，不能像 CPU 寄存器那样在任意指令边界保存/恢复。但每个 decode step 之间存在天然的安全边界（~50-100ms 间隔），可以在此时刻检查抢占标志。

### 3.2 抢占规则

```
可抢占的场景（仅 Q0 拥有抢占权）:
  Q0 到达 + 当前执行的是 Q1 或 Q2 → 触发抢占检查

不可抢占的场景:
  当前执行的是 Q0 → 不抢占（同级，且 Q0 本身就短）
  Q1 到达 + 当前 Q2 → 不抢占（只有 Q0 可抢占）
```

### 3.3 抢占流程

```
当前 Q1/Q2 任务在 run_pipeline() 中逐 token 生成
                    │
         ┌──────────▼──────────┐
         │ 每个 decode step 后  │
         │ 轮询 preempt_flag    │
         └──────────┬──────────┘
                    │
          ┌─────────┴─────────┐
          │ preempt_flag 置位? │
          └─────────┬─────────┘
              No    │    Yes
              │     │
              ▼     ▼
         继续生成  ┌─────────────────────┐
                   │ checkpoint_kv_cache  │
                   │ (CPU pin memory)    │
                   ├─────────────────────┤
                   │ 释放 _inference_lock │
                   ├─────────────────────┤
                   │ 执行 Q0 任务         │
                   ├─────────────────────┤
                   │ restore_kv_cache     │
                   ├─────────────────────┤
                   │ 继续被抢占任务       │
                   └─────────────────────┘
```

### 3.4 KV Cache Checkpoint / Restore

```python
def checkpoint_kv_cache(self, task_id: str) -> dict:
    """
    将分布式流水线各节点的 KV cache 保存到 CPU pinned memory。

    每层 KV cache: 2 tensors (K, V), shape (1, n_heads, seq_len, head_dim)
    Qwen-1.8B: 24 层 × 2 × (1, 16, seq, 128) × fp16
    seq_len=100 → ~1.2 MB, seq_len=500 → ~6 MB
    3 节点各 8 层 → 每节点 ~2 MB (seq=500)
    总计: ~6 MB → 拷贝耗时 < 10ms (PCIe 3.0 ×16)
    """
    checkpoints = {}
    for node_id, node_kv in self._pipeline_kv_cache.get(task_id, {}).items():
        checkpoints[node_id] = {
            'k': [k.cpu().pin_memory() for k in node_kv['k']],
            'v': [v.cpu().pin_memory() for v in node_kv['v']],
            'seq_len': node_kv['seq_len'],
        }
    return checkpoints


def restore_kv_cache(self, task_id: str, checkpoint: dict) -> None:
    """
    从 CPU pinned memory 恢复 KV cache 到各节点 GPU。

    恢复后各节点可直接从下一 token 继续 decode，无需重新 prefill。
    """
    for node_id, state in checkpoint.items():
        node_kv = {
            'k': [k.cuda(non_blocking=True) for k in state['k']],
            'v': [v.cuda(non_blocking=True) for v in state['v']],
            'seq_len': state['seq_len'],
        }
        self._pipeline_kv_cache[task_id][node_id] = node_kv
    torch.cuda.synchronize()  # 等待所有 H2D 传输完成
```

**Checkpoint 成本估算**：

| seq_len | KV cache 大小 (fp16) | CPU ← GPU 拷贝 | GPU ← CPU 拷贝 | 总计 |
|---------|---------------------|---------------|---------------|------|
| 64 | ~0.8 MB | < 2ms | < 2ms | ~4ms |
| 256 | ~3.2 MB | < 5ms | < 5ms | ~10ms |
| 512 | ~6.4 MB | < 10ms | < 10ms | ~20ms |

> 对于 50s 的长请求，额外 20ms 的 checkpoint 开销可以忽略。

### 3.5 抢占节制：防抖动

```python
MIN_PREEMPT_INTERVAL = 10.0  # 两次抢占之间最少间隔（秒）
MIN_GENERATED_BEFORE_PREEMPT = 16  # 至少生成 16 token 后才接受抢占
```

防止 Q0 请求频繁到达导致 Q2 任务被反复打断（thrashing）。

### 4.2 手动操作

每个排队任务支持：
- **↑/↓** — 在同级队列内调序
- **提升/降级** — 手动跨级移动（限管理员）
- **置顶** — 强制放到 Q0 队首
- **取消** — 移除排队任务

## 6. 调度效果对比（模拟）

假设 4 个请求同时到达：A(1024tk, Q2)、B(64tk, Q0)、C(256tk, Q1)、D(96tk, Q0)。

### FIFO（当前）

```
时间轴: |── A (50s) ──|── C (12s) ──|── D (5s) ──|── B (3s) ──|
响应时间:   A=50s       C=62s         D=67s        B=70s
平均响应:  (50+62+67+70)/4 = 62.3s
```

### MLFQ + SJF + 老化（新方案）

```
执行顺序: B(Q0,SJF) → D(Q0,SJF) → C(Q1,SJF) → A(Q2,SJF)
时间轴: |─ B (3s) ─|─ D (5s) ─|─ C (12s) ─|── A (50s) ──|
响应时间:  B=3s       D=8s       C=20s        A=70s
平均响应:  (3+8+20+70)/4 = 25.3s   （比 FIFO 提升 59%）
```

### MLFQ + 抢占（A 正在执行，B 到达）

```
A 执行到 token 200 时 B 到达（Q0 可抢占 Q2）:
时间轴: |─ A(10s) ─| checkpoint(20ms) |─ B(3s) ─| restore(20ms) |─ A 续(40s) ─|
B 响应时间: 10s（vs FIFO 的 53s）
A 总时间:   53s（vs FIFO 的 50s，多了 3s 的 checkpoint 开销）
```

## 8. 风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| KV checkpoint 体积过大（长序列） | checkpoint 耗时超标 | `PREEMPT_MAX_OVERHEAD_MS` 阈值，超限自动禁用抢占 |
| 抢占导致被中断任务总时间变长 | 用户体验下降 | `MIN_PREEMPT_INTERVAL` + `MIN_GENERATED_BEFORE_PREEMPT` 防抖动 |
| 老化参数设置不当 | 饥饿或频繁切换 | 前端面板可调节 + `PIPELINE_AGING_MAX_WAIT_SECONDS` 硬上限 |
| 抢占时 TCP 连接超时 | 从节点误判主节点离线 | 抢占前发送 `PIPELINE_PAUSE` → 抢占后发送 `PIPELINE_RESUME` |
