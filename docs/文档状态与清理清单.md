# 文档状态与维护规则

> 更新日期：2026-07-14
>
> 状态：现行

## 1. 状态定义

| 状态 | 含义 |
|------|------|
| 现行 | 描述当前代码、协议或操作方法，需要随实现持续维护 |
| 规划 | 尚未全部实现的目标方案，不能作为当前能力说明 |
| 部分实施 | 部分内容已经落地，未完成部分仍需明确标注和验收 |
| 历史参考 | 部分内容仍有参考价值，但状态、接口、行号或版本信息可能过时 |
| 废弃（待手动删除） | 已被其他文档、代码或测试取代；不再维护，确认无必要引用后可手动删除 |
| 已剔除 | 文件已不在仓库中，不应恢复其链接或继续引用 |

文档中固定的模型、节点数量、网络环境和硬件型号，除非明确标注为测试样例或历史条件，否则不得视为系统边界。QLH 的设备范围包括 Windows/Linux 台式机、工作站、服务器、笔记本、Android 手机和平板，以及后续通过能力协议接入的其他边缘设备。

## 2. 后续维护规则

1. 新文档必须在标题后声明状态、更新时间和适用范围。
2. 现行文档必须与当前代码、接口和自动化测试保持一致；具体函数签名和行为以源码为最终依据。
3. 规划文档必须明确已实现与未实现边界，不得把预研结论或计划能力写成当前支持。
4. 计划落地后，应将文档更新为现行说明，或标记为历史参考、废弃；不得长期保留错误的“待实施”状态。
5. 废弃文档必须在开头注明替代入口和废弃原因；删除操作由维护者手动执行。
6. 已剔除文件不得继续出现在 README、文档索引或现行文档链接中。
7. README 只链接现行入口和仍在执行的规划，不把废弃或纯历史文档列为推荐阅读。
8. 文档中的源码路径、接口名称、测试路径、版本号和状态结论发生变化时，应在同一修改中同步更新。
9. 模型名称相同不代表模型工件、量化格式、tokenizer 或执行引擎相同；涉及分布式执行时必须描述实际 artifact 和 Provider 边界。
10. “分布式”不得只用是/否表示；文档和测试应说明实际执行模式、Provider、参与节点、故障边界和降级路径。
11. 历史计划中的源码行号、工时估算和设备条件不构成当前接口或性能承诺。
12. 文档不得保存数据库密码、SMTP 授权码、集群密钥或其他真实凭据；发现泄露后应立即脱敏并轮换对应凭据。
