## 一、Linux 适配问题（v0.1.7）

来源：`v0.1.7-Linux适配BUG清单.md`

### P1 — 高优先级（7个）✅ 全部已修复

#### ✅ BUG-P1-05 | control-cuda:26 | "DLL" 应为 ".so"
- **状态**: ✅ 已修复
- **根因**：从 Windows 文档复制粘贴的残余错误
- **影响**：包描述出现 Windows 术语，专业度受损
- **修复**：已改为"CUDA 12.x 共享库 (.so)"
- **文件**：`packaging/linux/control-cuda#L26`

#### ✅ BUG-P1-06 | qlh-edge-inference.desktop:7 + build-deb.sh:83 | 图标名不匹配
- **状态**: ✅ 已修复
- **根因**：desktop 文件写 `Icon=qlh-edge-inference`，但 build-deb.sh 将图标复制为 `qlh.png`
- **影响**：Linux 桌面上应用图标不显示
- **修复**：已统一使用 `Icon=qlh`（匹配 `qlh.png`）
- **文件**：`packaging/linux/qlh-edge-inference.desktop#L7`、`packaging/linux/build-deb.sh#L89-90`

#### ✅ BUG-P1-07 | launcher.py:234 | "yesno" 映射为三按钮 MessageBox
- **状态**: ✅ 已修复
- **根因**：`_MB_YESNO = 0x00000004` 未定义，"yesno" 被错误映射为 `_MB_YESNOCANCEL`
- **影响**：Windows 上对话框出现多余的"取消"按钮
- **修复**：已添加 `_MB_YESNO = 0x00000004` 常量，并正确映射 `"yesno": _MB_YESNO | _MB_ICONQUESTION`
- **文件**：`packaging/launcher.py#L122`、`packaging/launcher.py#L243`

#### ✅ BUG-P1-08 | launcher.py:191-195 | Linux zenity "取消" 被映射为 "是"
- **状态**: ✅ 已修复
- **根因**：zenity `--extra-button=取消` 点击后退出码为 0（与"是"相同）
- **影响**：Linux 上点"取消"实际执行为"是"
- **修复**：已添加检查 `rc.stdout` 中是否包含按钮标签"取消"，若是则返回 `_IDCANCEL`
- **文件**：`packaging/launcher.py#L198-200`

#### ✅ BUG-P1-09 | launcher.sh:17 | `activate_this.py` 已废弃
- **状态**: ✅ 不适用
- **说明**：`launcher.sh` 文件已不存在，已被 Python 版本的 `launcher.py` 替代
- **文件**：N/A（文件已删除）

#### ✅ BUG-P1-10 | postinst:10 | `chmod 777` 安全风险
- **状态**: ✅ 已修复
- **根因**：`chmod 777` 使目录对所有用户可写
- **影响**：任意本地用户可删除/替换模型文件（模型投毒）
- **修复**：models 目录已改为 `chmod 755`，logs 目录使用 `chmod 1777`（sticky bit，适合 DynamicUser 写入日志）
- **文件**：`packaging/linux/postinst#L10-11`

#### ✅ BUG-P1-11 | device_profiler.py:725/771 | nvidia-smi 与 lspci 可能重复检测
- **状态**: ✅ 已修复
- **根因**：nvidia-smi 名称 ≠ lspci 名称，字符串全名匹配无法去重
- **影响**：同一 GPU 重复出现
- **修复**：已添加多层去重逻辑：(1) `seen_names` 集合检查（第725-726行、第771-772行），(2) NVIDIA 特定去重（第776-779行），(3) `dgpu_kw` 包含 "radeon pro" 避免 AMD Radeon PRO 误分类（第788行）
- **文件**：`src/device_profiler.py#L725-726, L771-772, L776-779, L788`

## 二、节点列表问题

来源：`节点列表BUG审查报告.md`

### Critical（2个）

#### C1 — `update_max_nodes()` 扩容时创建幽灵 `client{i}` 槽位
- **影响**：扩容后再缩容，节点列表里会残留永久离线幽灵节点
- **修复方向**：扩容不再创建空槽位，只修改上限
- **文件**：`src/scheduler.py#L3874-L3881`

#### C2 — `register_node()` TCP 注册路径无容量检查
- **影响**：任何 TCP 客户端都可以无限制注册，完全绕过 `max_nodes` 限制
- **修复方向**：在创建新节点前检查容量
- **文件**：`src/scheduler.py#L1159-L1171`

### Medium（3个）

#### M1 — `init_nodes()` 从数据库恢复所有历史节点，含陈旧/测试数据
- **影响**：跨重启累积，节点列表越来越长
- **修复方向**：增加可选的离线过期逻辑
- **文件**：`src/scheduler.py#L1043-L1051`

#### M2 — 离线节点占用容量，阻塞新注册
- **影响**：历史累积的离线节点会阻塞新节点注册
- **修复方向**：容量检查仅统计在线节点
- **文件**：`src/scheduler.py#L5927-L5930`

#### M3 — `get_nodes()` 无过滤机制，前端展示所有历史节点
- **影响**：离线、幽灵、陈旧、测试节点混在一起
- **修复方向**：前端增加"仅显示在线"过滤
- **文件**：`src/api_server.py#L2324-L2331`、`frontend/src/components/AdminPanel.jsx#L844-L847`

### Low（4个）

#### L1 — `deregister_node()` 仅标记 offline，不删除记录
- **说明**：设计如此，但建议在 UI 中明确区分"注销"和"删除"
- **文件**：`src/scheduler.py#L1223-L1268`

#### L2 — `DEFAULT_LAYER_CONFIG` 硬编码 `client1`/`client2`
- **影响**：旧固定槽位架构的遗留默认值
- **修复方向**：移除或改为空字典
- **文件**：`src/config.py#L90-L94`

#### L3 — `NodeRole.client_ids()` 死代码
- **修复方向**：删除或使其有实际作用
- **文件**：`src/scheduler.py#L87-L89`

#### L4 — `manual_register_node()` 可"更新"任意已存在节点
- **影响**：用户体验困惑
- **修复方向**：对真实 TCP 注册过的节点，手动更新不应改变关键标识信息
- **文件**：`src/scheduler.py#L5900-L5925`

## 三、分布式通信与日志问题

来源：`分布式通信与日志审查报告.md`

### 分布式通信（4个）

#### C1 (Medium) — 注册被拒后连接未关闭 + 泄露给调度器
- **影响**：调度器 `self.nodes` 被污染，被拒客户端占用连接资源
- **修复方向**：被拒后跳过 `on_message` 回调或关闭连接
- **文件**：`src/tcp_comm.py#L946-L1051`

#### C2 (Medium) — `TCPServer.clients` 字典无线程同步
- **影响**：低概率竞态崩溃
- **修复方向**：将 `self.clients` 操作包装在 `threading.Lock` 中
- **文件**：`src/tcp_comm.py#L802`

#### C3 (Low) — `_on_tcp_message` 无 `else` 分支处理未知消息类型
- **修复方向**：添加 `else: logger.debug(...)`
- **文件**：`src/scheduler.py#L2004-L2165`

#### C4 (Low) — `TCPClient.on_heartbeat` 回调异常被静默吞没
- **修复方向**：添加 `logger.debug(..., exc_info=True)`
- **文件**：`src/tcp_comm.py#L1459-L1463`

### 日志功能（6个）

#### L1 (Medium) — `traceback.print_exc()` 替代 `logger.error(..., exc_info=True)`
- **影响**：异常堆栈进 stderr，不进日志文件
- **修复**：替换为 `logger.error(..., exc_info=True)`
- **文件**：`src/scheduler.py#L1522, L4479, L5496, L5587`

#### L2 (Medium) — `tcp_comm.py` 错误日志缺少 `exc_info=True`
- **影响**：TCP 通信问题缺少调用栈
- **修复**：添加 `exc_info=True`
- **文件**：`src/tcp_comm.py#L860, L925, L944, L1142, L1376`

#### L3 (Low) — 调度器 `_on_tcp_disconnect` 无日志
- **修复**：添加 `logger.info(...)`
- **文件**：`src/scheduler.py#L2167-L2176`

#### L4 (Low) — `pipeline_pause`/`pipeline_resume` 仅 DEBUG 级别日志
- **修复**：升为 `logger.info`
- **文件**：`src/scheduler.py#L2154-L2160`

#### L5 (Low) — `detect_lan_ip`/`get_mac_addresses` 多层回退无声失败
- **修复**：添加 DEBUG 日志
- **文件**：`src/tcp_comm.py#L272-L408, L411-L550`

#### L6 (Low) — `_process_loop` 启动/停止无日志
