# 日志功能完善建议

> 日期: 2026-07-08  
> 范围: 后端日志配置、日志管理 API、分布式通信日志、调度队列日志、前端日志查看与诊断输出  
> 状态: 仅文档记录，暂不实施代码改动

---

## 现状评估

当前日志功能已经具备基础可用性：

- 后端通过 `RotatingFileHandler` 写入日志文件，支持 UTF-8 编码、按大小滚动、保留备份。
- `uvicorn` 日志会传播到 root logger，生产运行时可进入同一日志文件。
- 前端设置页提供日志列表、查看、删除、清理全部、复制全部等基础运维能力。
- 日志文件 API 做了文件名白名单校验，避免路径穿越。
- 日志读取最多返回末尾 1MB，避免大文件拖垮前端。
- 分布式通信、调度队列、抢占、断连、重连等核心路径已有较多业务日志。

综合评分：**7/10**。

该水平足以支撑本地调试、开发联调和小规模边缘集群运行；但若要长期无人值守运行，仍建议继续补强 API 异常日志、请求链路追踪、权限控制和前端诊断输出管理。

---

## 主要缺口

### 1. API 异常返回多，后端日志少

部分 API 中存在如下模式：

```python
except Exception as e:
    raise HTTPException(500, f"操作失败: {e}")
```

问题是：

- 前端能看到错误信息；
- 但后端日志文件中可能没有完整堆栈；
- 打包部署后排查成本较高。

建议逐步改为：

```python
except Exception as e:
    logger.error("操作失败", exc_info=True)
    raise HTTPException(500, f"操作失败: {e}")
```

优先覆盖：

- 模型加载 / 切换；
- 聊天与流式聊天；
- 分布式集群配置；
- 日志读取 / 删除；
- 审查工单；
- 邮件轮询；
- 会话与对话持久化。

---

### 2. 缺少 request_id / trace_id

当前主要依赖 `uvicorn.access` 和业务日志，但同一次请求在多条日志之间缺少统一关联 ID。

建议增加 FastAPI middleware：

- 每个请求生成 `request_id`；
- 写入响应头 `X-Request-ID`；
- 记录 method、path、status、duration_ms、request_id；
- 未捕获异常记录 `exc_info=True`；
- 前端错误 toast 可附带 request_id，方便用户反馈。

推荐日志格式：

```text
event=http_request request_id=... method=POST path=/api/chat status=200 duration_ms=1234
```

后续可将 request_id 继续贯穿：

- chat 请求；
- PipelineQueue task_id；
- 分布式 pipeline task_id；
- TCP 消息；
- 前端错误提示。

---

### 3. 日志 API 缺少权限控制

当前 `/api/logs` 系列接口具备文件名安全校验，但建议增加访问权限控制。

风险：

- 日志可能包含 IP、MAC、节点 ID、模型路径、异常堆栈等敏感运维信息；
- 删除日志属于高影响操作；
- 局域网部署时不应默认让所有访问者读取或删除日志。

建议：

- 仅主节点 / 管理员允许访问；
- 删除日志二次确认由前端保留；
- 后端也记录日志删除操作；
- 如果后续有用户体系，应接入鉴权。

---

### 4. 前端 console 日志较多

`ChatPanel.jsx` 中存在大量开发诊断 `console.log`，例如：

- 会话切换；
- 历史记录加载；
- 消息数组变化；
- 发送过程；
- 渲染分支。

问题：

- 生产环境控制台刷屏；
- 真实错误不易发现；
- 可能泄露会话状态或用户操作节奏；
- 高频 render 日志可能影响性能。

建议封装：

```js
const DEBUG_CHAT = import.meta.env.DEV;
if (DEBUG_CHAT) console.log(...);
```

或统一实现：

```js
function debugLog(scope, ...args) {
  if (import.meta.env.DEV) console.log(`[${scope}]`, ...args);
}
```

---

### 5. 日志删除存在 handler 重建窗口

当前日志删除流程会：

1. 关闭 logging handlers；
2. 删除文件；
3. 重新 `setup_logging()`。

这是 Windows 下避免文件占用的实用方案，但存在短暂并发窗口：

- 其他线程正在写日志时 handler 被关闭；
- 同时读取/删除日志可能竞态；
- 删除当天日志后会立即创建新的当天日志。

建议：

- 增加全局日志管理锁；
- 删除前后记录操作日志；
- 前端提示“当前日志会重新生成”；
- 若失败，返回更明确的失败原因。

---

### 6. 日志查看能力仍偏基础

当前前端支持查看和复制，但缺少：

- 按 level 过滤；
- 关键词搜索；
- 自动滚动到末尾；
- 下载单个日志；
- 下载全部日志压缩包；
- 显示日志是否被截断；
- 按时间范围过滤。

建议后续增强日志查看体验，优先：

1. 显示 `truncated` 标记；
2. 增加搜索框；
3. 增加“下载日志”；
4. 增加 level filter（ERROR/WARNING/INFO/DEBUG）。

## logging-module 可借鉴设计

> 背景：已有开发者提供了 `logging-module/` 参考实现，包含 Python 后端日志模块、日志 API 路由和 React `LogViewer`。该模块不建议直接复制覆盖进当前项目，但其中有若干设计值得吸收到现有日志体系中。

### 总体结论

当前项目已经有：

- PC 后端 `setup_logging()`、`RotatingFileHandler`、`/api/logs` 文件管理接口；
- PC 前端设置页日志列表、查看、删除、清理、复制能力；
- Android 原生 `QlhLogger.kt` 和设置页日志查看/复制/分享/清理能力。

因此更省事、更稳的路线是：**保留当前日志基础，按需吸收 `logging-module` 的局部设计**。不要按 `INTEGRATION_GUIDE.md` 直接复制 `logger.py` / `logging_routes.py` / `LogViewer.jsx`，否则会和当前 `/api/logs` 路由语义、前端 API 调用和日志删除流程冲突。

### 值得借鉴的后端思路

| 设计 | 参考实现 | 建议吸收方式 |
|------|----------|--------------|
| 内存环形日志缓冲区 | `deque(maxlen=BUFFER_SIZE)` + 自定义 `logging.Handler` | 在当前 `setup_logging()` 中新增一个 `BufferHandler`，用于实时查看最近日志 |
| 最近日志查询 API | `GET /api/logs` 返回内存日志 | 不复用该路径；新增 `GET /api/logs/recent`，避免破坏现有文件列表接口 |
| level / logger / node 过滤 | `level`、`name`、`node_id` 查询参数 | 用于 `/api/logs/recent?level=ERROR&name=scheduler&node_id=...` |
| node_id / device_ip 注入 | `logging.Filter` / 自定义 Formatter | 可在 scheduler 初始化后设置节点上下文，让 PC 日志带节点标识 |
| 日志统计信息 | 文件数、总大小、缓冲区数量、保留天数 | 新增 `GET /api/logs/stats`，供前端状态栏展示 |
| 单文件下载接口 | `FileResponse` 下载日志文件 | 当前已有读取接口，可补 `GET /api/logs/{filename}/download` 或 `GET /api/logs/download?name=...` |
| 错误日志单独文件 | `error.log` 记录 WARNING+ | 可选；若实现，需与当前 `qlh-YYYY-MM-DD.log` 命名策略统一 |

推荐新增的数据结构：

```python
{
    "timestamp": "2026-07-10 12:00:00",
    "level": "ERROR",
    "levelno": 40,
    "node_id": "master",
    "device_ip": "192.168.1.5",
    "name": "scheduler",
    "message": "pipeline failed",
    "filename": "scheduler.py",
    "lineno": 1234,
    "funcName": "dispatch",
}
```

### 值得借鉴的前端思路

`LogViewer.jsx` 的交互设计可以拆出来融入当前设置页，或远期做成独立日志页：

- 自动刷新最近日志，默认 3 秒刷新一次；
- level 快捷过滤：全部 / ERROR / WARNING / INFO / DEBUG；
- 按节点过滤；
- 前端关键词搜索，覆盖时间、级别、节点、logger 名称、消息内容；
- 自动滚动到底部，便于观察实时日志；
- 复制当前过滤后的全部日志；
- 双击复制单条日志；
- 日志文件折叠面板，展示文件名、大小、修改时间；
- 单文件下载；
- 显示日志目录、文件数、总大小、缓冲区使用量。

当前项目已有设置页日志管理，短期建议先增强 `SettingsModal.jsx`，不要急着新增主导航 `日志` Tab。等日志能力稳定后，再考虑把 `LogViewer` 独立成完整页面。

### 不建议直接照搬的部分

| 模块内容 | 问题 | 建议 |
|----------|------|------|
| `GET /api/logs` 返回最近内存日志 | 当前项目 `GET /api/logs` 已用于文件列表 | 改为 `/api/logs/recent` |
| `GET /api/logs/files` | 当前 `GET /api/logs/{filename:path}` 可能抢先匹配 `files` | 避免新增该路径，沿用当前 `/api/logs` 文件列表 |
| `DELETE /api/logs?confirm=yes` | 当前删除全部日志不需要 confirm 参数，前端 API 语义不同 | 保留当前接口，前端二次确认即可 |
| `clear_logs()` 直接删除文件 | 未关闭 file handlers，Windows 可能删不掉，Linux 可能继续写入已删除 inode | 继续使用当前关闭 handler -> 删除 -> 重建的策略，并加锁 |
| `setup_logging()` 幂等后不可强制重建 | 删除日志后 `setup_logging()` 可能直接返回，handler 不重建 | 需要 `force=True` 或独立 `reset_logging()` |
| 无权限控制的日志 API | 局域网用户可读取/删除日志 | 必须加主节点/管理员权限边界 |
| 直接新增 `LogViewer` 主 Tab | 当前导航和设置页已有日志入口 | 先增强设置页，远期再独立页面 |

### 建议改造路径

#### 阶段 L0：保留现状，补安全边界

1. 保留当前 `/api/logs` 文件列表、`/api/logs/{filename}` 读取、删除接口。
2. 给日志 API 增加主节点 / 管理员权限校验。
3. 日志删除加全局锁，避免 handler 重建窗口。
4. 删除当前正在写入的日志时，在前端提示“当前日志会重新生成”。

#### 阶段 L1：吸收内存实时日志

1. 在当前 `setup_logging()` 中增加内存环形缓冲 handler。
2. 新增 `GET /api/logs/recent`：

```text
GET /api/logs/recent?limit=200&level=ERROR&name=scheduler&node_id=master
```

3. 返回最近日志、缓冲区总数、是否截断等信息。
4. 不改变当前文件日志接口，保证现有设置页不破。

#### 阶段 L2：增强前端日志查看

1. 在 `SettingsModal.jsx` 中增加 level filter 和关键词搜索；
2. 显示 `truncated` 标记；
3. 增加“下载日志”按钮；
4. 增加“最近日志”实时视图；
5. 生产构建下隐藏或降噪前端 `console.log`。

#### 阶段 L3：长期运维增强

1. 引入 request_id middleware；
2. 日志字段半结构化；
3. 前端 toast 携带 request_id；
4. 支持日志压缩包导出；
5. 远期考虑多节点日志聚合。

### 平台适用边界

| 平台 | 当前项目现状 | `logging-module` 可借鉴部分 | 注意事项 |
|------|--------------|-----------------------------|----------|
| Windows PC | 已支持文件日志和删除前关闭 handler | 内存缓冲、实时查看、过滤 UI | 删除日志必须关闭 handler 并重建 |
| Linux PC | 已支持文件日志 | 内存缓冲、统计、下载、过滤 UI | 删除已打开日志文件前也应关闭 handler，避免进程继续写入已 unlink 文件 |
| Android 普通版 | 已有 `QlhLogger.kt` | 仅借鉴 UI 思路，如 level filter / 搜索 / 分享 | 不复用 Python 模块；继续使用 Kotlin 原生日志 |
| Android 极简版 | 不强调日志管理 | 基本不借鉴 | 保持轻量，不加入完整日志管理入口 |

为避免和当前接口冲突，建议保留现有语义并新增接口：

| `GET /api/logs/{filename}` | 现有：读取单个日志文件 |

| `DELETE /api/logs` | 现有：删除全部日志文件 |

## 半结构化日志字段规范（L5）

> 已于 2026-07-10 实施。以下字段为日志格式化字符串中的标准 key=value 对。

### 核心字段

| 字段 | 类型 | 含义 | 示例 | 适用范围 |
|------|------|------|------|----------|
| `event` | string | 事件类型标识 | `http_request`, `task_enqueue`, `task_dispatch`, `task_complete`, `task_failed`, `infer_forward`, `infer_forward_recv`, `log_export`, `log_retention_cleanup`, `log_aggregation_sent`, `log_aggregation_recv`, `client_error` | 所有模块 |
| `request_id` | hex(32) | HTTP 请求 ID（由 middleware 生成或继承 `X-Request-ID` 头） | `a1b2c3d4e5f6...` | API / Scheduler / TCP |
| `task_id` | string | 推理任务 ID（由调度器生成：`q_{uuid}` 或 `task_{timestamp}`） | `q_a1b2c3d4e5f6` | Scheduler / TCP |
| `node_id` | string | 节点标识 | `master`, `desktop-pc1`, `android-phone2` | 所有模块 |
| `device_ip` | string | 设备 IP 地址 | `192.168.1.100` | 日志记录上下文 |

### 辅助字段

| 字段 | 类型 | 含义 | 示例 |
|------|------|------|------|
| `client_id` | string | TCP 客户端 ID | `desktop-pc1` |
| `role` | string | 节点角色 | `master`, `client` |
| `method` | string | HTTP 方法 | `GET`, `POST`, `DELETE` |
| `path` | string | HTTP 请求路径 | `/api/chat` |
| `status` | int | HTTP 状态码 | `200`, `500` |
| `duration_ms` | float | 请求耗时（毫秒） | `1234.5` |
| `prompt_len` | int | 输入 prompt 长度 | `256` |
| `max_tokens` | int | 最大生成 token 数 | `512` |
| `elapsed` | float | 任务耗时（秒） | `3.5` |
| `error` | string | 错误信息摘要 | `CUDA out of memory` |
| `source` | string | 错误来源（前端上报） | `window.onerror`, `unhandledrejection` |

### 日志格式示例

```
2026-07-10 15:30:00 [INFO] request_id=a1b2c3d4 api_server: event=http_request method=POST path=/api/chat status=200 duration_ms=1234
2026-07-10 15:30:01 [INFO] request_id=a1b2c3d4 scheduler: event=task_enqueue task_id=q_x7y8z9a0b1c2 request_id=a1b2c3d4 priority_level=Q0 max_tokens=128 total_depth=3
2026-07-10 15:30:02 [INFO] request_id=a1b2c3d4 scheduler: event=task_dispatch task_id=q_x7y8z9a0b1c2 request_id=a1b2c3d4 Q0 orig=Q0 wait=2s
2026-07-10 15:30:05 [INFO] request_id=a1b2c3d4 scheduler: event=task_complete task_id=q_x7y8z9a0b1c2 request_id=a1b2c3d4 elapsed=3.1s
2026-07-10 15:30:05 [ERROR] request_id=b2c3d4e5 api_server: event=client_error source=window.onerror message=TypeError: ... url=http://...
2026-07-10 16:00:00 [INFO] request_id=- api_server: event=log_retention_cleanup deleted_age=5 deleted_size=2 remaining_files=12
```

### 链路串联方法

一次分布式推理可通过以下方式串联全部日志：

```
# 步骤 1: 从错误提示或响应头获取 request_id
前端 toast: "推理失败 (request_id: a1b2c3d4)"

# 步骤 2: 搜索 request_id 找到 HTTP 请求和入队日志
grep "request_id=a1b2c3d4" logs/*.log

# 步骤 3: 从入队日志提取 task_id
→ event=task_enqueue task_id=q_x7y8z9a0b1c2 request_id=a1b2c3d4

# 步骤 4: 用 task_id 搜索调度和 TCP 日志
grep "task_id=q_x7y8z9a0b1c2" logs/*.log

# 步骤 5: 多节点场景，用 node_id 定位具体节点
grep "node_id=android-phone2" logs/*.log
```

## 分期执行计划

### 总体原则

- **先加固现有能力，再引入新视图**：当前 `/api/logs` 文件管理接口已经被 PC 前端使用，优先保持兼容。
- **先 PC 后 Android**：PC 后端日志是主链路，Android 已有 `QlhLogger.kt`，只需要做体验对齐。
- **先权限与安全，再做实时日志**：日志可能包含 IP、路径、异常堆栈和节点信息，实时查看前必须先明确访问边界。
- **不直接合并 `logging-module`**：只吸收内存缓冲、过滤 UI、统计接口等局部设计。
- **每阶段都可独立回滚**：避免一次性重构日志系统导致定位问题的工具本身出问题。

### 阶段 L0：接口安全与删除稳定性（P0，0.5-1 天）

**目标**：让现有日志管理能力在 Win/Linux 下更可靠，并避免局域网任意访问者读取/删除日志。

| 任务 | 文件 | 要点 |
|------|------|------|
| 增加日志 API 权限判断 | `src/api_server.py` | 仅主节点、管理员或本机访问允许读取/删除日志 |
| 日志删除加全局锁 | `src/api_server.py` | 包住关闭 handler、删除文件、重建 handler 全流程 |
| 删除当前日志时明确提示 | `frontend/src/components/SettingsModal.jsx` | 提示“当前日志会重新生成” |
| 删除操作写审计日志 | `src/api_server.py` | 记录操作者、文件名、删除结果 |
| 失败原因细化 | `src/api_server.py` | 返回 failed 列表和错误摘要 |

**验收标准**：

- Windows 下删除当前正在写入的日志不会失败或导致后续日志丢失；

- 删除操作本身能在新日志里看到。

### 风险控制

- 不改 `/api/logs` 现有语义，避免破坏 `SettingsModal.jsx`；
- 新增实时日志接口必须有数量上限；
- 日志 API 权限控制失败时宁可拒绝访问，不默认放行；
- 删除日志必须串行化，避免多个请求同时关闭/重建 handler；
- request_id middleware 不能记录大请求体，避免泄露用户输入和拖慢聊天；
- Android 日志继续走 Kotlin 原生实现，不引入 Python/React 模块依赖。
